# Лекция 2: Создание инструментов в LangChain — Искусство вооружения агента

## Введение: От ремесла к конвейеру

В предыдущей лекции мы узнали, как работает Function Calling "под капотом" — специальные токены, JSON Schema, циклы вызова и ответа. Это было похоже на изучение физики двигателя внутреннего сгорания: полезно для понимания, но вы же не собираетесь собирать мотор вручную каждый раз, когда нужно куда-то поехать?

LangChain — это фабрика, которая превращает создание инструментов из ручного ремесла в конвейерное производство. Вместо того чтобы писать JSON Schema вручную, парсить ответы модели, обрабатывать ошибки и собирать всё это воедино — вы просто декорируете Python-функцию и получаете готовый к использованию инструмент. Это как разница между программированием на ассемблере и на Python: результат тот же, но трудозатраты несопоставимы.

Но LangChain — это не просто "синтаксический сахар". Фреймворк предлагает продуманную архитектуру, которая учитывает реальные потребности продакшн-систем: валидацию аргументов, обработку ошибок, асинхронное выполнение, интеграцию с различными провайдерами моделей. Понимание этой архитектуры — ключ к созданию надёжных агентов.

## Часть 1: Декоратор @tool — Магия одной строки

### Анатомия простейшего инструмента

Начнём с минимального примера:

```python
from langchain_core.tools import tool

@tool
def multiply(a: int, b: int) -> int:
    """Умножает два числа и возвращает результат."""
    return a * b
```

Три строки кода — и у вас готовый инструмент. Но за этой простотой скрывается серьёзная работа. Декоратор `@tool` выполняет четыре операции:

1. **Извлекает имя** функции ("multiply") и превращает его в имя инструмента
2. **Берёт docstring** и делает из него описание — то самое description, которое модель будет читать, решая, когда использовать этот инструмент
3. **Анализирует type hints** (a: int, b: int) и строит JSON Schema для параметров
4. **Оборачивает функцию** в объект класса Tool со стандартизированным интерфейсом

Важная деталь: параметры без значений по умолчанию попадают в `required` в JSON Schema. Если бы мы написали `b: int = 1`, параметр b стал бы опциональным. Модель обращает на это внимание — для required-параметров она всегда постарается найти значение.

### Искусство написания docstring

Docstring — это не просто комментарий для разработчиков. Для инструмента это *инструкция по применению*, которую будет читать языковая модель. Качество docstring напрямую определяет, насколько умно агент будет пользоваться инструментом.

Сравните два подхода:

**Плохой docstring:** `"""Поиск."""` — Модель не понимает: поиск чего? Где? Когда его использовать?

**Хороший docstring:**

```python
@tool
def search_web(query: str) -> str:
    """
    Поиск информации в интернете через поисковую систему.

    Используй этот инструмент когда:
    - Пользователь спрашивает о текущих событиях или новостях
    - Нужны актуальные данные (курсы валют, погода, цены)
    - Требуется проверить факт из внешнего источника

    НЕ используй когда:
    - Вопрос об общих знаниях, которые ты точно знаешь
    - Пользователь просит твоё мнение или рассуждение

    Args:
        query: Поисковый запрос на английском языке. Формулируй
               кратко и конкретно, как для Google.
    """
```

Этот docstring — почти промпт. Он объясняет модели контекст использования, даёт позитивные и негативные примеры, описывает ожидаемый формат входа. Да, это длинно. Но исследования показывают, что качество description — один из главных факторов, влияющих на точность выбора инструмента моделью.

**Практический совет:** пишите описания так, как будто объясняете стажёру, когда и как пользоваться инструментом. Чем конкретнее — тем лучше. "Используй для поиска" — плохо. "Используй для поиска актуальных новостей, когда пользователь спрашивает о событиях последних 24 часов" — хорошо.

### Параметры декоратора @tool

Декоратор принимает параметры для тонкой настройки:

```python
@tool(
    name="web_search",    # Переопределить имя (по умолчанию — имя функции)
    return_direct=False,  # Если True, результат сразу возвращается пользователю
    args_schema=None,     # Кастомная Pydantic-схема для аргументов
)
```

Параметр `return_direct` заслуживает особого внимания. По умолчанию результат инструмента возвращается модели, которая затем формулирует ответ пользователю. Но иногда это избыточно — если инструмент уже вернул готовый к отображению результат (график, таблицу, URL), нет смысла пропускать его через модель ещё раз. Значение `return_direct=True` говорит: "Результат готов к показу, не нужно его пересказывать".

Когда использовать `return_direct=True`:
- Инструмент возвращает форматированные данные (таблицы, графики, ссылки)
- Результат не требует интерпретации моделью
- Нужно сэкономить на лишнем вызове LLM

Когда **не** использовать:
- Результат — сырые данные, которые нужно осмыслить
- Модель должна комбинировать результат с другой информацией
- Нужен естественный ответ на естественном языке

## Часть 2: StructuredTool — Полный контроль

### Когда декоратора недостаточно

Декоратор `@tool` покрывает 80% случаев, но иногда нужен более гранулярный контроль. Типичные ситуации:

- Схема аргументов сложна и требует nested-моделей, кастомных валидаторов
- Одна функция используется несколькими инструментами с разными настройками
- Инструмент создаётся динамически (из конфигурации, из базы данных)
- Нужно отделить описание схемы от бизнес-логики

Класс `StructuredTool` позволяет создавать инструменты программно:

```python
from langchain_core.tools import StructuredTool
from pydantic import BaseModel, Field

class CalculatorInput(BaseModel):
    expression: str = Field(description="Математическое выражение, например '2 + 2 * 3'")
    precision: int = Field(default=2, description="Знаков после запятой", ge=0, le=10)

calculator = StructuredTool.from_function(
    func=safe_eval,
    name="calculator",
    description="Вычисляет математические выражения. Поддерживает +, -, *, /, **, скобки.",
    args_schema=CalculatorInput,
)
```

### @tool vs StructuredTool: как выбрать

| Критерий | @tool | StructuredTool |
|----------|-------|----------------|
| Простота создания | Одна строка | 5-10 строк |
| Контроль над схемой | Через type hints и docstring | Полный (Pydantic-модель) |
| Валидация | Базовая (типы) | Продвинутая (validators, constraints) |
| Динамическое создание | Нет | Да (from_function) |
| Async поддержка | Через async def | Через coroutine= |
| Когда использовать | Простые инструменты, прототипы | Сложные схемы, продакшен |

Общее правило: начинайте с `@tool`, переходите на `StructuredTool` когда растёт сложность. Преждевременная абстракция хуже, чем простой декоратор.

### Асинхронные инструменты

Современные агенты часто работают асинхронно — это критично для производительности при множественных параллельных вызовах. Если агент вызывает три API одновременно, синхронное выполнение займёт сумму времён, асинхронное — время самого медленного.

LangChain поддерживает async из коробки. Для `@tool` достаточно объявить функцию как `async def`. Для `StructuredTool` используйте параметр `coroutine=` вместо `func=`. Если передать оба — будет доступна и синхронная, и асинхронная версия.

## Часть 3: Валидация аргументов через Pydantic

### Почему валидация критически важна

Языковые модели не идеальны. Они могут вызвать функцию с некорректными аргументами: строка вместо числа, отрицательное значение где ожидается положительное, пустая строка вместо осмысленного запроса. Без валидации это приведёт к непредсказуемому поведению: исключениям в runtime, некорректным результатам, уязвимостям безопасности.

Pydantic — это библиотека валидации данных, которая идеально интегрируется с LangChain. Она позволяет **декларативно** описать ожидаемые данные и автоматически проверить их соответствие. Это как турникет в метро: пропускает только тех, у кого есть билет правильного формата.

### Пример: Поиск авиабилетов

Рассмотрим реалистичный пример с множественной валидацией:

```python
from pydantic import BaseModel, Field, field_validator
from datetime import date

class FlightSearchInput(BaseModel):
    origin: str = Field(description="Код аэропорта вылета (IATA)", min_length=3, max_length=3)
    destination: str = Field(description="Код аэропорта назначения (IATA)", min_length=3, max_length=3)
    departure_date: date = Field(description="Дата вылета (YYYY-MM-DD)")
    passengers: int = Field(default=1, ge=1, le=9, description="Количество пассажиров")

    @field_validator("origin", "destination")
    @classmethod
    def uppercase_airport_code(cls, v: str) -> str:
        return v.upper()  # svo → SVO

    @field_validator("departure_date")
    @classmethod
    def departure_not_in_past(cls, v: date) -> date:
        if v < date.today():
            raise ValueError("Дата вылета не может быть в прошлом")
        return v
```

Этот класс обеспечивает многоуровневую защиту:

- **Типовая валидация** — Pydantic автоматически пытается привести данные к нужным типам (строка "3" → int 3)
- **Constraints** — `min_length=3, max_length=3` гарантирует ровно 3 символа для IATA-кода, `ge=1, le=9` ограничивает пассажиров
- **Кастомные валидаторы** — нормализация (upper case) и бизнес-логика (дата не в прошлом)
- **JSON Schema** — всё это автоматически транслируется в JSON Schema для модели, включая ограничения

### Обработка ошибок валидации

Когда валидация не прошла, важно вернуть модели не stack trace, а понятное сообщение. Pydantic предоставляет структурированные ошибки: поле, тип ошибки, человекочитаемое описание. Их можно сформировать в инструкцию для модели: "Ошибка в параметрах: departure_date — дата вылета не может быть в прошлом. Попробуй исправить и вызвать снова."

Умная модель поймёт ошибку и скорректирует вызов — например, спросит пользователя о правильной дате. Это пример graceful degradation: вместо падения агент адаптируется.

### Когда валидация избыточна

Не каждый инструмент нуждается в Pydantic-схеме с кастомными валидаторами. Для простых инструментов (умножение чисел, конкатенация строк) базовых type hints достаточно. Pydantic добавляет overhead: время на валидацию, сложность кода, дополнительные зависимости.

Используйте полноценную валидацию когда:
- Инструмент принимает данные от пользователя (injection-риски)
- Ошибка может привести к необратимым последствиям (отправка email, платёж)
- Параметры имеют сложные взаимозависимости (дата возвращения после вылета)
- Инструмент вызывает дорогие или медленные операции

## Часть 4: handle_tool_error — Устойчивость к сбоям

### Философия graceful degradation

В продакшн-системах ошибки неизбежны. API может быть недоступен, сетевое соединение — нестабильно, данные — некорректны. Агент не должен "падать" при первой же проблеме. Вместо этого он должен либо попробовать снова, либо сообщить о проблеме и продолжить работу с другими инструментами.

Это философия **graceful degradation** — система деградирует плавно, сохраняя максимум функциональности при частичных сбоях. Как автомобиль, у которого перегорела лампочка приборной панели: он продолжает ехать, просто сообщает о проблеме.

### Механизм handle_tool_error

LangChain предоставляет три уровня обработки ошибок:

**Уровень 1: `handle_tool_error=True`** — исключение `ToolException` превращается в текстовое сообщение об ошибке, которое модель получит как "результат" инструмента.

```python
@tool(handle_tool_error=True)
def fetch_stock_price(ticker: str) -> str:
    """Получает текущую цену акции по тикеру."""
    response = requests.get(f"https://api.stocks.com/{ticker}")
    if not response.ok:
        raise ToolException(f"Не удалось получить данные для {ticker}")
    return f"Цена {ticker}: ${response.json()['price']}"
```

**Уровень 2: Кастомный обработчик** — функция, которая получает ошибку и возвращает информативное сообщение с рекомендациями:

```python
def smart_error_handler(error: ToolException) -> str:
    error_msg = str(error)
    if "rate limit" in error_msg.lower():
        return "API перегружен. Подожди минуту и попробуй снова."
    elif "not found" in error_msg.lower():
        return "Тикер не найден. Проверь написание (AAPL, GOOGL, TSLA)."
    return f"Ошибка: {error_msg}. Попробуй другой инструмент."

@tool(handle_tool_error=smart_error_handler)
def fetch_stock_price(ticker: str) -> str:
    ...
```

**Уровень 3: Retry с backoff** — для временных сбоев автоматически повторяем вызов с нарастающей задержкой. Библиотека `tenacity` или ручная реализация с формулой `delay = base_delay × 2^attempt` (1с → 2с → 4с).

### Как модель использует сообщения об ошибках

Качество сообщения об ошибке определяет, насколько умно модель на неё отреагирует. Сравните:

- `"Error"` → модель в растерянности, может повторить тот же вызов
- `"Тикер TSLA1 не найден"` → модель может попробовать TSLA
- `"Тикер TSLA1 не найден. Правильные примеры: AAPL, GOOGL, TSLA"` → модель исправит вызов
- `"API недоступен. Используй инструмент search_web для поиска цены акции"` → модель переключится на альтернативную стратегию

Хорошее сообщение об ошибке — это не жалоба, а инструкция к действию.

## Часть 5: Композиция и динамическое создание инструментов

### Инструменты, использующие другие инструменты

В реальных системах инструменты редко существуют изолированно. Часто один инструмент требует результатов другого. Например, "написать отчёт о компании" может внутри использовать "найти финансовые данные" и "найти новости".

Ключевой метод — `.invoke()`: стандартный способ вызвать инструмент из другого инструмента. Он принимает словарь аргументов и возвращает результат:

```python
@tool
def generate_company_report(ticker: str, company_name: str) -> str:
    """Генерирует комплексный отчёт, собирая данные из разных источников."""
    financials = get_company_financials.invoke({"ticker": ticker})
    news = get_company_news.invoke({"company_name": company_name})
    return f"# Отчёт о {company_name}\n\n## Финансы\n{financials}\n\n## Новости\n{news}"
```

Это паттерн **tool composition** — один инструмент оркестрирует другие. Агент видит только верхнеуровневый инструмент (`generate_company_report`), внутренняя логика скрыта. Это упрощает выбор для модели: вместо трёх инструментов она видит один.

Но есть и обратная сторона: если один из внутренних инструментов упадёт, весь составной инструмент вернёт ошибку. Поэтому внутри composition особенно важна обработка ошибок — каждый вызов `.invoke()` стоит оборачивать в try/except.

### Динамическое создание инструментов

Иногда набор инструментов заранее неизвестен — он зависит от контекста или конфигурации. Например, при подключении к базе данных нужно создать инструмент для каждой таблицы. Или при загрузке API-спецификации — инструмент для каждого эндпоинта.

`StructuredTool.from_function()` позволяет создавать инструменты программно, что делает возможными паттерны вроде:

- **Инструменты из конфигурации** — JSON-файл описывает доступные API, код генерирует инструменты
- **Инструменты из базы данных** — для каждой таблицы создаётся инструмент запросов
- **Инструменты из API-спецификации** — OpenAPI-документ → набор инструментов (подробнее в лекции 5)

Этот паттерн позволяет адаптировать агента к конкретному окружению без изменения кода.

## Часть 6: Инструменты и контекст выполнения

### Доступ к состоянию агента из инструмента

Иногда инструменту нужен доступ к контексту выполнения: текущему пользователю, истории диалога, настройкам. LangChain предоставляет механизм `RunnableConfig` для передачи контекста:

```python
from langchain_core.runnables import RunnableConfig

@tool
def get_user_preferences(config: RunnableConfig) -> str:
    """Получает настройки текущего пользователя."""
    user_id = config.get("configurable", {}).get("user_id")
    if not user_id:
        return "Пользователь не идентифицирован"
    return json.dumps(database.get_preferences(user_id))
```

Параметр `config` — специальный. LangChain автоматически передаёт в него текущую конфигурацию выполнения, **не включая его в JSON Schema** как аргумент для модели. Модель не знает о его существовании — она вызывает `get_user_preferences()` без аргументов, а `config` инжектируется фреймворком.

Это позволяет разделить "публичный API инструмента" (то, что видит модель) и "внутренний контекст" (то, что нужно для выполнения).

### Инструменты с побочными эффектами

Некоторые инструменты изменяют состояние мира: отправляют письма, создают записи в базе данных, совершают покупки. Такие инструменты требуют особого обращения, потому что ошибки необратимы.

Четыре принципа для инструментов с побочными эффектами:

1. **Явное разрешение** — проверяем флаг в конфигурации: `config["allow_email"]`. Нет флага — нет действия.
2. **Логирование** — записываем все действия для аудита. Кто, когда, что отправил.
3. **Предупреждение в описании** — модель должна знать, что действие необратимо. Слова "ВАЖНО: необратимое действие" в docstring меняют поведение модели.
4. **Human-in-the-loop** — для критических действий (платёж, удаление данных) агент должен запросить подтверждение пользователя перед выполнением.

Эти принципы особенно важны в контексте **prompt injection**: злоумышленник может попытаться заставить агента выполнить опасное действие через манипуляцию промптом. Многоуровневая защита (разрешения + логирование + подтверждение) минимизирует этот риск.

## Часть 7: Тестирование инструментов

### Стратегия тестирования

Инструменты — это обычные Python-функции, и их можно тестировать стандартными методами. Но есть специфика: инструменты часто зависят от внешних сервисов (API, базы данных), и их нужно мокировать.

**Три уровня тестирования:**

1. **Юнит-тесты** — проверка логики функции с мокированными зависимостями. Быстрые, запускаются при каждом коммите.

2. **Тесты схемы** — проверка, что JSON Schema корректно сгенерирована из type hints и Pydantic-модели. Ловят регрессии при рефакторинге.

3. **Интеграционные тесты** — проверка с реальными API. Медленные, требуют ключей, запускаются отдельно (маркер `@pytest.mark.integration`).

### Что именно тестировать

Для каждого инструмента стоит проверить:

- **Позитивный путь** — корректный вызов возвращает ожидаемый результат: `multiply.invoke({"a": 3, "b": 4}) == 12`
- **Негативный путь** — некорректные аргументы обрабатываются gracefully, а не падают с exception
- **Схему** — `tool.args_schema.model_json_schema()` содержит правильные типы и required-поля
- **Описание** — `tool.description` не пустое и содержит ключевые слова (регрессия описания может сломать поведение агента)
- **Edge cases** — пустые строки, None, граничные значения, Unicode

Для инструментов с внешними зависимостями используйте `unittest.mock.patch` для подмены API-вызовов. Мок должен возвращать реалистичные данные — это позволяет тестировать логику обработки ответа.

Полные примеры тестов (юнит, интеграционные, тесты валидации) доступны в файле примеров к этой лекции.

## Заключение: Инструменты как мост между мирами

Создание инструментов — это искусство строительства мостов между миром языковых моделей и миром реальных систем. Декоратор `@tool` делает этот процесс простым, `StructuredTool` даёт полный контроль, Pydantic обеспечивает безопасность, `handle_tool_error` гарантирует устойчивость.

Но помните: инструмент — это не просто функция. Это *контракт* между вашим кодом и языковой моделью. Качество описания (docstring) определяет, насколько умно модель будет пользоваться инструментом. Валидация аргументов защищает от некорректного использования. Обработка ошибок обеспечивает graceful degradation при сбоях.

В следующей лекции мы перейдём от создания отдельных инструментов к использованию готовых коллекций — Toolkit'ов, которые предоставляют целые наборы инструментов для конкретных задач: поиска в интернете, работы с файлами, выполнения кода.

## Ключевые термины

| Термин | Определение |
|--------|-------------|
| **@tool** | Декоратор LangChain для преобразования Python-функции в инструмент агента |
| **StructuredTool** | Класс для программного создания инструментов с полным контролем над параметрами |
| **args_schema** | Pydantic-модель, описывающая структуру и валидацию аргументов инструмента |
| **return_direct** | Флаг, указывающий вернуть результат инструмента напрямую пользователю без обработки моделью |
| **handle_tool_error** | Механизм перехвата и обработки ошибок инструмента для graceful degradation |
| **ToolException** | Специальное исключение для ошибок инструментов, обрабатываемое handle_tool_error |
| **RunnableConfig** | Объект конфигурации для передачи контекста выполнения (user_id, настройки) в инструмент |
| **Tool Composition** | Паттерн, при котором один инструмент вызывает другие через .invoke() |
| **Graceful Degradation** | Философия, при которой система деградирует плавно, сохраняя максимум функциональности |
