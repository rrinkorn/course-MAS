# Лекция 5: OpenAPI и API Chains — Агент как универсальный клиент

## Введение: Мир API

Современный интернет — это огромная сеть взаимосвязанных сервисов, общающихся друг с другом через API. Хотите узнать погоду? Есть API OpenWeatherMap. Отправить SMS? API Twilio. Забронировать авиабилет? API Aviasales. Обработать платёж? API Stripe. Буквально каждый сервис, которым вы пользуетесь, за кулисами общается с десятками других сервисов через программные интерфейсы.

И вот что интересно: большинство этих API описаны в стандартизированном формате — OpenAPI (ранее известном как Swagger). Это как техническая документация, но для машин. Файл OpenAPI-спецификации содержит полное описание API: какие эндпоинты доступны, какие параметры они принимают, какие ответы возвращают, как авторизоваться.

А теперь представьте: что если языковая модель сможет читать эти спецификации и автоматически вызывать нужные эндпоинты? Вместо того чтобы вручную писать инструменты для каждого API, мы могли бы просто "скормить" агенту OpenAPI-спецификацию, и он сам разобрался бы, как с ней работать.

Это и есть суть API Chains — паттерна, при котором агент динамически генерирует HTTP-запросы на основе документации API. Звучит как магия, но за ней стоит вполне конкретная инженерия, которую мы сегодня разберём.

## Часть 1: OpenAPI Specification — язык описания API

### Проблема: тысяча API — тысяча форматов

Представьте, что вы приехали в страну, где в каждом доме свой формат розеток. В гостинице — круглые штырьки, в ресторане — плоские, в офисе — вообще треугольные. Каждый раз вам нужен новый переходник. Именно так выглядела интеграция с API до появления стандартов: каждый сервис документировал свой API по-своему, в произвольном формате, и для каждого нужно было писать уникальный код интеграции.

OpenAPI Specification (OAS) — это тот самый "единый стандарт розеток". Это формализованный, машиночитаемый способ описать любой REST API. Формат прошёл эволюцию от Swagger (2010) через Swagger 2.0 к OpenAPI 3.0 и 3.1, став индустриальным стандартом, поддерживаемым Linux Foundation.

### Что содержит OpenAPI-документ

OpenAPI-спецификация — это YAML или JSON файл со строгой структурой. Рассмотрим её ключевые компоненты на сокращённом примере:

```yaml
openapi: 3.0.3
info:
  title: Task Management API
  version: 1.0.0

servers:
  - url: https://api.taskmanager.com/v1

paths:
  /tasks:
    get:
      summary: Получить список задач
      parameters:
        - name: status
          in: query
          schema:
            type: string
            enum: [open, in_progress, done]
    post:
      summary: Создать задачу
      requestBody:
        content:
          application/json:
            schema:
              $ref: '#/components/schemas/TaskCreate'

components:
  schemas:
    TaskCreate:
      type: object
      required: [title]
      properties:
        title:
          type: string
        assignee_id:
          type: integer

  securitySchemes:
    bearerAuth:
      type: http
      scheme: bearer
```

Даже в этом сокращённом примере видно, что документ содержит всё необходимое для машинного взаимодействия:

- **info** — метаданные API (название, версия, описание)
- **servers** — базовые URL для запросов
- **paths** — карта эндпоинтов с описанием методов, параметров и ответов
- **components/schemas** — переиспользуемые схемы данных (структуры объектов)
- **securitySchemes** — способы авторизации

Каждый параметр типизирован: модель знает, что `status` — строка с ограниченным набором значений, а `assignee_id` — целое число. Это даёт LLM достаточно информации, чтобы формировать корректные запросы без дополнительных инструкций.

### Почему OpenAPI меняет парадигму интеграции

Традиционный подход к подключению нового API — это цикл из нескольких этапов: прочитать документацию, написать обёртку на Python, оформить как LangChain-инструмент, протестировать. Для каждого API — отдельный труд, отдельный код, отдельные баги.

OpenAPI предлагает радикально другой подход. Поскольку спецификация — это машиночитаемый контракт, мы можем:

1. **Автоматически парсить** спецификацию, извлекая эндпоинты и параметры
2. **Программно генерировать** инструменты из описания
3. **Передавать описания** модели как контекст, позволяя ей формировать запросы самостоятельно

Это принципиально масштабируемый подход: один раз написать логику работы с OpenAPI — и она применима к тысячам API. Вместо того чтобы быть "переводчиком" между каждым конкретным API и агентом, мы создаём универсальный мост.

Однако у этого подхода есть и ограничения. Не все API имеют качественные OpenAPI-спецификации — некоторые описания неполны, устарели или вовсе отсутствуют. Кроме того, автоматически сгенерированные инструменты могут иметь слишком общие описания, что затрудняет выбор правильного эндпоинта для модели. Поэтому на практике OpenAPI-подход часто комбинируют с ручной доработкой ключевых инструментов.

## Часть 2: От спецификации к инструментам

### Три подхода к интеграции API

Существует спектр подходов к превращению API в инструменты для агента — от полностью автоматического до полностью ручного:

| Подход | Скорость разработки | Контроль | Качество описаний | Когда использовать |
|--------|-------------------|----------|------------------|-------------------|
| **OpenAPIToolkit** | Быстро | Низкий | Зависит от спецификации | Быстрый прототип, хорошая спецификация |
| **Ручные инструменты** | Медленно | Высокий | Полный контроль | Продакшен, критические API |
| **Гибридный** | Средне | Средний | Комбинация | Большинство реальных проектов |

### Автоматический подход: OpenAPIToolkit

LangChain предоставляет `OpenAPIToolkit`, который автоматически создаёт инструменты из спецификации:

```python
from langchain_community.agent_toolkits.openapi import OpenAPIToolkit
from langchain_community.utilities.openapi import OpenAPISpec
from langchain_community.utilities.requests import TextRequestsWrapper

spec = OpenAPISpec.from_url("https://api.taskmanager.com/openapi.json")
wrapper = TextRequestsWrapper(headers={"Authorization": f"Bearer {TOKEN}"})

toolkit = OpenAPIToolkit.from_openapi_spec(
    spec=spec, requests_wrapper=wrapper, llm=llm,
    allow_dangerous_requests=True  # Требуется для POST/PUT/DELETE
)
tools = toolkit.get_tools()
```

Toolkit автоматически создаёт инструмент для каждого эндпоинта: `GET_tasks`, `POST_tasks`, `GET_tasks_taskId`. Название формируется из метода и пути, описание берётся из спецификации.

Это удобно для прототипирования, но есть существенные ограничения. Описания инструментов наследуют качество спецификации — если описание в OpenAPI-файле скудное ("Get tasks"), модель будет хуже понимать, когда и как его использовать. Кроме того, вы теряете контроль над форматированием ответов, обработкой ошибок и валидацией параметров.

### Ручной подход: полный контроль

Для критически важных API лучше создавать инструменты вручную. Это даёт три ключевых преимущества:

1. **Оптимизированные описания** — вы пишете description специально для LLM, с примерами использования и граничными случаями
2. **Контролируемая обработка ошибок** — вместо "500 Internal Server Error" агент получает "Сервер временно недоступен, попробуй через минуту"
3. **Форматированные ответы** — вместо сырого JSON агент получает человекочитаемый текст

```python
@tool
def get_tasks(status: str = None, assignee_id: int = None) -> str:
    """
    Получает список задач из Task Manager.
    Используй для поиска задач по статусу или исполнителю.

    Args:
        status: Фильтр по статусу ('open', 'in_progress', 'done')
        assignee_id: ID исполнителя для фильтрации
    """
    params = {k: v for k, v in {"status": status, "assignee": assignee_id}.items() if v}
    result = api.get("/tasks", params=params)
    return json.dumps(result, indent=2)
```

Ключевое отличие от автоматического подхода — docstring написан специально для модели, с чёткими инструкциями "когда использовать" и перечислением допустимых значений.

### Как выбрать подход

Выбор зависит от контекста проекта. Если вы интегрируете хорошо документированный API для внутреннего прототипа — OpenAPIToolkit сэкономит часы работы. Если строите продакшен-систему, где агент взаимодействует с платёжным API или базой клиентов — ручные инструменты с продуманными описаниями и обработкой ошибок обязательны.

В большинстве реальных проектов работает гибридный подход: начать с OpenAPIToolkit для быстрого прототипа, затем постепенно заменять автоматические инструменты на ручные для наиболее критичных операций.

## Часть 3: API Chains — динамическая генерация запросов

### Концепция: статические vs динамические инструменты

До сих пор мы рассматривали подход, при котором набор инструментов фиксирован на этапе конфигурации — модель выбирает из готового списка. Но есть принципиально другой подход: дать модели **один** generic-инструмент, который позволяет вызвать произвольный эндпоинт API.

Это и есть API Chain — паттерн, при котором модель не выбирает из готовых инструментов, а *генерирует* HTTP-запросы на основе понимания API.

> **Важно:** Класс `APIChain` из LangChain был deprecated в версии 0.2.13 и **удалён в LangChain 1.0** (октябрь 2025). Современный подход — создание кастомного generic-инструмента:

```python
@tool
def query_api(endpoint: str, params: dict = None) -> str:
    """
    Выполняет GET-запрос к Task Manager API.

    Args:
        endpoint: Путь API (например, '/tasks', '/users/42')
        params: Query-параметры запроса
    """
    response = httpx.get(f"{API_BASE}{endpoint}", params=params, headers=HEADERS)
    response.raise_for_status()
    return response.json()
```

Модель самостоятельно решает, какой эндпоинт вызвать и с какими параметрами. На вопрос "Покажи открытые задачи Иванова" она сформирует: `query_api(endpoint="/tasks", params={"status": "open", "assignee": "42"})`.

### Плюсы и минусы динамического подхода

Динамическая генерация запросов привлекательна своей гибкостью. Один инструмент покрывает весь API, а модель адаптируется к новым эндпоинтам без изменения кода. Однако это несёт серьёзные риски:

**Гибкость** — модель может комбинировать параметры способами, которые вы не предвидели. Это и плюс (покрывает edge cases), и минус (может формировать некорректные запросы).

**Безопасность** — если дать модели generic-инструмент без ограничений, она может попытаться вызвать любой эндпоинт, включая опасные: `DELETE /users/1`, `PUT /settings/admin`. Prompt injection может эксплуатировать это для несанкционированных действий.

**Предсказуемость** — с типизированными инструментами вы точно знаете, какие операции возможны. С generic-инструментом поведение агента менее детерминировано и сложнее тестировать.

### Гибридный подход — золотая середина

На практике чаще всего используют комбинацию: типизированные инструменты для критических и частых операций + generic-инструмент (только для чтения!) для исследовательских запросов. Типизированные инструменты дают предсказуемость и безопасность, а generic-инструмент — гибкость для нестандартных случаев.

Важный принцип: generic-инструмент должен быть **read-only**. Операции записи (создание, обновление, удаление) всегда выполняются через явные инструменты с валидацией параметров.

## Часть 4: Авторизация в API

### Типы авторизации и когда их использовать

Практически каждый API за пределами публичных данных требует авторизации. Выбор схемы зависит от контекста использования:

| Тип | Безопасность | Сложность | Когда использовать | Пример заголовка |
|-----|-------------|-----------|-------------------|-----------------|
| **API Key** | Базовая | Простая | Внутренние сервисы, прототипы | `X-API-Key: abc123` |
| **Bearer Token** | Средняя | Средняя | Большинство публичных API | `Authorization: Bearer eyJhb...` |
| **Basic Auth** | Низкая | Простая | Legacy-системы, совместимость | `Authorization: Basic dXNlcjpw...` |
| **OAuth 2.0** | Высокая | Высокая | Действия от имени пользователя | Многоэтапный процесс |

**API Key** — простейший вариант: случайная строка, передаваемая в заголовке или query-параметре. Подходит для серверных интеграций, где ключ хранится безопасно. Главный недостаток — ключ не истекает и даёт полный доступ.

**Bearer Token** (часто JWT) — наиболее распространённый вариант для публичных API. Токен имеет ограниченный срок жизни и может содержать информацию о scope — какие операции разрешены.

**OAuth 2.0 с refresh** — используется, когда агент действует от имени конкретного пользователя (например, читает его Gmail или управляет GitHub-репозиторием). Самый безопасный, но и самый сложный вариант — требует потока авторизации с redirect, хранения refresh-токенов и их периодического обновления.

### Централизованное управление авторизацией

В контексте агентов критически важно централизовать управление авторизацией. Если каждый инструмент самостоятельно управляет токенами, возникают проблемы: дублирование кода, несогласованное обновление токенов, риск утечки секретов.

LangChain предоставляет `TextRequestsWrapper` для этой цели:

```python
from langchain_community.utilities.requests import TextRequestsWrapper

wrapper = TextRequestsWrapper(
    headers={"Authorization": f"Bearer {os.environ['API_TOKEN']}"}
)
```

Все запросы через этот wrapper автоматически включают нужные заголовки.

### Хранение секретов — не опционально

Хардкод токенов в коде — это не просто bad practice, это уязвимость. Токен в git-репозитории может быть найден через GitHub search даже после удаления из кода (он остаётся в истории коммитов). Статистика показывает, что утечка API-ключей через публичные репозитории — одна из самых частых причин компрометации сервисов.

Минимальный уровень защиты — переменные окружения и `.env` файл (добавленный в `.gitignore`). Для production — HashiCorp Vault, AWS Secrets Manager или Azure Key Vault, где секреты шифруются, версионируются и аудитируются.

## Часть 5: Обработка ошибок и retry-логика

### Типичные ошибки при работе с API

API могут отвечать ошибками по множеству причин, и критически важно правильно классифицировать эти ошибки для принятия решений:

**Временные ошибки (можно повторить):**
- 429 Too Many Requests — превышен rate limit, нужно подождать
- 500 Internal Server Error — сервер "упал", но может восстановиться
- 502/503/504 — проблемы с инфраструктурой, обычно кратковременные

**Постоянные ошибки (повторять бессмысленно):**
- 400 Bad Request — ваш запрос некорректен, нужно исправить параметры
- 401 Unauthorized — проблемы с авторизацией, нужен новый токен
- 403 Forbidden — нет прав, повтор не поможет
- 404 Not Found — ресурс не существует

Эта классификация определяет стратегию: для временных ошибок — автоматический retry с exponential backoff, для постоянных — информативное сообщение агенту.

### Exponential backoff — зачем и как

Наивный retry ("не получилось — попробуй ещё раз через секунду") создаёт проблему "стада": если сервер перегружен и тысяча клиентов одновременно повторяют запросы через секунду, нагрузка на сервер только растёт.

Exponential backoff решает это: каждая следующая попытка ждёт экспоненциально дольше (1с → 2с → 4с → 8с), часто с добавлением случайного jitter. Это распределяет нагрузку во времени и даёт серверу шанс восстановиться.

Библиотека `tenacity` делает это элегантно:

```python
from tenacity import retry, stop_after_attempt, wait_exponential, retry_if_exception_type

@retry(
    retry=retry_if_exception_type(RetryableError),
    stop=stop_after_attempt(3),
    wait=wait_exponential(multiplier=1, min=1, max=10)
)
def call_api_with_retry(method: str, url: str, **kwargs) -> dict:
    response = requests.request(method, url, **kwargs)
    return handle_api_response(response)
```

### Информативные сообщения для агента

Отдельного внимания заслуживает то, *как* ошибка возвращается агенту. Stack trace или сырой HTTP-ответ — бесполезны для LLM. Вместо этого нужно формулировать ошибки как понятные инструкции к действию:

- Вместо `HTTP 404` → `"Задача с ID 42 не найдена. Проверь правильность ID."`
- Вместо `HTTP 403` → `"Нет доступа к задаче 42. Возможно, она в закрытом проекте."`
- Вместо `HTTP 429` → `"Превышен лимит запросов. Подожди минуту или используй другой источник."`

Это позволяет агенту осмысленно реагировать на ошибку: переспросить пользователя, попробовать альтернативный путь или объяснить ситуацию.

### Circuit Breaker — защита от каскадных сбоев

Когда внешний сервис стабильно недоступен, нет смысла продолжать отправлять запросы (даже с retry). Паттерн Circuit Breaker работает как автоматический выключатель: после N последовательных неудач цепь "размыкается" — все последующие запросы немедленно отклоняются без попытки выполнения. Через определённое время цепь переходит в состояние "полуоткрыта" — пропускается один пробный запрос. Если он успешен — цепь замыкается, и работа возобновляется.

Три состояния Circuit Breaker:
- **Closed** (нормальная работа) — запросы проходят, ошибки считаются
- **Open** (защита активна) — запросы отклоняются, счётчик таймера тикает
- **Half-Open** (проверка) — один пробный запрос, результат определяет переход

Этот паттерн особенно важен для агентов, потому что без него агент может застрять в бесконечном цикле retry, тратя время и деньги на заведомо безуспешные запросы.

## Часть 6: Rate Limiting и управление нагрузкой

### Почему rate limiting критичен для агентов

Большинство API имеют лимиты на количество запросов: 100 в минуту, 1000 в час и т.д. Для обычного приложения с предсказуемым потоком запросов это редко проблема. Но агенты — другое дело.

Агент, решающий сложную задачу, может за секунды отправить десятки запросов: "найди все задачи пользователя → для каждой получи детали → для каждой получи комментарии → ...". Без контроля нагрузки агент легко исчерпает rate limit и получит каскад 429-х ошибок.

### Стратегии ограничения нагрузки

Существуют три основных алгоритма rate limiting:

**Token Bucket** — представьте ведро, в которое с постоянной скоростью капают жетоны. Каждый запрос "берёт" жетон из ведра. Если ведро пустое — запрос ждёт. Это позволяет кратковременные всплески (пока ведро не опустеет), но ограничивает среднюю скорость. Наиболее популярный алгоритм для клиентской стороны.

**Sliding Window** — подсчитывает количество запросов за скользящее окно времени (например, последние 60 секунд). Более строгий, чем token bucket — не допускает всплесков на стыке окон.

**Leaky Bucket** — запросы добавляются в очередь и "вытекают" с постоянной скоростью. Обеспечивает максимально ровный поток запросов, но может добавлять заметную задержку.

На практике для клиентского rate limiting чаще всего используют Token Bucket или его вариацию. Для синхронного кода можно написать простую реализацию на основе `deque` и `time.sleep()`, для асинхронного — использовать библиотеку `aiolimiter`:

```python
from aiolimiter import AsyncLimiter

limiter = AsyncLimiter(60, 60)  # 60 запросов в минуту

async def rate_limited_request(session, method, url, **kwargs):
    async with limiter:
        async with session.request(method, url, **kwargs) as response:
            return await response.json()
```

### Кэширование: когда помогает, а когда вредит

Кэширование API-ответов — мощный инструмент для снижения нагрузки, но его применение требует осмысленности.

**Кэшируйте:**
- Справочные данные (курсы валют за прошлые даты, информация о городах, документация)
- Результаты поисковых запросов (если допустима задержка в несколько минут)
- Метаданные (список доступных сервисов, схемы данных)

**Не кэшируйте:**
- Данные, критичные по времени (текущая цена акции, наличие билетов)
- Результаты операций записи (создание задачи, обновление статуса)
- Персонализированные данные, которые часто меняются

Простейший подход — TTL-кэш (Time To Live): каждая запись хранится определённое время, после чего устаревает. TTL выбирается исходя из частоты обновления данных: для погоды — 10-15 минут, для справочников — часы или дни, для новостей — минуты.

Полная реализация rate limiter и TTL-кэша представлена в файле примеров к этой лекции.

## Часть 7: Реальные примеры интеграций

### Паттерны комбинирования API

Реальные задачи агентов редко решаются одним API-вызовом. Чаще требуется комбинация нескольких сервисов. Существуют два основных паттерна:

**Orchestration (оркестрация)** — центральный контроллер (агент) последовательно вызывает нужные API, передавая результаты одного в параметры другого. Агент "дирижирует" процессом, принимая решения на каждом шаге. Пример: "Найди рейс → проверь погоду в пункте назначения → забронируй отель → собери резюме".

**Choreography (хореография)** — каждый сервис знает, что делать дальше, и передаёт управление сам. Агент лишь инициирует первый шаг. Этот паттерн реже используется с LLM-агентами, но встречается в event-driven архитектурах.

Для агентов оркестрация — более естественный паттерн, потому что LLM хорошо справляется с ролью "планировщика": анализирует задачу, разбивает на шаги, выполняет их последовательно, адаптируясь к промежуточным результатам.

### Пример: API погоды

Рассмотрим типичный паттерн интеграции с публичным API на примере OpenWeatherMap:

```python
@tool
def get_current_weather(city: str, units: str = "metric") -> str:
    """
    Получает текущую погоду в городе.

    Args:
        city: Название города (на английском работает лучше)
        units: 'metric' для °C, 'imperial' для °F
    """
    response = requests.get(
        f"{OPENWEATHER_BASE}/weather",
        params={"q": city, "appid": API_KEY, "units": units, "lang": "ru"}
    )
    if response.status_code == 404:
        return f"Город '{city}' не найден. Попробуй на английском."
    data = response.json()
    return f"Погода в {data['name']}: {data['main']['temp']}°C, {data['weather'][0]['description']}"
```

Обратите внимание на несколько важных паттернов:
- **Специфичная обработка 404** — вместо generic-ошибки агент получает подсказку
- **Форматированный ответ** — не сырой JSON, а человекочитаемый текст
- **Документация для LLM** — описание параметров с практическими рекомендациями

### Паттерн: Multi-API Workflow

Планирование multi-API workflow — это задача, где сила агентов проявляется максимально. Рассмотрим сценарий "Планирование поездки":

1. **Агент анализирует запрос** — извлекает город, даты, предпочтения
2. **Вызывает Weather API** — получает прогноз погоды
3. **Вызывает Wikipedia API** — получает справку о городе
4. **Обобщает результаты** — формирует структурированный отчёт с рекомендациями

На каждом шаге агент принимает решения: если город не найден в Weather API — может переспросить пользователя или попробовать альтернативное написание. Если Wikipedia не отвечает — продолжит без справки. Эта адаптивность — ключевое преимущество агентного подхода перед жёсткими пайплайнами.

При проектировании таких workflow важно продумать:
- **Порядок вызовов** — независимые запросы можно выполнять параллельно
- **Обработку частичных сбоев** — workflow должен деградировать gracefully, а не падать целиком
- **Таймауты** — суммарное время всех API-вызовов не должно превышать разумные пределы

Полные реализации примеров (Weather API, GitHub API, Trip Planner) доступны в файле примеров к этой лекции.

## Заключение: API как расширение возможностей

OpenAPI и API Chains открывают агентам доступ к огромной экосистеме сервисов. Погода, социальные сети, платёжные системы, корпоративные инструменты — всё это становится доступно через единый интерфейс естественного языка.

Мы рассмотрели спектр подходов — от автоматической генерации инструментов из OpenAPI-спецификации до ручного создания с продуманными описаниями и обработкой ошибок. Ключевой вывод: не существует единственно правильного подхода, выбор зависит от контекста.

Ключевые принципы работы с API:

1. **Используйте OpenAPI** когда возможно — это стандартизированный, машиночитаемый формат, который упрощает интеграцию.

2. **Управляйте авторизацией централизованно** — через RequestsWrapper или подобные абстракции. Никогда не хардкодьте секреты.

3. **Обрабатывайте ошибки gracefully** — различайте временные и постоянные ошибки, возвращайте информативные сообщения агенту.

4. **Соблюдайте rate limits** — используйте rate limiters и кэширование, помните что агенты склонны к burst-нагрузке.

5. **Думайте о безопасности** — ограничивайте операции записи, используйте whitelist, не давайте generic-инструментам доступ к опасным эндпоинтам.

В следующей лекции мы рассмотрим мультимодальные инструменты — работу с изображениями, аудио и видео. Это добавит агентам ещё одно измерение возможностей: не только читать и писать текст, но и "видеть" и "создавать" визуальный контент.

## Ключевые термины

| Термин | Определение |
|--------|-------------|
| **OpenAPI** | Стандарт описания REST API в машиночитаемом формате (YAML/JSON), ранее Swagger |
| **API Chain** | Паттерн, при котором LLM динамически генерирует HTTP-запросы на основе документации |
| **TextRequestsWrapper** | Класс LangChain для централизованного управления HTTP-запросами и авторизацией |
| **Rate Limiting** | Ограничение количества запросов к API за единицу времени |
| **Token Bucket** | Алгоритм rate limiting, допускающий кратковременные всплески нагрузки |
| **Exponential Backoff** | Стратегия retry с экспоненциально растущими интервалами между попытками |
| **Circuit Breaker** | Паттерн защиты от каскадных сбоев при недоступности внешнего сервиса |
| **Bearer Token** | Тип авторизации, при котором токен передаётся в заголовке Authorization |
| **TTL Cache** | Кэш с ограниченным временем жизни записей (Time To Live) |
| **Orchestration** | Паттерн управления multi-API workflow через центральный контроллер |
