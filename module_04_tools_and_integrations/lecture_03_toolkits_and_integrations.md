# Лекция 3: Toolkit и готовые интеграции — Арсенал современного агента

## Введение: Зачем изобретать велосипед?

В предыдущих лекциях мы научились создавать инструменты с нуля — писать функции, оборачивать их декораторами, валидировать аргументы. Это фундаментальное знание, без которого невозможно понять, как работают агенты. Но в реальных проектах вы редко будете писать инструмент для поиска в интернете или работы с файлами самостоятельно — это уже сделали до вас.

LangChain предоставляет богатую экосистему готовых интеграций — **Toolkit'ов**. Toolkit — это коллекция связанных инструментов для решения определённого класса задач. Toolkit для поиска даёт инструменты поиска в Google, Bing, специализированных базах. Toolkit для работы с файлами — инструменты чтения, записи, поиска файлов. Toolkit для выполнения кода — интерпретаторы Python, JavaScript, shell.

Использование готовых Toolkit'ов — это не "срезание углов", а инженерная мудрость. Эти интеграции прошли тысячи часов тестирования, оптимизированы для производительности, защищены от типичных ошибок. Ваша задача — не писать их заново, а понимать, как выбрать правильный инструмент и адаптировать его под конкретную задачу.

## Часть 1: Архитектура Toolkit'ов в LangChain

### Что такое Toolkit и зачем он нужен

Toolkit в терминологии LangChain — это класс, который предоставляет набор связанных инструментов через метод `get_tools()`:

```python
from langchain_community.agent_toolkits import FileManagementToolkit

file_toolkit = FileManagementToolkit(root_dir="/path/to/workspace")
tools = file_toolkit.get_tools()
# [ReadFileTool, WriteFileTool, CopyFileTool, DeleteFileTool, ...]
```

Toolkit инкапсулирует общую конфигурацию (рабочую директорию, API-ключ, подключение к БД) и создаёт согласованный набор инструментов, которые работают вместе.

### Три преимущества Toolkit'ов перед отдельными инструментами

**Согласованность конфигурации.** Все инструменты для работы с файлами получают одну и ту же `root_dir`. Все инструменты для работы с GitHub используют один токен авторизации. Нет риска, что один инструмент видит одну директорию, а другой — другую.

**Логическая группировка.** Когда вы добавляете FileManagementToolkit, вы получаете все операции с файлами разом — не нужно вспоминать, какие именно инструменты нужны.

**Кастомизация через selected_tools.** Toolkit позволяет выбрать подмножество инструментов, что критично для безопасности:

```python
# Только безопасные операции — чтение и копирование, без удаления
file_toolkit = FileManagementToolkit(
    root_dir="/workspace",
    selected_tools=["read_file", "copy_file", "list_directory"]
)
```

Это ключевой паттерн: давайте агенту **минимально необходимый** набор инструментов. Если задача не требует удаления файлов — не давайте `delete_file`. Принцип наименьших привилегий работает и для AI-агентов.

### Когда создавать свой Toolkit vs использовать готовый

Решение зависит от зрелости вашего проекта и специфичности задачи:

| Ситуация | Рекомендация |
|----------|-------------|
| Стандартная задача (поиск, файлы, SQL) | Используйте готовый Toolkit |
| Корпоративная система (Jira, CRM, ERP) | Создавайте свой Toolkit |
| Прототип / MVP | Готовые инструменты + @tool для кастомных |
| Продакшен с особыми требованиями | Свой Toolkit с продуманной безопасностью |

Общее правило: сначала ищите готовое решение. Создавайте своё только когда готовое не подходит — и всегда на основе архитектуры BaseToolkit, чтобы сохранить совместимость с экосистемой.

## Часть 2: Поисковые интеграции — Глаза агента в интернете

### Ландшафт поисковых решений

Поиск — одна из самых востребованных возможностей агента. Но "поисковый инструмент" — понятие растяжимое. Одному агенту нужны свежие новости, другому — структура SERP (Search Engine Results Page), третьему — извлечённый контент страницы. Для каждого случая — свой инструмент.

### Сравнение поисковых решений

| Характеристика | Tavily | SerpAPI | Serper | DuckDuckGo |
|----------------|--------|---------|--------|------------|
| **Цель** | AI-агенты | Парсинг SERP | Только Google | Бесплатный поиск |
| **Что возвращает** | Извлечённый контент | Структуру результатов | Структуру результатов | Текст результатов |
| **Цена** | ~$0.01/запрос | $0.001-0.05/запрос | ~$0.001/запрос | Бесплатно |
| **AI-ответ** | Да | Нет | Нет | Нет |
| **Полный текст страниц** | Да | Нет | Нет | Нет |
| **Локализация** | Ограничена | Полная | Полная | Ограничена |
| **API-ключ** | Требуется | Требуется | Требуется | Не нужен |

### Tavily — AI-оптимизированный поиск

Tavily — это поисковый движок, специально созданный для AI-агентов. В отличие от Google или Bing, которые возвращают ссылки, Tavily возвращает *извлечённый контент* — уже готовый к использованию текст. Режим `search_depth="advanced"` глубже анализирует страницы и может включать AI-генерированный ответ на основе найденного.

```python
from langchain_tavily import TavilySearch

search = TavilySearch(max_results=5, search_depth="advanced", include_answer=True)
result = search.invoke({"query": "последние новости о квантовых компьютерах"})
```

Для большинства AI-агентов Tavily — лучший выбор "из коробки", потому что его результаты уже подготовлены для подачи в LLM: компактные, релевантные, содержащие контент а не просто ссылки.

### Когда нужны SerpAPI, Serper или DuckDuckGo

**SerpAPI** — когда вам нужна точная структура результатов Google (позиции, featured snippets, knowledge graph), мультиязычность и поддержка различных поисковых систем (Google, Bing, Yahoo, Yandex). Самый дорогой, но самый полный.

**Serper** — бюджетная альтернатива SerpAPI, специализирующаяся на Google. Поддерживает типы поиска: `"search"`, `"news"`, `"images"`, `"places"`. Отличный выбор, если нужен именно Google с локализацией.

**DuckDuckGo** — не требует API-ключа, идеален для прототипирования и некоммерческих проектов. Качество ниже, лимиты строже, но для начала экспериментов — достаточно.

## Часть 3: File Management — Агент с доступом к файловой системе

### Возможности и риски

FileManagementToolkit предоставляет полный набор операций: чтение, запись, копирование, удаление, поиск файлов, просмотр директорий. Это мощно, но и опасно — агент с доступом к файловой системе может случайно (или под воздействием prompt injection) повредить данные.

```python
from langchain_community.agent_toolkits import FileManagementToolkit

toolkit = FileManagementToolkit(
    root_dir="/home/user/workspace",
    selected_tools=["read_file", "write_file", "list_directory"]
)
```

Критически важен параметр `root_dir` — он ограничивает рабочую область агента. Попытка обратиться к файлу за пределами этой директории (например, через `../../../etc/passwd`) будет заблокирована с ошибкой `ValueError: Path is outside root directory`.

### Четыре правила безопасности файловых операций

1. **Всегда ограничивайте root_dir** — агент не должен иметь доступ ко всей файловой системе. Даже если вам "нужен доступ везде" — создайте символическую ссылку на нужные директории внутри workspace.

2. **Выбирайте минимум операций** — если агенту не нужно удалять файлы, не давайте `delete_file`. Это не паранойя, а принцип наименьших привилегий.

3. **Логируйте все операции записи** — для аудита и отладки. Callback-хэндлер, отслеживающий `write_file`, `delete_file`, `copy_file`, поможет понять, что именно агент делал с файлами.

4. **Рассмотрите sandbox** — для особо чувствительных случаев запускайте агента в Docker-контейнере с read-only файловой системой, монтируя только нужные директории.

### Поиск по содержимому файлов

FileManagementToolkit не включает grep-подобную функциональность. Для поиска внутри файлов нужен отдельный инструмент — обёртка над `subprocess.run(["grep", ...])` с обязательными ограничениями: `cwd` на рабочую директорию и `timeout` на время выполнения. Полная реализация — в файле примеров.

## Часть 4: Python REPL — Code Interpreter

### Агент, который умеет программировать

Python REPL (Read-Eval-Print Loop) — один из самых мощных инструментов агента. Он позволяет выполнять произвольный Python-код: математические вычисления, обработку данных с Pandas, генерацию графиков, работу с API.

```python
from langchain_experimental.tools import PythonREPLTool

python_repl = PythonREPLTool()
result = python_repl.invoke("import math; print(f'Площадь круга: {math.pi * 5**2:.2f}')")
# "Площадь круга: 78.54"
```

Это паттерн **Code Interpreter**, популяризированный ChatGPT: модель генерирует код для решения задачи, REPL выполняет его, результат возвращается модели для интерпретации.

### Безопасность: три уровня защиты

Выполнение произвольного кода — это огромный риск. Модель может (случайно или под воздействием prompt injection) выполнить вредоносный код: удалить файлы, отправить данные на внешний сервер, запустить бесконечный цикл.

**Уровень 1: Ограничение модулей.** Передайте в REPL только разрешённые модули через `_globals`: `math`, `statistics`, `json`. Попытка импортировать `os` или `subprocess` будет заблокирована.

**Уровень 2: Таймауты и лимиты.** Ограничьте время выполнения (`timeout=30`), потребление памяти и CPU. Бесконечный цикл `while True` не должен подвешивать весь сервис.

**Уровень 3: Docker-sandbox.** Для production — выполнение в изолированном Docker-контейнере с `network_disabled=True`, ограничениями памяти (`mem_limit="256m"`) и CPU. Контейнер создаётся для каждого вызова и уничтожается после выполнения. Это самый надёжный, но и самый медленный подход (overhead на создание контейнера ~1-2 секунды).

Выбор уровня зависит от контекста: для внутреннего инструмента разработчика достаточно уровня 1-2. Для публичного сервиса — только уровень 3.

### Pandas и визуализация

Одно из главных применений Python REPL — анализ данных. LangChain предоставляет специализированный агент `create_pandas_dataframe_agent`, который автоматически понимает структуру DataFrame, генерирует Pandas-код для ответа на вопрос и интерпретирует результат.

```python
from langchain_experimental.agents.agent_toolkits import create_pandas_dataframe_agent

agent = create_pandas_dataframe_agent(
    llm=llm, df=df, agent_type="tool-calling",
    allow_dangerous_code=True  # Явное подтверждение
)
result = agent.invoke("Какой продукт принёс наибольшую выручку в Q3?")
```

Агент может не только анализировать, но и визуализировать данные — генерируя Matplotlib-код для графиков. Полные примеры (создание графиков, работа с несколькими DataFrame, sandbox-выполнение) в файле примеров.

## Часть 5: Shell и системные команды

### Осторожный доступ к командной строке

Иногда агенту нужно выполнять shell-команды: работать с git, запускать скрипты, управлять процессами. LangChain предоставляет `ShellTool`, но давать агенту неограниченный доступ к shell — рецепт катастрофы. Агент с полным shell-доступом может выполнить `rm -rf /`, и ни один промпт не гарантирует, что он этого не сделает.

### Паттерн: специализированные команды вместо generic shell

Вместо одного "универсального" shell-инструмента создавайте отдельные инструменты для конкретных операций. Для git: `git_status`, `git_diff`, `git_commit`. Для системы: `disk_usage`, `process_list`. Каждый инструмент — обёртка над `subprocess.run()` с захардкоженной командой, рабочей директорией и таймаутом.

Преимущества:
- Модель видит конкретные инструменты с чёткими описаниями, а не абстрактный "shell"
- Невозможно выполнить произвольную команду
- Каждый инструмент можно тестировать и логировать отдельно

### Паттерн: белый список команд

Для более гибкого подхода можно создать shell с белым списком разрешённых команд и аргументов:

```python
ALLOWED_COMMANDS = {
    "ls": {"args": ["-la", "-l", "-a"]},
    "cat": {"args": []},
    "head": {"args": ["-n"]},
    "grep": {"args": ["-i", "-n", "-r"]},
}

@tool
def safe_shell(command: str) -> str:
    """Выполняет безопасные shell-команды. Доступны: ls, cat, head, grep."""
    parts = shlex.split(command)
    cmd_name = parts[0]
    if cmd_name not in ALLOWED_COMMANDS:
        return f"Команда '{cmd_name}' не разрешена. Доступны: {list(ALLOWED_COMMANDS.keys())}"
    # ... валидация аргументов и выполнение с timeout
```

Белый список проверяет и саму команду, и допустимые флаги. Агент может выполнить `grep -i pattern file.txt`, но не `grep --include='*.py' -r / -e "password"`.

## Часть 6: Комбинирование Toolkit'ов

### Проблема: слишком много инструментов

В реальных приложениях агенту часто нужен доступ к нескольким категориям инструментов: поиск + файлы + код + кастомные. Собрать их вместе просто — `all_tools = search_tools + file_tools + code_tools`. Но возникает проблема: чем больше инструментов, тем сложнее модели выбрать правильный.

Исследования показывают, что качество выбора заметно снижается, когда доступно более **10-15 инструментов**. Модель начинает путать семантически близкие инструменты, выбирать "не тот" или вызывать инструменты, которые не нужны.

### Три стратегии управления большим арсеналом

**Стратегия 1: Категоризация в системном промпте.** Явно сгруппируйте инструменты и объясните, когда использовать каждую категорию:

```
У тебя есть инструменты трёх категорий:
ПОИСК: tavily_search — для актуальной информации из интернета
ФАЙЛЫ: read_file, write_file — для работы с документами
КОД: python_repl — для вычислений и обработки данных
Выбирай категорию, затем инструмент внутри неё.
```

**Стратегия 2: Иерархический выбор.** Создайте "мета-инструмент", который рекомендует нужный toolkit. Модель сначала вызывает `choose_toolkit("search")`, получает рекомендацию, затем вызывает конкретный инструмент. Это добавляет один round-trip, но повышает точность выбора.

**Стратегия 3: Мульти-агентная архитектура.** Если инструментов больше 15-20, рассмотрите разделение на специализированных агентов: "поисковый агент", "файловый агент", "аналитический агент". Оркестратор-агент решает, какому специалисту делегировать задачу. Это тема следующих модулей курса, но важно знать о ней уже сейчас — при проектировании.

### Принципы комбинирования

При сборке набора инструментов руководствуйтесь тремя принципами:

1. **Минимальная достаточность** — добавляйте только инструменты, необходимые для конкретной задачи. Не "на всякий случай".

2. **Непересекающиеся описания** — если два инструмента делают похожие вещи, их описания должны чётко разграничивать области применения. "Используй search_web для новостей, используй search_database для корпоративных данных".

3. **Согласованность** — все инструменты должны возвращать данные в совместимом формате. Если один инструмент возвращает JSON, а другой — свободный текст, модель будет путаться.

## Часть 7: Создание собственных Toolkit'ов

### Когда и зачем создавать свой Toolkit

Создание собственного Toolkit оправдано, когда у вас есть набор связанных инструментов, которые:
- Разделяют общую конфигурацию (URL, токен, проект)
- Используются вместе как единый набор
- Нуждаются в едином управлении доступом (selected_tools)

### Архитектура кастомного Toolkit

Кастомный Toolkit наследует от `BaseToolkit` и реализует метод `get_tools()`:

```python
from langchain_core.tools import BaseTool, BaseToolkit, StructuredTool
from pydantic import Field
from typing import List

class JiraToolkit(BaseToolkit):
    """Toolkit для работы с Jira."""

    base_url: str = Field(description="URL Jira-сервера")
    api_token: str = Field(description="API токен")
    project_key: str = Field(description="Ключ проекта")
    selected_tools: List[str] = Field(
        default=["search", "get_issue"],  # По умолчанию только чтение
    )

    def get_tools(self) -> List[BaseTool]:
        all_tools = {
            "create_issue": self._create_issue_tool(),
            "search": self._search_tool(),
            "get_issue": self._get_issue_tool(),
        }
        return [all_tools[name] for name in self.selected_tools if name in all_tools]
```

Ключевые решения при проектировании:

**Безопасность по умолчанию.** `selected_tools` по умолчанию включает только read-операции. Пользователь должен явно добавить `"create_issue"` или `"delete_issue"` — это осознанное действие.

**Инкапсуляция конфигурации.** `base_url`, `api_token` передаются один раз в конструкторе и доступны всем инструментам через `self`. Каждый внутренний метод (`_create_issue_tool`, `_search_tool`) создаёт StructuredTool с замыканием на `self`.

**Расширяемость.** Новые инструменты добавляются как новые методы `_xxx_tool()` и регистрируются в словаре `all_tools`. Существующий код не меняется.

### Паттерн: Toolkit для корпоративных систем

В корпоративном контексте Toolkit'ы часто оборачивают внутренние API: CRM, ERP, системы тикетов, хранилища документов. При проектировании таких Toolkit'ов учитывайте:

- **Аудит** — логируйте каждый вызов с указанием пользователя (через RunnableConfig)
- **Rate limiting** — корпоративные API часто имеют жёсткие лимиты
- **Кэширование** — одни и те же данные (список проектов, структура организации) запрашиваются многократно
- **Обработка ошибок** — корпоративные API могут быть нестабильны, все инструменты должны иметь graceful degradation

## Заключение: Экосистема как суперсила

Готовые интеграции — это не костыль для ленивых, а умное использование коллективного опыта сообщества. Tavily оптимизирован для AI, PythonREPL прошёл тысячи edge cases, FileManagementToolkit учитывает безопасность. Ваша задача — выбрать правильные инструменты и адаптировать их под задачу.

Ключевые принципы:

1. **Начинайте с готового** — прежде чем писать своё, проверьте, нет ли уже готового решения.

2. **Ограничивайте возможности** — используйте `selected_tools`, `root_dir`, белые списки команд. Принцип наименьших привилегий — не рекомендация, а требование.

3. **Комбинируйте осмысленно** — группируйте инструменты по задачам, ограничивайте общее количество до 10-15.

4. **Думайте о безопасности** — особенно для файловых операций и выполнения кода. Sandbox — не роскошь, а необходимость для production.

В следующей лекции мы погрузимся в одну из самых мощных интеграций — работу с базами данных через SQL. Это открывает агентам доступ к структурированным корпоративным данным, но требует особого внимания к безопасности и оптимизации.

## Ключевые термины

| Термин | Определение |
|--------|-------------|
| **Toolkit** | Класс LangChain, предоставляющий набор связанных инструментов через метод get_tools() |
| **selected_tools** | Параметр Toolkit для выбора подмножества доступных инструментов (принцип наименьших привилегий) |
| **Tavily** | AI-оптимизированный поисковый движок, возвращающий извлечённый контент вместо ссылок |
| **SerpAPI/Serper** | Сервисы программного доступа к результатам Google и других поисковиков |
| **Python REPL** | Инструмент для выполнения Python-кода в контексте агента (паттерн Code Interpreter) |
| **FileManagementToolkit** | Toolkit для операций с файловой системой с ограничением рабочей директории (root_dir) |
| **Sandbox** | Изолированная среда выполнения кода (например, Docker-контейнер) для защиты от вредоносного кода |
| **Белый список команд** | Паттерн безопасности: только явно разрешённые shell-команды доступны агенту |
| **Мульти-агентная архитектура** | Подход с несколькими специализированными агентами для управления большим набором инструментов |
