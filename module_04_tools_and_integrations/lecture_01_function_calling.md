# Лекция 1: Function Calling — Когда слова обретают силу действия

## Введение: Переломный момент в истории ИИ

Представьте себе человека, запертого в комнате без окон и дверей. У него феноменальная память, он прочитал миллиарды книг, может рассуждать о квантовой физике и средневековой поэзии одновременно. Но попросите его включить свет — и он бессилен. Попросите узнать погоду за окном — и он может лишь гадать. Попросите отправить письмо — и всё, что он способен сделать, это красиво описать, как он *хотел бы* это сделать.

Именно так выглядели языковые модели до появления механизма Function Calling. GPT-3, каким бы впечатляющим он ни был, представлял собой гениального узника собственного текстового мира. Он мог генерировать поэзию, писать код, объяснять теорию относительности — но взаимодействовать с реальным миром был неспособен. Вы спрашивали его о текущем курсе доллара — он либо выдавал устаревшие данные из своих "воспоминаний", либо честно признавался в неведении, либо, что хуже всего, уверенно галлюцинировал случайное число.

Function Calling — это тот самый ключ, который отпер двери темницы. Это механизм, благодаря которому языковая модель перестала быть просто генератором текста и превратилась в нечто принципиально иное: в *агента*, способного воздействовать на окружающий мир. Модель теперь может не просто рассказать вам о погоде — она может запросить актуальные данные у метеослужбы. Не просто описать идеальный маршрут путешествия — но забронировать билеты. Не просто объяснить, как работает SQL — но выполнить реальный запрос к базе данных.

Это фундаментальный сдвиг парадигмы, сравнимый с тем моментом, когда программы перестали быть просто калькуляторами и получили доступ к файловой системе, сети, периферийным устройствам. Компьютер, умеющий только считать, полезен. Компьютер, умеющий считать *и* сохранять результаты, взаимодействовать с пользователем, подключаться к интернету — это уже совершенно другой уровень полезности. То же самое происходит сейчас с языковыми моделями: они эволюционируют из впечатляющих, но изолированных систем в связанные с миром интеллектуальные агенты.

## Часть 1: Как работает Function Calling под капотом

### Специальные токены — тайный язык машин

Чтобы понять Function Calling, нужно сначала осознать, как вообще "мыслит" языковая модель. В своей основе она делает одну простую вещь: предсказывает следующий токен в последовательности. Токен — это не совсем слово, а скорее фрагмент текста, который модель воспринимает как единую единицу. Английское слово "function" — один токен, русское "функция" — возможно, два или три, в зависимости от токенизатора. Знаки пунктуации, пробелы, даже части слов могут быть отдельными токенами.

Когда модель обучают поддержке Function Calling, в её словарь добавляются специальные токены — своего рода "служебные слова", которых нет в обычном языке. Эти токены сигнализируют: "сейчас я буду не просто говорить, а *вызывать функцию*". У разных провайдеров они выглядят по-разному. У OpenAI это может быть что-то вроде особой структуры в JSON-ответе. У Anthropic — специальный формат tool_use внутри сообщения.

Представьте, что вы общаетесь с очень умным, но не говорящим на вашем языке иностранцем через переводчика. Обычно переводчик передаёт вам слова собеседника как есть. Но иногда иностранец делает особый жест рукой — и переводчик понимает: "Это не слова, это просьба принести чай". Function Calling работает похоже: специальные токены — это жесты, которые переводчик (ваш код) должен интерпретировать как команды к действию.

### Обучение модели вызывать функции

Способность к Function Calling не появляется сама собой. Модель нужно специально обучить (fine-tune) на примерах того, когда и как следует вызывать функции. Обучающая выборка содержит тысячи примеров диалогов, где человек просит что-то, что требует внешнего действия, и модель корректно формирует вызов функции вместо текстового ответа.

Этот процесс можно представить как обучение ребёнка пользоваться телефоном. Сначала вы показываете примеры: "Когда хочешь узнать, работает ли магазин, звонишь по этому номеру и спрашиваешь время работы". "Когда потерял кошку — звонишь в приют и описываешь её". После сотен таких примеров ребёнок начинает понимать закономерность: есть ситуации, когда говорить недостаточно, нужно *действовать* — и действовать определённым образом.

Критически важно то, что модель учится не только *когда* вызывать функцию, но и *какие аргументы* ей передавать. Если пользователь спрашивает "Какая погода в Москве?", модель должна понять, что нужно вызвать функцию get_weather с аргументом city="Moscow" (или "Москва", в зависимости от API). Если пользователь спрашивает "Будет ли завтра дождь в Питере?", модель должна понять, что нужна та же функция, но с аргументами city="Saint Petersburg" и возможно date="tomorrow".

### JSON Schema — контракт между моделью и кодом

Но как модель узнаёт, какие функции вообще существуют и какие у них параметры? Через JSON Schema — стандартизированный способ описания структуры данных. Перед вызовом модели вы передаёте ей описания всех доступных функций в специальном формате.

Рассмотрим пример. Вы хотите дать модели возможность искать информацию в интернете. Описание такой функции может выглядеть так:

```json
{
  "name": "web_search",
  "description": "Поиск информации в интернете. Используй, когда нужны актуальные данные, новости, или информация, которой может не быть в твоей базе знаний.",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {
        "type": "string",
        "description": "Поисковый запрос на английском языке"
      },
      "num_results": {
        "type": "integer",
        "description": "Количество результатов (от 1 до 10)",
        "default": 5
      }
    },
    "required": ["query"]
  }
}
```

Это как инструкция по использованию прибора. Поле "name" — название функции, которое модель укажет при вызове. Поле "description" — критически важное объяснение, *когда* и *зачем* использовать эту функцию. Модель читает это описание и на его основе решает, подходит ли функция для текущей задачи. Блок "parameters" описывает, какие данные функция принимает: типы, допустимые значения, какие параметры обязательны, а какие имеют значения по умолчанию.

Качество этого описания напрямую влияет на то, насколько умно модель будет пользоваться функцией. Расплывчатое описание — расплывчатое поведение. Если написать "Поиск в интернете", модель может вызывать функцию когда угодно, даже когда это не нужно. Если написать "Поиск в интернете. Используй ТОЛЬКО когда пользователь явно просит найти что-то актуальное, чего ты точно не знаешь" — поведение будет точнее.

## Часть 2: OpenAI Functions API — стандарт де-факто

### Эволюция API: от Chat Completions к Tools

OpenAI первой среди крупных провайдеров внедрила поддержку Function Calling в июне 2023 года, и это событие разделило историю AI-приложений на "до" и "после". Изначально механизм назывался просто "Functions", позже эволюционировал в "Tools" — более общую концепцию, охватывающую не только вызовы функций.

Давайте проследим, как выглядит вызов API с включённым Function Calling:

```python
from openai import OpenAI

client = OpenAI()

tools = [
    {
        "type": "function",
        "function": {
            "name": "get_current_weather",
            "description": "Получить текущую погоду в указанном городе",
            "parameters": {
                "type": "object",
                "properties": {
                    "location": {
                        "type": "string",
                        "description": "Город, например 'Москва' или 'Paris'"
                    },
                    "unit": {
                        "type": "string",
                        "enum": ["celsius", "fahrenheit"],
                        "description": "Единица измерения температуры"
                    }
                },
                "required": ["location"]
            }
        }
    }
]

messages = [
    {"role": "user", "content": "Какая сейчас погода в Токио?"}
]

response = client.chat.completions.create(
    model="gpt-5-mini",
    messages=messages,
    tools=tools,
    tool_choice="auto"  # Модель сама решает, вызывать ли функцию
)
```

Обратите внимание на параметр tool_choice. Значение "auto" означает: "Модель, решай сама — нужна ли здесь функция или можно ответить текстом". Можно указать "none", чтобы запретить вызов функций, или "required", чтобы принудить модель использовать одну из них. Можно даже указать конкретную функцию, которую модель *обязана* вызвать.

### Анатомия ответа модели

Когда модель решает вызвать функцию, ответ выглядит особым образом:

```python
# response.choices[0].message содержит:
{
    "role": "assistant",
    "content": null,  # Текстового содержимого нет!
    "tool_calls": [
        {
            "id": "call_abc123",
            "type": "function",
            "function": {
                "name": "get_current_weather",
                "arguments": "{\"location\": \"Tokyo\", \"unit\": \"celsius\"}"
            }
        }
    ]
}
```

Здесь важно несколько моментов. Поле content равно null — модель не пишет текст, когда вызывает функцию. Вместо этого она заполняет tool_calls — массив вызовов. Да, именно массив! Модель может решить, что для ответа на вопрос нужно вызвать *несколько* функций параллельно. Например, на вопрос "Сравни погоду в Москве и Лондоне" модель может вернуть два вызова get_current_weather с разными городами.

Каждый вызов имеет уникальный id — это важно для последующей обработки, когда мы будем возвращать результаты модели. Аргументы функции — это JSON-строка, которую нужно распарсить.

### Цикл Thought-Action-Observation

Получив вызов функции от модели, наш код должен выполнить реальную работу: запросить API погоды, сходить в базу данных, выполнить вычисление — что угодно. Затем результат нужно вернуть модели для финального ответа. Это формирует цикл, который часто называют "Thought-Action-Observation" или "ReAct loop".

```python
import json

# Шаг 1: Модель "думает" и решает вызвать функцию (уже сделано выше)

# Шаг 2: Мы исполняем функцию
def get_current_weather(location, unit="celsius"):
    # В реальности здесь был бы запрос к API погоды
    # Для примера возвращаем заглушку
    return {
        "location": location,
        "temperature": 22,
        "unit": unit,
        "condition": "sunny"
    }

tool_call = response.choices[0].message.tool_calls[0]
function_name = tool_call.function.name
function_args = json.loads(tool_call.function.arguments)

# Выполняем функцию
result = get_current_weather(**function_args)

# Шаг 3: Возвращаем результат модели
messages.append(response.choices[0].message)  # Сообщение ассистента с tool_calls
messages.append({
    "role": "tool",
    "tool_call_id": tool_call.id,
    "content": json.dumps(result)
})

# Шаг 4: Модель генерирует финальный ответ
final_response = client.chat.completions.create(
    model="gpt-5-mini",
    messages=messages,
    tools=tools
)

print(final_response.choices[0].message.content)
# "Сейчас в Токио 22°C и солнечно!"
```

Этот цикл может повторяться несколько раз. Модель может понять, что одного вызова недостаточно, и запросить ещё один. Или результат первого вызова может подсказать, что нужен второй. Например, сначала поиск информации, затем — уточняющий запрос на основе найденного.

## Часть 3: Anthropic Tools — альтернативный подход

### Философия Claude

Anthropic, создатели Claude, пришли к Function Calling немного позже OpenAI, но их реализация имеет свои особенности. Если OpenAI делала акцент на максимальной гибкости и совместимости с существующими практиками разработки, Anthropic сфокусировалась на безопасности и предсказуемости.

В терминологии Anthropic механизм называется "Tool Use", а не "Function Calling". Это не просто разница в названиях — это отражение философии: инструмент (tool) — это что-то, чем агент *пользуется*, а не *часть* агента. Это подчёркивает разделение между интеллектом модели и её способностью воздействовать на мир.

### Структура запроса и ответа

```python
import anthropic

client = anthropic.Anthropic()

tools = [
    {
        "name": "get_stock_price",
        "description": "Получить текущую цену акции по тикеру. Используй для вопросов о стоимости акций публичных компаний.",
        "input_schema": {
            "type": "object",
            "properties": {
                "ticker": {
                    "type": "string",
                    "description": "Биржевой тикер акции, например 'AAPL' для Apple"
                }
            },
            "required": ["ticker"]
        }
    }
]

response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=tools,
    messages=[
        {"role": "user", "content": "Сколько сейчас стоит акция Tesla?"}
    ]
)
```

Ответ Claude структурирован иначе, чем у OpenAI:

```python
# response.content может содержать несколько блоков:
[
    {
        "type": "text",
        "text": "Я проверю текущую цену акций Tesla для вас."
    },
    {
        "type": "tool_use",
        "id": "toolu_01XYZ",
        "name": "get_stock_price",
        "input": {"ticker": "TSLA"}
    }
]
```

Обратите внимание на критическое отличие: Claude может *одновременно* писать текст и вызывать функцию! Модель сначала объясняет, что собирается сделать ("Я проверю текущую цену..."), а затем вызывает инструмент. Это делает поведение более прозрачным и предсказуемым для пользователя — он видит "мыслительный процесс" агента.

### Обработка результатов в Claude

Возвращение результатов инструмента тоже немного отличается:

```python
# Добавляем сообщение ассистента как есть
messages.append({
    "role": "assistant",
    "content": response.content
})

# Добавляем результат инструмента
messages.append({
    "role": "user",
    "content": [
        {
            "type": "tool_result",
            "tool_use_id": "toolu_01XYZ",
            "content": json.dumps({"price": 248.50, "currency": "USD", "change": "+2.3%"})
        }
    ]
})

# Получаем финальный ответ
final_response = client.messages.create(
    model="claude-sonnet-4-6",
    max_tokens=1024,
    tools=tools,
    messages=messages
)
```

## Часть 4: Конвертация Python-функций в JSON Schema

### От кода к описанию

В реальных проектах вы не хотите вручную писать JSON Schema для каждой функции — это утомительно и чревато ошибками. К счастью, современные фреймворки умеют автоматически конвертировать Python-функции вместе с их type hints и docstrings в нужный формат.

Вот как выглядит "правильно оформленная" функция:

```python
def calculate_mortgage(
    principal: float,
    annual_rate: float,
    years: int,
    down_payment: float = 0.0
) -> dict:
    """
    Рассчитать ежемесячный платёж по ипотеке.

    Используй эту функцию, когда пользователь спрашивает о расчёте
    ипотечного кредита, ежемесячных платежей или сравнении условий.

    Args:
        principal: Стоимость недвижимости в рублях
        annual_rate: Годовая процентная ставка (например, 12.5 для 12.5%)
        years: Срок кредита в годах
        down_payment: Первоначальный взнос в рублях (по умолчанию 0)

    Returns:
        Словарь с суммой кредита, ежемесячным платежом и общей переплатой
    """
    loan_amount = principal - down_payment
    monthly_rate = annual_rate / 100 / 12
    num_payments = years * 12

    # Формула аннуитетного платежа
    if monthly_rate > 0:
        monthly_payment = loan_amount * (monthly_rate * (1 + monthly_rate)**num_payments) / ((1 + monthly_rate)**num_payments - 1)
    else:
        monthly_payment = loan_amount / num_payments

    total_paid = monthly_payment * num_payments

    return {
        "loan_amount": round(loan_amount, 2),
        "monthly_payment": round(monthly_payment, 2),
        "total_paid": round(total_paid, 2),
        "overpayment": round(total_paid - loan_amount, 2)
    }
```

Docstring здесь играет двойную роль: для разработчиков это документация, для LLM — инструкция по применению. Первый абзац становится описанием функции. Блок Args превращается в описания параметров. Type hints определяют типы в JSON Schema.

### Автоматическая конвертация

LangChain и другие фреймворки предоставляют утилиты для автоматической конвертации:

```python
from langchain_core.utils.function_calling import convert_to_openai_tool

# Магическое превращение Python-функции в JSON Schema (формат Tools API)
openai_tool = convert_to_openai_tool(calculate_mortgage)

# Результат:
{
    "name": "calculate_mortgage",
    "description": "Рассчитать ежемесячный платёж по ипотеке.\n\nИспользуй эту функцию...",
    "parameters": {
        "type": "object",
        "properties": {
            "principal": {
                "type": "number",
                "description": "Стоимость недвижимости в рублях"
            },
            "annual_rate": {
                "type": "number",
                "description": "Годовая процентная ставка (например, 12.5 для 12.5%)"
            },
            "years": {
                "type": "integer",
                "description": "Срок кредита в годах"
            },
            "down_payment": {
                "type": "number",
                "description": "Первоначальный взнос в рублях (по умолчанию 0)"
            }
        },
        "required": ["principal", "annual_rate", "years"]
    }
}
```

Обратите внимание: параметр down_payment не попал в required, потому что у него есть значение по умолчанию. Конвертер умный — он извлекает максимум информации из кода.

### Pydantic — ещё более мощный подход

Для сложных структур данных Pydantic предоставляет ещё более выразительный способ описания:

```python
from pydantic import BaseModel, Field
from typing import Literal
from enum import Enum

class PropertyType(str, Enum):
    APARTMENT = "apartment"
    HOUSE = "house"
    COMMERCIAL = "commercial"

class MortgageInput(BaseModel):
    """Входные параметры для расчёта ипотеки"""

    principal: float = Field(
        ...,  # Обязательный параметр
        description="Стоимость недвижимости в рублях",
        gt=0,  # Валидация: больше нуля
        examples=[5000000, 10000000]
    )
    annual_rate: float = Field(
        ...,
        description="Годовая процентная ставка в процентах",
        ge=0,
        le=100
    )
    years: int = Field(
        ...,
        description="Срок кредита в годах",
        ge=1,
        le=30
    )
    down_payment: float = Field(
        default=0.0,
        description="Первоначальный взнос в рублях",
        ge=0
    )
    property_type: PropertyType = Field(
        default=PropertyType.APARTMENT,
        description="Тип недвижимости"
    )

# Pydantic-модель автоматически конвертируется в JSON Schema
schema = MortgageInput.model_json_schema()
```

Pydantic добавляет несколько суперспособностей. Во-первых, валидация — мы указываем, что principal должен быть больше нуля, years — от 1 до 30 и так далее. Если модель вызовет функцию с невалидными аргументами, Pydantic поймает ошибку до выполнения реальной логики. Во-вторых, примеры (examples) помогают модели лучше понять, какие значения ожидаются. В-третьих, Enum для property_type превращается в enum в JSON Schema — модель будет знать, что допустимы только три значения.

## Часть 5: Параллельный вызов функций

### Когда одного действия недостаточно

Одна из мощнейших возможностей современных моделей — способность планировать и запрашивать *несколько* действий одновременно. Вместо последовательного цикла "вызвал функцию → получил результат → вызвал ещё одну" модель может в одном ответе запросить параллельное выполнение нескольких функций.

Рассмотрим сценарий: пользователь спрашивает "Сравни погоду в Москве, Париже и Токио". Без параллельного вызова модель должна была бы:

1. Вызвать get_weather("Moscow") → получить результат
2. Вызвать get_weather("Paris") → получить результат
3. Вызвать get_weather("Tokyo") → получить результат
4. Сформировать сравнительный ответ

Это четыре round-trip к модели! С параллельным вызовом всё происходит за два шага:

1. Модель возвращает три вызова get_weather одновременно
2. Мы исполняем все три параллельно, возвращаем результаты, модель формирует ответ

```python
# Ответ модели содержит несколько tool_calls:
{
    "tool_calls": [
        {
            "id": "call_1",
            "function": {"name": "get_weather", "arguments": "{\"city\": \"Moscow\"}"}
        },
        {
            "id": "call_2",
            "function": {"name": "get_weather", "arguments": "{\"city\": \"Paris\"}"}
        },
        {
            "id": "call_3",
            "function": {"name": "get_weather", "arguments": "{\"city\": \"Tokyo\"}"}
        }
    ]
}
```

### Асинхронное исполнение

Чтобы воспользоваться параллелизмом по-настоящему, наш код тоже должен исполнять функции параллельно:

```python
import asyncio
import aiohttp

async def get_weather_async(city: str) -> dict:
    """Асинхронный запрос к API погоды"""
    async with aiohttp.ClientSession() as session:
        async with session.get(f"https://api.weather.com/{city}") as resp:
            return await resp.json()

async def execute_tool_calls(tool_calls):
    """Параллельное исполнение всех вызовов"""
    tasks = []
    for call in tool_calls:
        args = json.loads(call.function.arguments)
        if call.function.name == "get_weather":
            tasks.append(get_weather_async(**args))

    # asyncio.gather исполняет все задачи параллельно
    results = await asyncio.gather(*tasks)

    # Сопоставляем результаты с id вызовов
    return [
        {"tool_call_id": call.id, "result": result}
        for call, result in zip(tool_calls, results)
    ]
```

Разница в производительности может быть колоссальной. Если каждый API-вызов занимает 200 мс, последовательное исполнение трёх вызовов займёт 600 мс. Параллельное — те же 200 мс (плюс минимальный overhead на координацию).

## Часть 6: Сравнительный анализ API разных провайдеров

### Таблица: OpenAI vs Anthropic vs Open Source

| Характеристика | OpenAI (GPT-5) | Anthropic (Claude) | Llama 4 / Mistral |
|----------------|----------------|-------------------|-------------------|
| **Название механизма** | Tools / Function Calling | Tool Use | Varies by implementation |
| **Формат описания** | JSON Schema | JSON Schema | JSON Schema |
| **Параллельные вызовы** | Да | Да | Зависит от реализации |
| **Текст + вызов одновременно** | Да (зависит от модели) | Да | Зависит от реализации |
| **Структурированный вывод** | JSON Mode, Structured Outputs | Tool Use с Pydantic | Varies |
| **Обязательный вызов** | tool_choice: "required" | tool_choice: {"type": "any"} | Varies |
| **Максимум функций** | ~128 | Не задокументировано (на практике сотни) | Varies |
| **Стриминг вызовов** | Да | Да | Varies |

### Нюансы выбора провайдера

При выборе провайдера для приложения с Function Calling следует учитывать несколько факторов.

GPT-5 отличается наибольшей гибкостью и обширной экосистемой. Если вам нужно максимальное количество функций или сложная логика выбора между ними — это часто лучший выбор. Однако стоимость выше, и OpenAI известна изменениями в поведении модели между версиями.

Claude от Anthropic предлагает более предсказуемое и "объяснимое" поведение. Возможность одновременно писать текст и вызывать функции делает взаимодействие более естественным. Формальный лимит на количество инструментов не задокументирован, но на практике модели хорошо работают с десятками и сотнями инструментов.

Открытые модели (Llama, Mistral) привлекательны отсутствием платы за API и возможностью локального запуска. Однако качество Function Calling у них обычно ниже, и требуется дополнительная работа по fine-tuning или prompt engineering.

## Часть 7: Подводные камни и лучшие практики

### Галлюцинации в аргументах

Модели иногда "выдумывают" аргументы функций. Вы спрашиваете "Какая погода?", не уточняя город, а модель уверенно вызывает get_weather("New York") — просто потому, что ей нужно что-то указать. Это не галлюцинация в классическом смысле, но всё равно нежелательное поведение.

Защита: валидация аргументов на стороне кода, переспрос пользователя при неоднозначности, prompt engineering с инструкциями типа "Если пользователь не указал город, СПРОСИ, а не угадывай".

### Выбор "не той" функции

Когда доступно много функций, модель может выбрать семантически близкую, но неправильную. Функции get_user_info и get_user_settings могут казаться похожими — особенно если их описания недостаточно различаются.

Защита: чёткие, непересекающиеся описания функций. Используйте конкретные примеры в description: "Используй get_user_info для имени, email, даты регистрации. Используй get_user_settings для настроек уведомлений, языка интерфейса, темы оформления."

### Бесконечные циклы

Агент может войти в бесконечный цикл вызовов, особенно если логика сложная или результаты функций неинформативны. Модель вызывает функцию, получает непонятный результат, вызывает ещё раз с другими параметрами, снова непонятно, и так далее.

Защита: ограничение количества вызовов (max_iterations), мониторинг повторяющихся паттернов, улучшение информативности результатов функций.

### Безопасность

Function Calling — это способ выполнить произвольный код на основе решения LLM. Это открывает вектора атак: prompt injection может заставить модель вызвать опасную функцию или передать вредоносные аргументы.

Защита: принцип минимальных привилегий (функции имеют только необходимые права), валидация и санитизация аргументов, подтверждение критических действий человеком, логирование всех вызовов.

## Заключение: Функции как руки и ноги разума

Function Calling — это не просто техническая фича, это фундаментальный сдвиг в понимании того, чем может быть языковая модель. До Function Calling LLM были "мозгом в банке" — впечатляющим, но изолированным от мира. С Function Calling они получили возможность *действовать*.

Эта лекция заложила фундамент: вы теперь понимаете, как работает механизм под капотом (специальные токены, fine-tuning), как выглядят API разных провайдеров (OpenAI Functions, Anthropic Tools), как описывать функции через JSON Schema и Pydantic, как обрабатывать параллельные вызовы.

Но это только начало. Сами по себе функции — это просто провода, соединяющие модель с миром. Реальная магия происходит, когда эти провода подключены к чему-то полезному: к поисковым системам, базам данных, внешним API, файловым системам. Именно этому посвящены следующие лекции модуля.

В следующей лекции мы перейдём от теории к практике с LangChain — фреймворком, который превращает создание инструментов из ручной работы с JSON в элегантный декларативный код.

## Ключевые термины

| Термин | Определение |
|--------|-------------|
| **Function Calling** | Механизм, позволяющий LLM запрашивать выполнение внешних функций вместо генерации текста |
| **JSON Schema** | Стандарт описания структуры JSON-данных, используемый для определения параметров функций |
| **Tool Use** | Термин Anthropic для Function Calling, подчёркивающий разделение интеллекта и действий |
| **tool_calls** | Массив запросов на выполнение функций в ответе модели |
| **tool_choice** | Параметр, управляющий тем, обязана ли модель вызвать функцию |
| **Parallel Function Calling** | Способность модели запросить выполнение нескольких функций одновременно |
| **ReAct Loop** | Цикл "Reasoning + Acting" — итеративный процесс рассуждения и выполнения действий |
