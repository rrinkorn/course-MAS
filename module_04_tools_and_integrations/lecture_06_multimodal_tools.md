# Лекция 6: Мультимодальные инструменты — Когда агент обретает зрение и голос

## Введение: За пределами текста

На протяжении всего курса мы работали с агентами, которые существуют в мире текста. Они читают текст, думают текстом, отвечают текстом. Инструменты, которые мы создавали, принимают текстовые параметры и возвращают текстовые результаты. Это мощно, но это лишь часть картины.

Человек воспринимает мир через множество каналов одновременно: мы видим изображения, слышим звуки, читаем тексты, ощущаем текстуры. Наш интеллект — мультимодальный по своей природе. И современные AI-модели движутся в том же направлении.

GPT-5, Claude Sonnet 4.6, Gemini 2.5 — эти модели умеют не только читать текст, но и "видеть" изображения. GPT Image, Midjourney, Stable Diffusion — генерируют изображения по текстовому описанию. Модели gpt-4o-transcribe транскрибируют речь в текст, а модели вроде Eleven Labs синтезируют речь из текста. Соединив всё это вместе, мы получаем агента, который может взаимодействовать с миром так же богато, как человек.

Эта лекция посвящена мультимодальным инструментам — инструментам, которые работают с изображениями, аудио и видео. Мы научимся передавать изображения в модель для анализа, генерировать картинки как результат работы агента, распознавать речь и синтезировать её. Это финальный штрих, превращающий текстового ассистента в полноценного мультимодального агента.

## Часть 1: Vision — Агент, который видит

### Как модели "видят" изображения

Прежде чем создавать инструменты для работы с изображениями, важно понять, как современные Vision-модели обрабатывают визуальную информацию — потому что понимание механизма определяет и возможности, и ограничения.

В основе Vision-моделей лежит адаптация архитектуры трансформера. Изображение разбивается на **патчи** — небольшие квадратные фрагменты (обычно 14x14 или 16x16 пикселей). Каждый патч проходит через линейную проекцию и превращается в эмбеддинг — числовой вектор фиксированной размерности. Эти эмбеддинги поступают в трансформер наряду с текстовыми токенами.

Для модели изображение — это буквально ещё одна последовательность "токенов". Изображение 1024x1024 при размере патча 16x16 даёт 4096 визуальных токенов — столько же, сколько примерно 3000 слов текста. Это объясняет, почему работа с изображениями заметно дороже текста: вы платите за эти дополнительные токены.

Важно понимать, что Vision-модель не "видит" изображение так, как человек. Она оперирует статистическими паттернами, выученными на миллиардах пар "изображение-описание". Это приводит к характерным сильным и слабым сторонам.

### Что Vision-модели делают хорошо и плохо

**Сильные стороны:**

- **Распознавание объектов и сцен** — модели превосходно определяют, что изображено на фото: люди, животные, здания, еда, природа. Точность сопоставима с человеческой для стандартных сцен.
- **Чтение текста (OCR)** — современные модели хорошо считывают печатный текст на фотографиях, скриншотах, вывесках. Особенно при использовании `detail: "high"`.
- **Анализ графиков и диаграмм** — модели извлекают данные из bar chart, line chart, pie chart, интерпретируют тренды.
- **Понимание контекста** — модель может связать содержимое изображения с вопросом: "Безопасно ли здесь переходить дорогу?" про фото перекрёстка.

**Слабые стороны:**

- **Подсчёт объектов** — модели плохо считают: "Сколько людей на фото?" часто даёт неточный ответ, особенно если людей больше ~10.
- **Пространственные отношения** — "Что находится слева от красной машины?" — модели могут путать лево/право, выше/ниже.
- **Мелкие детали** — мелкий текст, номера телефонов, QR-коды могут не распознаваться, особенно при `detail: "low"`.
- **Галлюцинации** — модель может "увидеть" объекты, которых нет, или неправильно прочитать текст. Это особенно опасно для задач, где точность критична (медицина, юридические документы).

Понимание этих ограничений определяет дизайн инструментов: для задач, требующих точного подсчёта или чтения мелкого текста, может быть лучше использовать специализированные CV-модели (YOLO для подсчёта, Tesseract для OCR) вместо мультимодальной LLM.

### Vision-модели vs специализированные CV-решения

Выбор между мультимодальной LLM и специализированной моделью компьютерного зрения — это trade-off между универсальностью и точностью:

| Задача | Vision LLM | Специализированная CV | Рекомендация |
|--------|-----------|---------------------|-------------|
| Описание изображения | Отлично | Не применимо | Vision LLM |
| OCR (печатный текст) | Хорошо | Отлично (Tesseract, PaddleOCR) | Зависит от точности |
| Подсчёт объектов | Плохо | Отлично (YOLO, Detectron) | CV |
| Классификация | Хорошо | Отлично | Зависит от объёма |
| Анализ графиков | Хорошо | Средне | Vision LLM |
| Вопрос-ответ по фото | Отлично | Не применимо | Vision LLM |

Оптимальный подход для агента — комбинировать: Vision LLM для задач, требующих понимания контекста и рассуждений, специализированные модели — для задач, требующих точности.

### Передача изображений через API

OpenAI API поддерживает два способа передачи изображений: по URL и в формате base64. На практике base64 надёжнее — URL может быть недоступен или требовать авторизации:

```python
from openai import OpenAI
import base64

client = OpenAI()

def encode_image(image_path: str) -> str:
    with open(image_path, "rb") as f:
        return base64.b64encode(f.read()).decode("utf-8")

response = client.chat.completions.create(
    model="gpt-5-mini",
    messages=[{
        "role": "user",
        "content": [
            {"type": "text", "text": "Что изображено на этой картинке?"},
            {"type": "image_url", "image_url": {
                "url": f"data:image/jpeg;base64,{encode_image('photo.jpg')}",
                "detail": "high"
            }}
        ]
    }]
)
```

Параметр `detail` контролирует качество обработки и напрямую влияет на стоимость и скорость:

- `"low"` — изображение сжимается до 512x512, обрабатывается быстро и дёшево. Подходит для общего понимания сцены: "Это фото кота или собаки?"
- `"high"` — изображение обрабатывается в полном разрешении (до 2048x2048) с разбиением на тайлы. Необходим для чтения текста, анализа мелких деталей, работы с графиками.
- `"auto"` — модель сама выбирает на основе размера и содержимого.

Правило выбора: начинайте с `"low"`, переключайтесь на `"high"` только если задача требует детальности. Разница в стоимости может быть 10-кратной.

### Проектирование Vision-инструментов для агента

При создании инструментов для работы с изображениями ключевая задача — правильно сформулировать промпт для Vision-модели. Общий инструмент "опиши картинку" полезен, но специализированные инструменты работают точнее.

Например, для OCR промпт должен чётко указывать: "Извлеки ВЕСЬ текст, сохраняя структуру, указывая языки". Для анализа графика: "Определи тип графика, прочитай оси и легенду, опиши тренды, приведи конкретные числа". Для сравнения изображений: "Найди общее и различия, опиши возможные причины различий".

Каждый специализированный инструмент — это по сути обёртка над базовым вызовом Vision API с оптимизированным промптом. Полные реализации (analyze_image, extract_text_from_image, analyze_chart, compare_images) доступны в файле примеров к этой лекции.

## Часть 2: Генерация изображений — когда агент становится художником

### Эволюция моделей генерации

Генеративные модели изображений прошли стремительную эволюцию за последние несколько лет:

**DALL-E (2021)** — первая массовая модель, показавшая возможность генерации из текста. Качество было ограниченным, но концепция была революционной.

**DALL-E 3 (2023)** — кардинальное улучшение качества и следования промпту. Ключевая инновация — модель сама "улучшала" промпт перед генерацией, добавляя детали для лучшего результата. Это и плюс (качественнее результат), и минус (сложнее контролировать).

**GPT Image / gpt-image-1.5 (2025)** — актуальная модель OpenAI. Объединяет генерацию и редактирование изображений. Важные отличия от DALL-E 3: возвращает base64 вместо URL, отсутствует параметр `style`, нет `revised_prompt` — промпт обрабатывается внутри модели.

**Параллельно развиваются:** Midjourney (фокус на художественном качестве), Stable Diffusion (open source, локальный запуск), Imagen (Google, интеграция в Gemini).

### Как модели "понимают" промпты

Генеративная модель не "рисует" в привычном смысле. Она обучена на миллионах пар "изображение-описание" и научилась статистическим связям между текстовыми концепциями и визуальными паттернами. Слово "закат" связано с определёнными цветовыми распределениями, "киберпанк" — с неоновыми цветами и футуристической архитектурой.

Из этого следуют важные принципы **prompt engineering для изображений:**

1. **Конкретность** — "Золотистый ретривер бежит по осеннему парку" лучше, чем "красивая собака"
2. **Стилистические указания** — "в стиле масляной живописи импрессионизма", "фотореалистичный", "минималистичная иллюстрация"
3. **Композиция** — "крупный план", "вид сверху", "симметричная композиция"
4. **Освещение** — "мягкий дневной свет", "драматическая подсветка", "golden hour"
5. **Язык** — английские промпты обычно дают лучшие результаты, т.к. обучающие данные преимущественно на английском

### Интеграция GPT Image в агента

Базовый вызов API прост:

```python
response = client.images.generate(
    model="gpt-image-1.5",
    prompt="A futuristic city at sunset with neon lights reflecting on water",
    size="1024x1024",   # или "1536x1024" (ландшафт), "1024x1536" (портрет)
    quality="medium",   # "low", "medium", "high"
    n=1
)

# GPT Image возвращает base64, не URL!
image_b64 = response.data[0].b64_json
with open("result.png", "wb") as f:
    f.write(base64.b64decode(image_b64))
```

При обёртке в инструмент для агента важно учесть несколько нюансов:

- **Content policy** — модель может отклонить запрос, если он нарушает политику контента (насилие, откровенность, реальные люди). Инструмент должен ловить эту ошибку и возвращать понятное сообщение, а не stack trace.
- **Формат ответа** — base64-строка — не информативный ответ для агента. Инструмент должен сохранять файл и возвращать путь.
- **Стоимость** — генерация дороже анализа. Для `quality: "high"` — $0.133 за изображение. Агент должен генерировать только когда это действительно нужно.

### Итеративная генерация

Часто первый результат неидеален, и пользователь хочет корректировки: "Сделай фон темнее", "Добавь облака", "Поменяй стиль на акварель". Для этого создают инструмент "улучшения промпта", который на основе исходного промпта и обратной связи генерирует новый, более точный промпт для повторной генерации.

Этот паттерн — пример того, как LLM помогает LLM: одна модель (текстовая) оптимизирует промпт для другой модели (генеративной). Такие цепочки — характерная черта мультимодальных агентов.

## Часть 3: Работа с аудио — Speech-to-Text и Text-to-Speech

### Как работает распознавание речи

Современные модели STT (Speech-to-Text) прошли путь от rule-based систем через Hidden Markov Models и RNN к архитектуре трансформера. Whisper от OpenAI (2022) стал переломным моментом — модель, обученная на 680,000 часов аудио, показала качество, сопоставимое с коммерческими сервисами, и при этом была open source.

Архитектура Whisper (и её наследника gpt-4o-transcribe) — это encoder-decoder трансформер:

1. **Аудио** преобразуется в мел-спектрограмму — визуальное представление частот звука во времени
2. **Encoder** обрабатывает спектрограмму, извлекая акустические паттерны
3. **Decoder** генерирует текст токен за токеном, используя внимание к encoder и предыдущим токенам

Модель **gpt-4o-transcribe** — это эволюция Whisper с улучшенной точностью ($0.006/мин). Она лучше справляется с акцентами, шумными записями и переключением языков. Для задач, требующих временных меток (субтитры), по-прежнему используется whisper-1 с форматом `verbose_json`.

### Сравнение STT-моделей

| Модель | Точность | Стоимость | Временные метки | Перевод | Когда использовать |
|--------|---------|-----------|----------------|---------|-------------------|
| gpt-4o-transcribe | Высокая | $0.006/мин | Нет | Нет | Лучшая точность |
| whisper-1 | Хорошая | $0.006/мин | Да (verbose_json) | Да (→ англ.) | Субтитры, перевод |
| Deepgram | Высокая | От $0.0043/мин | Да | Да | Стриминг, real-time |
| Assembly AI | Высокая | $0.009/мин | Да | Нет | Дополнительная аналитика |

Выбор зависит от задачи: для простой транскрипции — gpt-4o-transcribe, для субтитров — whisper-1, для real-time стриминга — Deepgram.

### Базовые вызовы API

**Транскрипция аудио:**

```python
with open("recording.mp3", "rb") as audio_file:
    transcript = client.audio.transcriptions.create(
        model="gpt-4o-transcribe",
        file=audio_file,
        language="ru",
        response_format="text"
    )
print(transcript)
```

**Синтез речи (TTS):**

```python
response = client.audio.speech.create(
    model="tts-1",       # или "tts-1-hd" для высокого качества
    voice="nova",        # alloy, echo, fable, onyx, nova, shimmer
    input="Привет! Как дела?"
)
response.stream_to_file("output.mp3")
```

OpenAI предлагает 6 голосов, каждый с своим характером: `alloy` (нейтральный), `echo` (мужской), `fable` (с британским оттенком), `onyx` (глубокий мужской), `nova` (женский), `shimmer` (мягкий женский). Модель `tts-1` оптимизирована для скорости, `tts-1-hd` — для качества.

### Как работает синтез речи

Современный нейросетевой TTS кардинально отличается от "роботизированных" голосов прошлого. Модели учатся не конкатенации звуков, а *генерации* звуковой волны, подобно тому как GPT генерирует текст. Обучение на тысячах часов записей позволяет воспроизводить интонации, паузы, ударения — делая речь почти неотличимой от человеческой.

Это открывает интересные возможности для агентов: голосовой интерфейс, озвучивание документов, создание аудиоконтента. Но создаёт и этические вопросы — синтезированный голос может использоваться для deepfake и мошенничества.

### Use cases для STT и TTS в агентах

**STT в агентах:**

- **Голосовой ввод** — пользователь диктует запрос вместо набора текста
- **Анализ записей** — транскрипция митингов, интервью, лекций с последующим суммаризированием
- **Мониторинг** — автоматическая транскрипция звонков клиентской поддержки

**TTS в агентах:**

- **Голосовой вывод** — ответ агента озвучивается для accessibility или удобства
- **Аудиоконтент** — генерация подкастов, аудиокниг, обучающих материалов
- **Уведомления** — агент озвучивает важные алерты

## Часть 4: Комплексные мультимодальные сценарии

### Паттерны мультимодальных пайплайнов

Мультимодальные инструменты по-настоящему раскрываются, когда объединяются в цепочки — **модальные пайплайны**, где результат одной модальности становится входом для другой.

Типичные цепочки модальностей:

**Аудио → Текст → Текст → Аудио** (голосовой ассистент): пользователь говорит → STT транскрибирует → агент обрабатывает текстово → TTS озвучивает ответ. Это самый распространённый паттерн, реализуемый через три последовательных вызова инструментов.

**Изображение → Текст → Изображение** (визуальная трансформация): пользователь присылает фото → Vision-модель описывает его → на основе описания + инструкций генерируется новое изображение. Например: "Перерисуй этот пейзаж в стиле Ван Гога".

**Видео → Изображения → Текст** (анализ видео): видео разбивается на ключевые кадры → каждый кадр анализируется Vision-моделью → описания синтезируются в общий анализ видео.

**Текст → Аудио** (аудиокнига): длинный документ разбивается на фрагменты → каждый фрагмент озвучивается TTS → результат объединяется в аудиокнигу.

### Сценарий: Голосовой ассистент

Голосовой ассистент — это агент с тремя дополнительными "модулями": STT на входе, TTS на выходе, и обычный текстовый агент в середине. Архитектурно это выглядит так:

```
Микрофон → [STT] → текст запроса → [Агент + инструменты] → текст ответа → [TTS] → Динамик
```

Ключевые решения при проектировании:

- **Стриминг** — пользователь не хочет ждать полной транскрипции. Оптимально использовать streaming STT (Deepgram, AssemblyAI) для получения промежуточных результатов.
- **Latency** — суммарная задержка STT + LLM + TTS может быть 3-5 секунд. Для real-time диалога это на грани допустимого. Оптимизация: streaming TTS (начинать озвучивание до завершения генерации), кэширование частых ответов.
- **Voice Activity Detection (VAD)** — нужно определять, когда пользователь начал и закончил говорить. Без VAD агент будет "перебивать" или долго ждать.

### Сценарий: Анализ видео

Видеомодели существуют (Gemini 2.5 Pro нативно принимает видео), но для большинства задач достаточно покадрового анализа: извлечь N ключевых кадров, проанализировать каждый через Vision-модель, синтезировать общую картину.

Подход: видео открывается через OpenCV, кадры извлекаются через равные интервалы (например, 5 кадров из 30-секундного видео = 1 кадр каждые 6 секунд). Каждый кадр анализируется Vision-моделью с вопросом "Что происходит на этом кадре?". Затем все описания подаются на финальный синтез: "На основе анализа N кадров, опиши что происходит в видео".

Ограничения этого подхода: теряется информация о движении и аудио, между кадрами могут быть значимые события. Для точного анализа может потребоваться больше кадров (выше стоимость) или нативная видеомодель.

### Сценарий: Документ → Аудиокнига

Превращение текстового документа в аудио — задача, где TTS раскрывается полностью. Основная сложность — TTS API имеют лимит на длину текста (~4000 символов для OpenAI). Поэтому документ разбивается на фрагменты, каждый озвучивается отдельно, результаты собираются в серию аудиофайлов.

Важные нюансы:

- **Разбиение** — нельзя резать текст на произвольных позициях, нужно разбивать по абзацам или предложениям, иначе интонация будет неестественной.
- **Единство голоса** — все фрагменты должны озвучиваться одним голосом для консистентности.
- **Метаданные** — полезно сохранять маппинг "фрагмент текста → аудиофайл" для навигации.

Полные реализации всех трёх сценариев (голосовой ассистент, анализ видео, аудиокнига) доступны в файле примеров к этой лекции.

## Часть 5: Мультимодальный агент — архитектурные решения

### Как устроен мультимодальный агент

Мультимодальный агент — это по сути обычный текстовый агент (LLM + инструменты), где часть инструментов работает с нетекстовыми данными. Архитектурно всё просто: модель-"мозг" остаётся текстовой (или мультимодальной, если нужно передавать изображения напрямую), а инструменты обеспечивают конвертацию между модальностями.

```python
from langgraph.prebuilt import create_react_agent
from langchain_openai import ChatOpenAI

llm = ChatOpenAI(model="gpt-5-mini")

tools = [
    analyze_image,          # Изображение → Текст
    extract_text_from_image, # Изображение → Текст (OCR)
    generate_image,          # Текст → Изображение
    transcribe_audio,        # Аудио → Текст
    generate_speech,         # Текст → Аудио
]

agent = create_react_agent(llm, tools)
```

### Когда передавать изображение напрямую vs через описание

Критический архитектурный вопрос: должен ли агент получать изображение как часть контекста (мультимодальное сообщение) или вызывать инструмент анализа?

**Передача напрямую в контекст:**

- Плюсы: модель "видит" изображение и может рассуждать о нём в реальном времени
- Минусы: занимает много токенов контекстного окна (до 4000+ токенов на изображение)
- Когда: одно-два изображения, нужен интерактивный диалог о содержимом

**Через инструмент:**

- Плюсы: экономит контекст, результат анализа — компактный текст
- Минусы: теряются детали (описание всегда менее информативно, чем само изображение)
- Когда: много изображений, изображение не является центральной темой

На практике часто используют комбинацию: первое изображение передаётся напрямую для детального обсуждения, остальные — через инструмент для экономии контекста.

### Управление контекстным окном

Мультимодальные данные быстро заполняют контекстное окно. Одно изображение при `detail: "high"` может стоить столько же, сколько страница текста. Стратегии управления:

- **Lazy loading** — не загружать изображение, пока оно не понадобится. Инструмент `analyze_image` вызывается только когда агент решит, что нужен визуальный анализ.
- **Предварительная оптимизация** — уменьшать изображения перед отправкой в API. Для общего понимания сцены 512x512 вполне достаточно.
- **Текстовые прокси** — после анализа изображения работать с текстовым описанием, а не с самим изображением.
- **Селективная детализация** — использовать `detail: "low"` по умолчанию, переключаться на `"high"` только для задач, требующих точности (OCR, мелкие детали).

## Часть 6: Практические соображения

### Стоимость мультимодальных операций

Мультимодальные модели дороже текстовых, и это нужно учитывать при проектировании:

| Операция | Примерная стоимость |
|----------|-------------------|
| GPT-5-mini (текст + vision) | $0.25 / 1M input токенов |
| GPT-5 (текст + vision) | $1.25 / 1M input токенов |
| GPT Image gpt-image-1.5 (medium) | $0.034 / изображение |
| GPT Image gpt-image-1.5 (high) | $0.133 / изображение |
| gpt-4o-transcribe | $0.006 / минута |
| TTS tts-1 | $0.015 / 1000 символов |
| TTS tts-1-hd | $0.030 / 1000 символов |

Для контекста: анализ одного изображения (1024x1024, high detail) стоит примерно столько же, сколько обработка ~3000 слов текста. Генерация одного изображения в high quality ($0.133) — как 100,000 входных токенов GPT-5-mini.

### Стратегии оптимизации расходов

1. **Начинайте с low detail** — для большинства задач (определение содержимого, общее описание) `detail: "low"` достаточно. Переключайтесь на `"high"` только для OCR и анализа мелких деталей.

2. **Оптимизируйте изображения** — уменьшайте разрешение и конвертируйте в JPEG перед отправкой. Изображение 4000x4000 и 1024x1024 дают сопоставимый результат при `detail: "auto"`, но стоимость отличается в разы.

3. **Кэшируйте анализ** — если одно и то же изображение анализируется с разными вопросами, сначала получите подробное описание, затем задавайте вопросы к описанию (текстовому), а не к изображению.

4. **Batching TTS** — TTS тарифицируется по символам. Если нужно озвучить несколько коротких фраз — объедините их в один запрос с разделителями, затем разрежьте аудио.

5. **Используйте специализированные модели** — для задач, где не нужен "интеллект" LLM (простое OCR, распознавание лиц, классификация изображений), специализированные модели дешевле и точнее.

### Когда мультимодальность оправдана

Мультимодальные инструменты добавляют сложность, стоимость и latency. Используйте их, когда:

- Задача **по природе мультимодальна** — пользователь присылает фото и задаёт вопрос о нём
- **Текстовой альтернативы нет** — нельзя попросить пользователя описать содержимое изображения текстом
- **Ценность превышает стоимость** — анализ медицинских снимков, контроль качества на производстве, создание контента

Не используйте, когда:

- Задача решается текстом — "нарисуй таблицу" лучше решить markdown-таблицей, а не генерацией изображения
- Пользователь может описать содержимое — иногда быстрее спросить "что на фото?", чем передавать изображение в API
- Стоимость не оправдана — массовая обработка изображений через Vision LLM может быть неоправданно дорогой

## Ключевые термины

| Термин | Определение |
|--------|-------------|
| **Vision Model** | Модель, способная анализировать изображения наряду с текстом (GPT-5, Claude, Gemini) |
| **Патчи (patches)** | Небольшие квадратные фрагменты изображения, преобразуемые в эмбеддинги для трансформера |
| **GPT Image** | Модель OpenAI для генерации изображений по текстовому описанию (gpt-image-1.5) |
| **gpt-4o-transcribe** | Модель OpenAI для распознавания и транскрипции речи (эволюция Whisper) |
| **TTS (Text-to-Speech)** | Технология синтеза речи из текста |
| **STT (Speech-to-Text)** | Технология распознавания речи и преобразования в текст |
| **OCR** | Optical Character Recognition — распознавание текста на изображениях |
| **Base64** | Способ кодирования бинарных данных (изображений, аудио) в текстовую строку для передачи через API |
| **Мультимодальный пайплайн** | Цепочка обработки, где данные проходят через инструменты разных модальностей |
| **Voice Activity Detection** | Технология определения моментов начала и окончания речи пользователя |
