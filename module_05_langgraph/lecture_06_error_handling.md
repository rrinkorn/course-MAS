# Лекция 6: Обработка ошибок — как сделать агента устойчивым

## Введение: Всё ломается

В продакшне всё ломается. API возвращает 500. Модель упирается в rate limit. Инструмент тайминг-аутит. Сеть мигает. Это не вопрос «если», а вопрос «когда». Граф без обработки ошибок рухнет на первом сбое — и пользователь получит трейсбек вместо ответа.

Представьте электростанцию. У неё есть основной генератор, но к нему прилагается целая система защит: автоматика перезапуска при кратковременном сбое, резервный генератор на случай полной поломки, аварийное отключение при перегрузке, предохранители на каждой линии. Ни одна электростанция не работает на голой надежде, что основной генератор никогда не сломается. Агент в продакшне — то же самое.

В этой лекции мы пройдём путь от самых простых защит к самым мощным. Начнём с того, что можно сделать внутри одного узла, и дойдём до встроенных механизмов LangGraph, которые защищают граф целиком. А в следующей лекции разберём middleware — декларативную альтернативу, которая делает то же самое одной строкой.

---

## Часть 1: Деградация в узле — лучше частичный ответ, чем никакого

Самый простой приём — ловить ошибку внутри узла и возвращать «деградированный» результат вместо того, чтобы бросать исключение наверх. Узел вызывает внешний API поиска, оборачивает вызов в `try/except`, и при ошибке возвращает сообщение вроде «Поиск недоступен, отвечаю из своих знаний». Граф продолжает работу. Последующие узлы получают хоть и неидеальный, но валидный результат.

Принцип звучит так: узел никогда не должен бросать необработанное исключение, если это можно предотвратить. Необработанное исключение останавливает весь граф. Если при этом нет persistence — теряется вся выполненная работа. Если persistence есть — граф можно возобновить, но пользователь всё равно ждёт ответа, которого нет. Деградированный результат решает обе проблемы: граф продолжает, пользователь получает хотя бы что-то.

Но есть нюанс. Деградация подходит не для всех ошибок. Если API недоступен — логично вернуть «отвечаю без поиска». Если ошибка в коде самого узла (TypeError, KeyError) — это баг, и глушить его деградацией опасно. Баг нужно увидеть и починить, а не прятать за «частичным ответом». Хорошее правило: деградация — для внешних сбоев, исключения — для внутренних багов.

> Полный пример: examples_06_error_handling.py, функция example_graceful_degradation()

## Часть 2: Retry с backoff — иногда достаточно попробовать ещё раз

Многие сбои транзиентны. Сервер перегружен — через секунду освободится. Rate limit — через пять секунд сбросится. Сеть мигнула — через мгновение восстановится. Для таких случаев есть retry — просто попробовать ещё раз.

Наивный retry — это цикл с фиксированной задержкой. Не сработало — подождал секунду — попробовал — не сработало — подождал секунду — попробовал. Проблема в том, что если сервер перегружен, десять клиентов, одновременно получивших отказ, через ту же секунду ломанутся снова. И снова перегрузят сервер. Это называется thundering herd — «стадо буйволов».

Экспоненциальный backoff решает проблему. Задержка между попытками растёт: секунда, две, четыре, восемь. К каждой задержке добавляется случайный разброс — jitter. В результате десять клиентов, одновременно получивших отказ, повторяют попытки в разное время, и сервер успевает восстановиться.

Retry внутри узла — это пять-шесть строк: цикл, try/except, задержка, и при исчерпании попыток — деградация из предыдущей части. Две стратегии комбинируются: retry пытается получить нормальный результат, а если все попытки провалились — деградация не даёт графу упасть.

> Полный пример: examples_06_error_handling.py, функция example_retry_with_backoff()

## Часть 3: RetryPolicy — встроенный retry LangGraph

Писать retry вручную в каждом узле утомительно. Если в графе десять узлов с внешними вызовами — это пятьдесят строк одинакового кода. LangGraph предлагает встроенное решение: `RetryPolicy` из `langgraph.types`.

RetryPolicy подключается при добавлении узла в граф. Вы передаёте `retry_policy=RetryPolicy(...)` в `add_node`, и LangGraph сам оборачивает этот узел в retry-логику. Если узел выбросил исключение — LangGraph повторяет его целиком с экспоненциальным backoff и jitter.

```python
from langgraph.types import RetryPolicy

builder.add_node(
    "search", search_node,
    retry_policy=RetryPolicy(max_attempts=3, initial_interval=1.0)
)
```

У RetryPolicy есть важная деталь: `retry_on`. По умолчанию LangGraph повторяет не при любой ошибке. Он исключает «программистские» ошибки — ValueError, TypeError, KeyError, SyntaxError. Логика простая: если аргументы функции невалидны, повторный вызов с теми же аргументами не поможет. Retry имеет смысл только для транзиентных ошибок — сетевых таймаутов, 5xx, rate limit. Можно переопределить фильтр, передав `retry_on` свой список типов или функцию-предикат.

RetryPolicy работает не только в Graph API. В Functional API тот же `RetryPolicy` передаётся в декоратор `@task(retry_policy=...)`, и LangGraph повторяет задачу при сбое. Один механизм — два способа подключения.

> Полный пример: examples_06_error_handling.py, функция example_retry_policy()

## Часть 4: Attempt-aware узлы — знать, какая это попытка

Начиная с LangGraph 1.1.3, узел может узнать, какая это попытка выполнения. Объект `runtime`, доступный узлу вторым аргументом, содержит `execution_info` с полем `node_attempt` — номер текущей попытки, начиная с единицы.

Зачем это нужно? Представьте узел, который вызывает основную модель. При первой попытке — полноценный запрос к мощной модели. Если она недоступна и RetryPolicy запускает вторую попытку — узел видит `node_attempt == 2` и переключается на запасную, более дешёвую модель. Не нужно писать отдельные узлы и fallback-рёбра — вся логика живёт в одном узле, а RetryPolicy запускает повторные попытки автоматически.

Помимо номера попытки, `execution_info` хранит `node_first_attempt_time` — время первой попытки. Это позволяет реализовать бюджет времени: если с первой попытки прошло больше тридцати секунд — не пробовать снова, а сразу вернуть деградированный результат.

> Полный пример: examples_06_error_handling.py, функция example_attempt_aware_node()

## Часть 5: Fallback-ребро — запасной путь на уровне графа

До сих пор мы защищали отдельные узлы. Но иногда нужна защита на уровне графа — если основной путь обработки не сработал, направить поток в запасной узел.

Идея простая. Основной узел выполняет работу и при ошибке записывает информацию об ошибке в поле `error` состояния. После него стоит условное ребро, функция-маршрутизатор которого проверяет это поле. Если ошибки нет — обычный путь дальше. Если есть — переход в fallback-узел, который делает более простую, но надёжную обработку. Например, основной узел вызывает сложную RAG-цепочку с несколькими источниками, а fallback просто отвечает из знаний модели без поиска.

Этот паттерн масштабируется. Можно построить каскад: основной путь → первый fallback → второй fallback → финализация. Каждый уровень проще предыдущего, но надёжнее. Пользователь получает наилучший доступный результат, а не ошибку.

Fallback-рёбра особенно хорошо сочетаются с persistence. Если основной узел упал, но его ошибка сохранена в checkpoint, граф можно возобновить с условного ребра. Не нужно перевыполнять узлы, которые уже отработали успешно.

> Полный пример: examples_06_error_handling.py, функция example_fallback_edge()

## Часть 6: ToolNode и обработка ошибок инструментов

В лекции 3 мы разбирали `ToolNode` — готовый узел для выполнения инструментов в агентском цикле. У него есть параметр `handle_tool_errors`, который определяет, что делать при ошибке инструмента.

Важное изменение: начиная с `langgraph-prebuilt 1.0.1`, поведение по умолчанию изменилось. Раньше `handle_tool_errors` был равен `True` — любая ошибка инструмента перехватывалась и возвращалась модели как `ToolMessage`. Теперь по умолчанию перехватываются только ошибки валидации аргументов — `ToolInvocationError`. Все остальные ошибки пробрасываются наверх.

Логика изменения: ошибка валидации — это ожидаемая ситуация, модель подобрала неправильные аргументы, и ей полезно увидеть, что пошло не так, чтобы попробовать снова. А вот ошибка подключения к базе данных — это не проблема аргументов, и модель ничего не сможет с ней сделать. Такую ошибку лучше пробросить наверх, где RetryPolicy или fallback-ребро обработают её на уровне графа.

Если вы хотите старое поведение — когда любая ошибка возвращается модели — укажите `handle_tool_errors=True` явно. Это полезно для агентов, которые должны уметь корректировать подход при любой ошибке инструмента: попробовать другие аргументы, выбрать другой инструмент или ответить без инструмента.

Можно передать и функцию — она получит исключение и вернёт текст ошибки, который увидит модель. Это позволяет форматировать ошибку в человекочитаемое сообщение, скрывая технические детали вроде трейсбеков и внутренних URL.

> Полный пример: examples_06_error_handling.py, функция example_tool_error_handling()

## Часть 7: Глобальные лимиты — recursion_limit и step_timeout

Иногда отдельные узлы работают нормально, но граф в целом ведёт себя неадекватно. Модель раз за разом вызывает инструменты, немного меняя аргументы, но не получает нужного результата. Или условное ребро создаёт бесконечный цикл, потому что условие выхода никогда не выполняется. Без ограничений граф будет крутиться, пока не закончатся деньги или терпение пользователя.

LangGraph имеет два встроенных механизма защиты. Первый — `recursion_limit`. Он ограничивает количество superstep'ов — шагов выполнения графа. По умолчанию лимит равен 25. Если граф за 25 шагов не дошёл до END, LangGraph бросает `GraphRecursionError`. Лимит передаётся через конфигурацию при вызове графа.

Второй — `step_timeout`. Он ограничивает время выполнения каждого superstep'а. Если один узел или группа параллельных узлов выполняется дольше заданного времени, LangGraph прерывает их. Таймаут устанавливается на скомпилированном графе как атрибут.

Оба механизма работают как страховочная сетка. Они не заменяют нормальную обработку ошибок в узлах, но защищают от катастрофических сценариев. Лучше получить `GraphRecursionError` на 25-м шаге, чем обнаружить утром, что агент потратил тысячу долларов на API-вызовы, крутясь в бесконечном цикле.

Благодаря persistence даже при принудительном прерывании промежуточные результаты сохранены в checkpoint'ах. Можно посмотреть, что агент успел сделать, понять, почему зациклился, и при необходимости вытащить полезные результаты из предыдущих шагов.

> Полный пример: examples_06_error_handling.py, функция example_global_limits()

## Часть 8: Комбинирование стратегий

Стратегии обработки ошибок не конкурируют друг с другом — они дополняют. Типичный продакшн-граф использует несколько уровней защиты одновременно.

На самом нижнем уровне — деградация внутри узлов. Каждый узел, вызывающий внешний сервис, оборачивает вызов в try/except и при неудаче возвращает частичный результат. Это первая линия обороны.

Уровнем выше — RetryPolicy на узлах с внешними вызовами. Транзиентный сбой не доходит до деградации: RetryPolicy повторяет узел автоматически, и в большинстве случаев вторая или третья попытка успешна. Деградация срабатывает только если все попытки исчерпаны.

Ещё выше — fallback-рёбра для критических узлов. Если основной путь обработки полностью провалился — даже после retry и деградации результат неприемлем — условное ребро направляет поток в запасной узел с более простой, но гарантированно работающей логикой.

На самом верхнем уровне — recursion_limit как страховочная сетка от зацикливания.

Эти четыре уровня — ручные стратегии. Для каждого узла вы решаете, какие уровни нужны, и пишете соответствующий код. Это даёт полный контроль, но и полную ответственность. В следующей лекции мы увидим альтернативу: middleware, которые подключают retry, fallback и лимиты одной строкой при создании агента, без изменения кода узлов.

## Итоги

Обработка ошибок — не опциональное украшение, а обязательная часть любого продакшн-графа. Деградация в узлах возвращает частичный результат вместо краша. Retry с backoff справляется с транзиентными сбоями. RetryPolicy из `langgraph.types` автоматизирует retry на уровне узла или задачи. Attempt-aware узлы через `runtime.execution_info` адаптируют поведение в зависимости от номера попытки. Fallback-рёбра перенаправляют поток в запасные узлы. ToolNode с `handle_tool_errors` возвращает ошибку модели вместо краша. Recursion_limit и step_timeout защищают от зацикливания.

Всё это — ручные стратегии. В лекции 7 мы разберём middleware — декларативный подход, который делает то же самое, но без ручного кода в каждом узле.
