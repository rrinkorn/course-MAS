# Лекция 8: Send — динамический параллелизм

## Введение: когда рёбер недостаточно

В лекции 2 мы видели, как внутри подграфа из START идут два ребра к двум узлам — web_search и doc_search. LangGraph запускает оба параллельно и ждёт результатов, прежде чем передать управление дальше. Напомним терминологию: **fan-out** — когда один поток разветвляется на несколько параллельных, **fan-in** — когда несколько параллельных потоков сходятся обратно в один. Два ребра — два параллельных потока. Три ребра — три потока. Число веток задано раз и навсегда при построении графа.

Но представьте другую ситуацию. Пользователь загрузил пять документов — и каждый нужно обработать параллельно. Планировщик разбил отчёт на семь секций — каждую нужно написать отдельно. Поисковик нашёл десять источников — каждый нужно проанализировать. Количество задач заранее неизвестно — оно определяется данными в момент выполнения. Жёстко прописывать рёбра для каждого возможного варианта невозможно: граф уже скомпилирован, рёбра добавлять поздно.

Для этого в LangGraph есть `Send` — примитив динамического fan-out. Он позволяет прямо в runtime создать произвольное число параллельных вызовов одного и того же узла, каждый со своими входными данными. Три элемента в списке — три параллельных вызова. Десять — десять. Ноль — ни одного. Решение принимается в момент выполнения, а не при построении графа.

---

## Часть 1: Статический fan-out — напоминание

В лекции 2 при разборе подграфов мы уже строили статический fan-out. Из START шли два ребра к двум поисковым узлам, оба запускались одновременно, результаты объединялись через reducer `operator.add` и поступали в узел суммаризации. Это работает отлично, когда набор параллельных задач фиксирован: всегда ровно два поиска, всегда ровно три валидатора. Число веток не зависит от входных данных.

Ограничение проявляется, как только количество параллельных задач начинает зависеть от данных. Генератор тем выдал пять тем — нужно пять параллельных исследований. Планировщик разбил задачу на три подзадачи — нужно три воркера. Пользователь загрузил восемь файлов — нужно восемь обработчиков. Рёбра в скомпилированный граф добавить нельзя. Нужен другой механизм.

> Полный пример статического fan-out: examples_08_send.ipynb, функция `example_static_fanout()`

## Часть 2: Send — как это работает

`Send` — это класс из модуля `langgraph.types`. Он принимает два аргумента: имя целевого узла и данные, которые будут переданы этому узлу. Один объект `Send` — это инструкция: «запусти вот этот узел с вот этими данными». Сам по себе один `Send` не особенно интересен. Сила в том, что можно вернуть список из нескольких `Send`-объектов — и LangGraph запустит все указанные вызовы параллельно.

Классический способ подключить Send к графу — через conditional edge. Вспомним лекцию 1: обычно функция-маршрутизатор возвращает строку — имя следующего узла. С `Send` функция-маршрутизатор возвращает не строку, а список `Send`-объектов. LangGraph видит этот список и для каждого элемента запускает указанный узел с указанными данными — всё параллельно.

```python
def fan_out_to_jokes(state: OverallState):
    return [Send("generate_joke", {"subject": s}) for s in state["subjects"]]

builder.add_conditional_edges("generate_topics", fan_out_to_jokes, ["generate_joke"])
```

Разберём по шагам. Узел `generate_topics` отработал и записал в состояние список тем — допустим, `["львы", "слоны", "пингвины"]`. Затем LangGraph вызывает функцию `fan_out_to_jokes`, которая читает этот список и для каждого элемента создаёт `Send` — инструкцию вызвать узел `generate_joke` с конкретной темой. Три темы — три `Send`-объекта — три параллельных вызова. Третий аргумент `add_conditional_edges` — список `["generate_joke"]` — это перечень **всех возможных целевых узлов**, нужный для валидации графа и визуализации. В этом примере он содержит один элемент, потому что все Send идут в один и тот же узел. Но Send может направлять задачи в разные узлы — и тогда список станет длиннее. Об этом — в части 8.

Важный момент: каждый вызов через `Send` получает **не полное состояние графа**, а именно тот словарь, который передан вторым аргументом. Первый вызов `generate_joke` получит `{"subject": "львы"}`, второй — `{"subject": "слоны"}`, третий — `{"subject": "пингвины"}`. Каждый узел работает со своей порцией данных, ничего не зная о соседних вызовах.

> Полный пример: examples_08_send.ipynb, функция `example_send_basic()`

## Часть 3: Send через Command — fan-out прямо из узла

Conditional edge — не единственный способ использовать `Send`. Начиная с LangGraph 1.0, объект `Command` поддерживает `Send` в параметре `goto`. Это позволяет запускать параллельные ветки прямо из узла, без отдельной функции-маршрутизатора.

```python
def planner(state: State) -> Command:
    tasks = generate_tasks(state["topic"])
    return Command(
        update={"status": "dispatched"},
        goto=[Send("worker", {"task": t}) for t in tasks]
    )
```

Узел `planner` одновременно обновляет состояние (`status`) и запускает параллельных воркеров. Раньше для этого требовались две сущности: узел, который обновляет состояние, и conditional edge с функцией, возвращающей `Send`. Теперь всё в одном месте.

Когда какой подход использовать? Conditional edge остаётся естественным выбором, когда fan-out — это чистая маршрутизация без обновления состояния: один узел отработал, следующий шаг — распределить работу. `Command` с `Send` удобнее, когда узел должен и обновить состояние, и запустить параллельные ветки — всё в одном возвращаемом значении. Результат одинаковый, выбор — вопрос стиля.

> Полный пример: examples_08_send.ipynb, функция `example_send_via_command()`

## Часть 4: Reducer как механизм fan-in

Три параллельных вызова `generate_joke` завершились. Каждый вернул свой результат: один — шутку про львов, другой — про слонов, третий — про пингвинов. Три обновления для одного и того же поля `jokes`. Что произойдёт дальше?

Если поле `jokes` определено без reducer'а — последний завершившийся вызов затрёт результаты предыдущих. Останется одна шутка из трёх. Причём какая именно — непредсказуемо, потому что порядок завершения параллельных задач не гарантирован.

Если же поле определено как `Annotated[list[str], operator.add]` — все три списка конкатенируются. Получится полный набор: шутка про львов, шутка про слонов, шутка про пингвинов. Ни один результат не потерян.

В лекции 2 мы разбирали reducer'ы как механизм защиты от затирания: чтобы два узла, пишущих в одно поле, не уничтожали результаты друг друга. В контексте `Send` reducer приобретает новую, более важную роль — он становится механизмом **сбора результатов** параллельных веток. Без reducer'а `Send` бесполезен: вы создадите десять параллельных вызовов, но девять результатов из десяти потеряются. Поэтому поле, в которое пишут узлы, вызываемые через `Send`, **обязательно** должно иметь reducer.

Формула динамического параллелизма в LangGraph проста: fan-out — это `Send`, который создаёт N параллельных вызовов, а fan-in — это reducer на поле состояния, который собирает N результатов в один список. Одно без другого не работает.

> Полный пример: examples_08_send.ipynb, функция `example_send_basic()`

## Часть 5: Паттерн Map-Reduce

Map-Reduce — классический паттерн параллельной обработки, и именно для него `Send` подходит идеально. Фаза Map: разбить задачу на N независимых подзадач и выполнить каждую параллельно. Фаза Reduce: собрать результаты и объединить в итоговый ответ.

В LangGraph это реализуется четырьмя шагами. Первый узел (планировщик) анализирует задачу и определяет список подзадач. Conditional edge или Command с `Send` создаёт параллельные вызовы — по одному на каждую подзадачу. Reducer на поле состояния автоматически собирает все результаты в общий список. Финальный узел (синтезатор) обрабатывает собранные результаты и формирует итоговый ответ.

Структура графа выглядит так:

```text
START → planner → [Send × N] → worker (×N параллельно) → synthesizer → END
```

Обратите внимание на ребро от worker к synthesizer. Оно означает: «когда **все** вызовы worker завершатся — перейди к synthesizer». LangGraph автоматически дожидается завершения всех параллельных `Send`-вызовов перед переходом к следующему узлу. Никакой ручной логики ожидания писать не нужно.

По-настоящему паттерн раскрывается с LLM. Типичный сценарий — параллельное исследование. Планировщик получает тему «Влияние ИИ на рынок труда» и разбивает её на аспекты: экономика, технологии, социальные последствия. Через `Send` запускается исследователь для каждого аспекта — три экземпляра параллельно, каждый вызывает LLM со своим фокусом. Reducer собирает все находки. Синтезатор получает полный набор данных и формирует итоговый отчёт. Три вызова LLM выполняются одновременно вместо последовательно — заметный выигрыш в скорости.

> Полный пример: examples_08_send.ipynb, функция `example_map_reduce()`

## Часть 6: Orchestrator-Worker — обобщение Map-Reduce

Map-Reduce — частный случай более общего паттерна Orchestrator-Worker. Оркестратор анализирует задачу и распределяет подзадачи между воркерами. Каждый воркер выполняет свою часть. Результаты собираются и синтезируются. Отличие от чистого Map-Reduce в том, что оркестратор может передавать воркерам не только их кусочек данных, но и общий контекст — например, список всех секций отчёта, чтобы каждый воркер писал свою секцию с учётом соседних.

Представьте задачу: написать отчёт на заданную тему. Оркестратор планирует секции — введение, основная часть, выводы. Через `Send` для каждой секции создаётся воркер, причём каждый получает не только свою секцию, но и план всего отчёта целиком — для связности текста. Reducer собирает написанные секции, синтезатор склеивает их в единый документ.

Этот паттерн — основа многих архитектур, которые мы будем строить в модуле 8. Голосование — это fan-out к N голосующим агентам, каждый возвращает свой голос, reducer собирает голоса, финальный узел считает большинство. Ансамбль — fan-out к N моделям или агентам, каждый генерирует свой вариант, финальный узел выбирает лучший. Динамическое порождение — fan-out к заранее неизвестному числу агентов. Один и тот же примитив — `Send` плюс reducer — лежит в основе всех этих паттернов.

> Полный пример: examples_08_send.ipynb, функция `example_orchestrator_worker()`

## Часть 8: Гетерогенный fan-out — Send в разные узлы

Во всех предыдущих примерах `Send` направлял все задачи в один и тот же узел — `generate_joke`, `researcher`, `worker`. Это **гомогенный** fan-out: N копий одного узла с разными данными. Но `Send` не ограничен одним узлом. Каждый `Send`-объект указывает имя целевого узла независимо — и разные объекты в одном списке могут указывать на разные узлы.

Представьте менеджера проекта, который анализирует задачу и определяет подзадачи разных типов: исследование рынка, написание кода, создание дизайна. Каждый тип требует своего специалиста. Функция-маршрутизатор создаёт `Send` в разные узлы в зависимости от типа задачи:

```python
def dispatch_tasks(state: ProjectState):
    sends = []
    for task in state["tasks"]:
        if task["type"] == "research":
            sends.append(Send("researcher", {"description": task["description"]}))
        elif task["type"] == "code":
            sends.append(Send("coder", {"description": task["description"]}))
    return sends

builder.add_conditional_edges("manager", dispatch_tasks, ["researcher", "coder"])
```

Вот здесь третий аргумент `["researcher", "coder"]` становится по-настоящему важен — он объявляет все возможные целевые узлы. LangGraph использует этот список для валидации графа при компиляции и для визуализации — на диаграмме от `manager` пойдут рёбра к обоим узлам.

Важная деталь fan-in: если `researcher` и `coder` оба пишут в поле `deliverables` с reducer'ом `operator.add`, то LangGraph дождётся завершения **всех** параллельных вызовов — и исследовательских, и кодовых — прежде чем перейти к следующему узлу. Гомогенный или гетерогенный — логика ожидания одинакова.

Этот паттерн — прямой мост к мультиагентным системам из модуля 8. Когда оркестратор распределяет задачи между агентами с разной специализацией — это именно гетерогенный fan-out. Только вместо простых узлов-функций целевыми узлами становятся подграфы, каждый из которых представляет полноценного агента.

> Полный пример: examples_08_send.ipynb, функция `example_heterogeneous_fanout()`

## Часть 9: Когда использовать Send

Когда выбирать `Send`, а когда достаточно обычных средств?

Параллельные рёбра подходят, когда набор параллельных задач фиксирован и известен заранее. Поиск в трёх конкретных источниках, проверка двумя валидаторами — число веток не зависит от входных данных, рёбра можно прописать при построении графа.

`Send` нужен, когда число параллельных задач определяется в runtime. Обработка N документов, исследование N аспектов, голосование N агентов — количество становится известно только при выполнении. Это территория `Send`.

Последовательный цикл обрабатывает элементы один за другим. Если каждый элемент независим от предыдущих — `Send` даёт выигрыш в скорости, пропорциональный числу элементов. Но если результат обработки одного элемента нужен для обработки следующего — цикл, а не `Send`.

Стоит помнить и о Functional API из лекции 9. Там параллельный запуск задач достигается через list comprehension и `.result()` — без Send. Если ваш пайплайн и так написан через Functional API, Send не нужен. Но для StateGraph-агентов Send остаётся единственным способом динамического fan-out.

## Итоги

**Send** из `langgraph.types` — примитив динамического fan-out. Два способа использования: через conditional edge (функция-маршрутизатор возвращает список `Send`-объектов) или через `Command(goto=[Send(...)])` прямо из узла.

**Reducer** (`Annotated[list, operator.add]`) — механизм fan-in, который собирает результаты параллельных веток в один список. Без reducer'а результаты `Send` теряются — это обязательная пара.

**Map-Reduce** — основной паттерн: планировщик разбивает задачу на подзадачи, `Send` запускает воркеров параллельно, reducer собирает результаты, синтезатор объединяет. **Гетерогенный fan-out** — каждый `Send` может направлять задачу в свой узел, что позволяет маршрутизировать задачи разных типов к разным специалистам в одном шаге. В модуле 8 эти примитивы используются в паттернах голосования, ансамбля, map-reduce и динамического порождения агентов.
