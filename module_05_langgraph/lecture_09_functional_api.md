# Лекция 9: Functional API — агенты без графа

## Введение: Когда граф — лишняя церемония

Мы прошли восемь лекций, строя графы через StateGraph. Узлы, рёбра, условные рёбра, компиляция. Для сложных агентов с ветвлениями, параллельными путями и многоуровневыми подграфами — это именно то, что нужно. Но давайте честно: для линейного пайплайна из трёх шагов с одним условием это избыточно. Шесть строк на инфраструктуру графа, три строки на логику. Хвост виляет собакой.

Представьте себе два способа добраться из точки А в точку Б. Первый — нарисовать маршрут на карте, разметить каждый поворот, каждый перекрёсток с указанием, куда ехать при каком условии. Это StateGraph. Второй — просто сесть в машину и ехать: прямо, направо, если пробка — объехать. Это Functional API. Результат один — вы добираетесь до Б. Но для простого маршрута второй способ очевидно удобнее.

Functional API предлагает писать агентов как обычные Python-функции. Два декоратора — `@entrypoint` и `@task` — и вся оркестрация выглядит как обычный Python-код. Условия — обычный `if`. Циклы — обычный `for`. Обработка ошибок — обычный `try/except`. При этом checkpointing, persistence и recovery работают автоматически — те же гарантии, что и у StateGraph, но без явного графа.

## Часть 1: @entrypoint и @task — основы

Два декоратора — вот и весь API. Декоратор `@task` превращает обычную функцию в задачу с автоматическим checkpointing. Каждый вызов такой функции сохраняет результат. Если процесс прервётся и будет перезапущен, уже выполненные задачи не повторяются — результат берётся из checkpoint'а.

Декоратор `@entrypoint` определяет точку входа — это аналог `graph.compile()`. Он принимает checkpointer и обеспечивает persistence, recovery и управление потоками. Результат — полноценный Runnable, который можно вызывать через `invoke`, `stream`, `ainvoke`, получать историю состояний, использовать `thread_id` для привязки к сессии.

```python
@entrypoint(checkpointer=InMemorySaver())
def article_pipeline(topic: str) -> str:
    facts = research(topic).result()
    draft = write_article(topic, facts).result()
    final = edit_article(draft).result()
    return final
```

Здесь `research`, `write_article` и `edit_article` — функции с декоратором `@task`. Вызов задачи возвращает не результат, а future — объект-обещание. Метод `.result()` блокирует выполнение до получения результата. Если процесс прервётся после `research`, при перезапуске исследование не будет повторяться — результат возьмётся из checkpoint'а, и выполнение продолжится с `write_article`.

Это ключевая механика: при возобновлении entrypoint перевыполняется целиком с начала, но все ранее завершённые задачи моментально возвращают кэшированный результат. Выполнение по сути «перематывается» до точки, где остановилось.

> Полный пример: examples_09_functional_api.py, функция example_basic_pipeline()

## Часть 2: Условная логика — просто Python

Вот где Functional API по-настоящему раскрывается. Вспомните, как в StateGraph реализуется цикл рефлексии: отдельный узел `evaluate`, условное ребро `should_continue` с функцией-маршрутизатором, маппинг строк на узлы, обратное ребро к `generate`. Четыре сущности для одной простой мысли: «генерируй, оценивай, если плохо — улучшай и повторяй».

В Functional API та же логика записывается как обычный цикл:

```python
for i in range(3):
    critique = evaluate(draft).result()
    if "APPROVED" in critique:
        break
    draft = improve(draft, critique).result()
```

Три строки. Обычный `for`, обычный `if`, обычный `break`. Не нужно переводить мысль в абстракции графа — мысль записывается напрямую. Если результат поиска короткий — пишете `if len(facts) < 100` и дописываете факты. Если нужно обработать ошибку — пишете `try/except`. Если нужен ранний выход — пишете `return`. Весь арсенал Python к вашим услугам.

И при этом каждый вызов `@task`-функции — это checkpoint. Если процесс упадёт на третьей итерации цикла, при перезапуске первые две итерации пролетят мгновенно, потому что результаты `evaluate` и `improve` уже сохранены.

> Полный пример: examples_09_functional_api.py, функция example_conditions_and_loops()

## Часть 3: Побочные эффекты и детерминизм — важный gotcha

Есть правило, нарушение которого приводит к трудноуловимым багам: **все побочные эффекты должны быть внутри `@task`**. Вызовы API, запись в файл, отправка email, обращение к базе данных — всё это должно быть обёрнуто в `@task`. Код вне задач — обычные вычисления, условия, циклы — должен быть детерминированным.

Почему? При возобновлении после прерывания entrypoint перевыполняется целиком. Задачи берут результат из checkpoint'а — они не перевыполняются. Но код между задачами — перевыполняется. Если этот код содержит побочный эффект, он выполнится повторно. Вызов API без `@task` произойдёт дважды. Отправка email без `@task` — отправит письмо дважды. Запись в базу — задвоит запись.

Это как правило идемпотентности, только наоборот: вместо того чтобы делать операции идемпотентными, вы помечаете неидемпотентные операции декоратором `@task`, и LangGraph гарантирует их однократное выполнение через checkpoint'ы. Всё, что вне `@task`, считается «чистым» кодом и может быть перевыполнено.

> Полный пример: examples_09_functional_api.py, функция example_side_effects_gotcha()

## Часть 4: Retry и кэширование на уровне задачи

Декоратор `@task` принимает два необязательных параметра, которые делают задачи ещё устойчивее.

`retry_policy` — тот же `RetryPolicy` из `langgraph.types`, что мы разбирали в лекции 6. Если задача выбросила исключение — LangGraph повторяет её с экспоненциальным backoff. В лекции 6 мы подключали RetryPolicy через `add_node`. Здесь — через декоратор `@task`. Механизм один и тот же, но способ подключения соответствует Functional API.

`cache_policy` — механизм кэширования результатов. Если задача вызывается с теми же аргументами, что и раньше, результат берётся из кэша, а не вычисляется заново. Параметр `ttl` задаёт время жизни кэша в секундах. Это полезно для дорогих операций, которые не меняются часто — например, запрос к внешнему API, который обновляется раз в час. Зачем тратить деньги на повторный вызов, если результат тот же?

Retry и кэширование комбинируются. Задача, помеченная `@task(retry_policy=..., cache_policy=...)`, сначала проверяет кэш. Если там есть свежий результат — возвращает его без вызова. Если нет — выполняет функцию. Если функция упала — повторяет с retry. Несколько уровней защиты в одной строке декоратора.

> Полный пример: examples_09_functional_api.py, функция example_retry_and_cache()

## Часть 5: Параллельное выполнение задач

Вызов `@task`-функции возвращает future — объект-обещание результата. До тех пор, пока вы не вызвали `.result()`, задача может выполняться в фоне. Это открывает возможность для параллелизма.

Если задачи независимы — запустите их все, а потом соберите результаты:

```python
futures = [analyze(doc) for doc in documents]
results = [f.result() for f in futures]
```

Первая строка запускает задачи. Вторая блокирует и собирает результаты. Задачи выполняются параллельно — три документа обрабатываются одновременно, а не последовательно. Это аналог `Send` из лекции 8, но в функциональном стиле: вместо conditional edge с `Send`-объектами — обычный list comprehension.

Для асинхронного кода параллелизм ещё нагляднее. Асинхронные задачи возвращают awaitable-объекты, и вместо `.result()` можно использовать `await`. Но и в синхронном режиме LangGraph оптимизирует выполнение — задачи, запущенные до первого `.result()`, могут выполняться параллельно.

> Полный пример: examples_09_functional_api.py, функция example_parallel_tasks()

## Часть 6: entrypoint.final — разделение возвращаемого и сохраняемого

По умолчанию `@entrypoint`-функция возвращает значение и вызывающему коду, и в checkpoint (для параметра `previous` при следующем вызове). Но иногда нужно вернуть одно, а сохранить другое.

Представьте чат-бота. Каждый вызов должен вернуть только ответ на текущий вопрос — это то, что увидит пользователь. Но в checkpoint нужно сохранить всю историю разговора, чтобы при следующем вызове бот помнил контекст. Если вернуть всю историю — пользователь увидит не ответ, а массив сообщений. Если сохранить только ответ — бот потеряет память.

`entrypoint.final` решает эту дилемму. Вместо обычного `return` вы возвращаете `entrypoint.final(value=ответ, save=история)`. Параметр `value` — это то, что получит вызывающий код. Параметр `save` — то, что сохранится в checkpoint и будет доступно через `previous` при следующем вызове.

> Полный пример: examples_09_functional_api.py, функция example_entrypoint_final()

## Часть 7: previous — короткосрочная память между вызовами

Keyword-аргумент `previous` в entrypoint-функции — это доступ к результату предыдущего вызова на том же thread'е. При первом вызове `previous` равен `None`. При каждом последующем — содержит то, что было сохранено (или возвращено, если `entrypoint.final` не используется) в прошлый раз.

Это механизм короткосрочной памяти. Чат-бот, который помнит историю разговора. Аккумулятор, который накапливает данные между вызовами. Конвейер, который при каждом вызове обрабатывает новую порцию данных, но помнит результаты предыдущих.

Всё это работает благодаря persistence. Значение `previous` хранится в checkpoint, привязанном к `thread_id`. Разные thread'ы — разная память. Процесс может быть перезапущен — `previous` загрузится из хранилища. Та же архитектура, что в лекциях 4 и 5, но без явного графа и без явного состояния — LangGraph управляет всем под капотом.

> Полный пример: examples_09_functional_api.py, функция example_previous_memory()

## Часть 8: interrupt() в Functional API

Функция `interrupt()` работает в Functional API точно так же, как в StateGraph. Вызов останавливает выполнение, сохраняет состояние в checkpoint и передаёт значение вызывающему коду. Возобновление через `Command(resume=...)` подставляет переданное значение как возвращаемое из `interrupt()`.

Все паттерны HITL из лекции 5 применимы: Approval Gate, Review and Edit, Tiered Approval. Единственная разница — синтаксис. В StateGraph interrupt вызывается внутри узла. В Functional API — внутри entrypoint или task. Механизм один и тот же.

Важный момент: при возобновлении entrypoint перевыполняется с начала, но задачи берутся из checkpoint'а. Это значит, что `interrupt()` внутри `@task` работает корректно — при возобновлении задачи до прерывания не перевыполняются, а задача с `interrupt()` возобновляется с нужной точки.

> Полный пример: examples_09_functional_api.py, функция example_interrupt_in_functional()

## Часть 9: Кастомный стриминг

В StateGraph стриминг привязан к узлам: каждый узел генерирует один chunk в stream. В Functional API узлов нет, и стриминг реализуется через `get_stream_writer()` — функцию, которая возвращает writer для отправки кастомных данных в stream.

Writer можно вызывать в любой точке entrypoint: между задачами, внутри задачи, в условиях. Каждый вызов отправляет данные клиенту в реальном времени. Это удобно для индикации прогресса: «генерирую черновик», «оцениваю качество», «улучшаю третью итерацию». Пользователь видит, что агент работает, а не завис.

Для приёма кастомных данных клиент передаёт `stream_mode=["custom", "updates"]` в вызов `.stream()`. Режим `"custom"` — это данные от writer, `"updates"` — стандартные обновления состояния.

> Полный пример: examples_09_functional_api.py, функция example_custom_streaming()

## Часть 10: Когда Functional API, а когда StateGraph

Два подхода не конкурируют — они закрывают разные ниши. Выбор зависит от характера задачи.

Functional API идеален, когда логика в основном линейная — пусть даже с условиями, циклами и обработкой ошибок. Код получается компактнее и читабельнее, потому что оркестрация записывается обычным Python, без абстракций графа. Кривая обучения ниже — если вы знаете Python, вы уже знаете Functional API.

StateGraph необходим, когда нужны параллельные ветви через `Send` — динамический fan-out с десятками параллельных воркеров. Когда нужны подграфы — вложенные графы с собственным состоянием. Когда нужна визуализация — возможность нарисовать диаграмму графа, которой у Functional API нет, потому что графа в явном виде нет. И когда маршрутизация по-настоящему сложная — множество ветвей, переходов, условий, которые естественнее выражаются как граф, а не как вложенные if/else.

Оба подхода можно комбинировать. Task-функция может внутри вызвать скомпилированный подграф — если для какой-то части пайплайна нужна мощь StateGraph. А узел StateGraph может вызвать entrypoint-функцию — если какой-то узел проще реализовать функционально.

## Итоги

Functional API — альтернативный способ создания агентов в LangGraph, без явного графа. Два декоратора: `@entrypoint` определяет точку входа с checkpointing, `@task` оборачивает функцию в задачу с автоматическим сохранением результата. Оркестрация записывается обычным Python — условия, циклы, обработка ошибок, ранний выход.

Побочные эффекты обязаны быть внутри `@task` — при возобновлении entrypoint перевыполняется, а задачи берутся из checkpoint'а. `retry_policy` и `cache_policy` на задачах дают retry и кэширование одной строкой. `entrypoint.final` разделяет возвращаемое и сохраняемое значение. `previous` даёт короткосрочную память между вызовами. `interrupt()` работает так же, как в StateGraph.

Правило выбора: линейная логика с условиями → Functional API. Сложная маршрутизация, параллелизм, подграфы, визуализация → StateGraph. Оба подхода комбинируются.
