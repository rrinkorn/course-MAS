# Лекция 3: Агент с инструментами — tools, streaming, Command

## Введение: От графа к агенту

Разберём, как LangGraph работает с инструментами: механизм `bind_tools`, готовый исполнитель `ToolNode`, агентный цикл ReAct. Затем — все режимы потоковой обработки (streaming) и `Command` — механизм, который позволяет узлам самим решать, куда пойдёт граф дальше.

---

## Часть 1: Инструменты в LangGraph

### bind_tools: подключаем инструменты к модели

Вспомним, как работает механизм вызова инструментов (tool calling). Модель получает описание доступных функций в формате JSON Schema. Когда пользователь задаёт вопрос, модель решает: ответить текстом или вызвать функцию. Если вызвать — возвращает структурированный запрос (имя функции + аргументы), а не текст.

В LangChain инструменты привязываются к модели через метод `bind_tools`. Вы определяете Python-функции с декоратором `@tool`, собираете их в список и передаёте модели. После этого модель «знает» об этих функциях. Каждый инструмент должен иметь docstring — именно он становится описанием функции в JSON Schema, и по нему модель принимает решение, какой инструмент вызвать.

```python
tools = [multiply, web_search, get_weather]
llm = ChatOpenAI(model="gpt-5.4-mini").bind_tools(tools)
```

Когда модель решает вызвать инструмент, в ответе (`AIMessage`) появляется поле `tool_calls` — список вызовов с именами и аргументами. Если модель считает, что может ответить сама, `tool_calls` будет пустым, а ответ — обычным текстом.

> Полный пример: examples_7_3_agent_tools.py, функция example_bind_tools()

### ToolNode: кто выполняет вызовы

Модель **не выполняет** функции. Она только говорит: «хочу вызвать `multiply(17, 23)`» — возвращает `AIMessage` с полем `tool_calls`. Это просто запрос на вызов, но не сам вызов. Кто-то должен взять этот запрос и реально выполнить Python-функцию.

`ToolNode` — это готовый узел графа, который делает именно это: берёт запрос модели, находит нужную функцию, вызывает её и возвращает результат обратно в граф.

```python
from langgraph.prebuilt import ToolNode
tool_node = ToolNode(tools)  # передаём список функций, которые он может вызывать
```

По шагам:

1. Смотрит на последнее сообщение в состоянии — это `AIMessage` с `tool_calls`
2. Видит: `{name: "multiply", args: {a: 17, b: 23}}`
3. Находит функцию `multiply` в списке `tools`, вызывает `multiply(17, 23)` → `391`
4. Возвращает результат как `ToolMessage(content="391")`

Если модель запросила несколько инструментов за один ход (например, `get_weather` и `multiply` одновременно) — `ToolNode` выполнит все и вернёт несколько `ToolMessage`.

`ToolNode` — стандартный способ, который используется практически в каждом агенте. Когда нужно что-то сверх стандартного (логирование, обработка ошибок, валидация аргументов) — пишут кастомный узел (см. Часть 4).

> Полный пример: examples_7_3_agent_tools.py, функция example_tool_node()

### Собираем ReAct-агента вручную

Агентный цикл ReAct (Reasoning + Acting) — это два узла и одно условное ребро. Узел `agent` вызывает LLM с привязанными инструментами. Узел `tools` выполняет запрошенные вызовы через `ToolNode`. Условное ребро после `agent` проверяет, есть ли `tool_calls` в ответе модели: если есть — переходим к `tools`, если нет — модель ответила текстом и граф завершается.

```python
graph.add_edge(START, "agent")
graph.add_conditional_edges(
    "agent", 
    should_use_tools, 
    {"tools": "tools", "end": END},
)
graph.add_edge("tools", "agent")
```

Цикл работает следующим образом. Сначала модель получает сообщения и решает, что делать. Если она вернула `tool_calls`, выполняются инструменты и результаты возвращаются модели для следующего шага. Если `tool_calls` нет — модель ответила текстом, и граф завершает работу. Цикл повторяется, пока модель не решит ответить без инструментов. Это означает, что агент может выполнить цепочку из нескольких инструментов последовательно или вызвать несколько инструментов параллельно за один шаг.

Например, на вопрос «Какая погода в Москве и чему равно 2^10?» модель вызовет два инструмента параллельно (`get_weather` и `multiply`), получит оба результата и сформулирует единый ответ.

> Полный пример: examples_7_3_agent_tools.py, функция example_react_agent_manual()

### create_agent: быстрый старт

Паттерн ReAct настолько типичен, что для него существует фабричная функция `create_agent` (из пакета `langchain`). Хотя она импортируется из `langchain`, под капотом она создаёт LangGraph `StateGraph` — тот же граф с циклом `agent → tools → agent`. Возвращает скомпилированный граф, с которым работаешь как обычно (`invoke`, `stream`):

```python
from langchain.agents import create_agent

app = create_agent(model=llm, tools=tools)
app.invoke({"messages": [("user", "Какая погода в Москве?")]})
    
```

Фабрика удобна для прототипов и экспериментов. Когда нужна кастомная логика — например, динамический выбор инструментов, специфическая обработка ошибок или дополнительные поля в состоянии — переходите на ручную сборку.

### Кастомизация create_agent

Фабрика принимает дополнительные параметры. Можно задать системный промпт через `system_prompt`, подключить checkpointer для сохранения состояния между вызовами, настроить `interrupt_before` для паузы перед определёнными узлами. Параметр `system_prompt` особенно полезен: он задаёт роль и поведение агента, определяя, когда и какие инструменты использовать.

Начиная с LangChain 1.2.12, `create_agent` поддерживает несколько важных дополнительных параметров:

- `**response_format**` — Pydantic-модель для структурированного вывода. Агент вызывает инструменты как обычно, но финальный ответ приводится к указанной схеме. Результат доступен через `result["structured_response"]`. Это полезно, когда агент должен возвращать данные в фиксированном формате — для дальнейшей обработки или интеграции с другими системами.
- `**middleware**` — список middleware-обёрток, которые декларативно добавляют типовые паттерны: автоматический retry при ошибках модели (`ModelRetryMiddleware`), суммаризация длинной истории (`SummarizationMiddleware`) и другие. Middleware заменяют ручные реализации этих паттернов.
- `**context_schema**` — TypedDict, описывающий runtime-контекст, который передаётся через `config["configurable"]`. Агент получает доступ к данным пользователя, настройкам языка и другим параметрам, не засоряя историю сообщений.
- `**model` как строка** — модель можно передать строковым идентификатором, например `"openai:gpt-5.4-mini"`. LangChain автоматически создаст соответствующий объект. При работе через OpenRouter по-прежнему передаётся объект `ChatOpenAI`.

> **Примечание:** В старых туториалах вы можете встретить `create_react_agent` из `langgraph.prebuilt` — эта функция deprecated начиная с LangGraph 1.0 и будет удалена в версии 2.0. Используйте `create_agent` из `langchain.agents`.

> Полный пример: examples_7_3_agent_tools.py, функции example_react_agent_prebuilt(), example_react_agent_with_prompt() и example_create_agent_features()

## Часть 2: Command — маршрутизация изнутри узла

### Проблема: условные рёбра ограничены

Мы видели `add_conditional_edges` — функцию-маршрутизатор, которая смотрит на состояние и выбирает следующий узел. Это работает, но имеет ограничение: маршрутизатор — это отдельная функция, внешняя по отношению к узлам. Узел не может сам решить, куда пойдёт граф.

А иногда это нужно. Узел обработал данные и в процессе понял, что нужно не просто вернуть результат, а перенаправить выполнение в конкретный узел. Или обновить состояние определённым образом и одновременно задать направление.

### Command: решение

`Command` — объект, который узел может вернуть вместо обычного словаря. Он объединяет три действия: обновление состояния (поле `update`, как обычный возврат), перенаправление в конкретный узел (поле `goto`) и продолжение после прерывания (поле `resume`, для HITL — разберём в следующих лекциях).

Рассмотрим типичный сценарий — классификатор запросов в службе поддержки. Узел анализирует текст запроса и на основе ключевых слов решает, куда направить обработку: в техническую поддержку, на общий ответ или на эскалацию менеджеру. Без `Command` пришлось бы разделить эту логику на два места — узел классифицирует, а отдельная функция-маршрутизатор читает результат и направляет граф. С `Command` всё происходит в одном месте:

```python
def classifier(state) -> Command[Literal["technical", "general", "escalate"]]:
    if "баг" in state["query"].lower():
        return Command(update={"category": "technical"}, goto="technical")
    ...
```

Обратите внимание: в графе **нет `add_conditional_edges`**. Узел `classifier` сам решает, куда идти, через `Command(goto=...)`. Это чище и ближе к реальной логике — решение принимается там, где есть вся информация.

Типовая аннотация `Command[Literal["technical", "general", "escalate"]]` сообщает LangGraph, какие узлы являются возможными целями. Это нужно для корректного построения графа и его визуализации.

> Полный пример: examples_7_3_agent_tools.py, функция example_command_routing()

### Зачем нужен Command, если есть add_conditional_edges

Без `Command` классификатор пришлось бы писать так:

```python
# Нода сохраняет решение в state...
def classifier(state):
    if "баг" in state["query"]:
        return {"category": "technical"}

# ...а отдельная функция тут же читает его обратно
def route_by_category(state):
    return state["category"]

graph.add_conditional_edges("classifier", route_by_category, {
    "technical": "technical",
    "general": "general",
})
```

Нода положила `"technical"` в state, роутер тут же прочитал `"technical"` из state. Бессмысленный круг — решение уже принято внутри ноды, но приходится прогонять его через state и отдельную функцию.

`Command` убирает этого посредника — нода сразу говорит «обнови state и иди туда»:

```python
def classifier(state):
    if "баг" in state["query"]:
        return Command(update={"category": "technical"}, goto="technical")
```

Одно действие вместо двух.

**Когда что использовать:**

- `**add_conditional_edges`** — нода делает свою работу (вызов LLM, обработку данных), а маршрутизация — побочная проверка результата. Нода возвращает обычный `dict`, отдельная функция решает куда дальше.
- `**Command`** — вся задача ноды — это решить, куда направить. Нода и есть роутер. Возвращает `Command` вместо `dict`.

## Часть 3: Streaming — потоковая обработка

### Зачем стримить

Когда агент выполняет сложную задачу (три вызова инструментов, два раунда рассуждений), пользователь может ждать десятки секунд. Без стриминга — пустой экран. Со стримингом — пользователь видит процесс в реальном времени: какие инструменты вызываются, какие результаты приходят, как формируется ответ.

LangGraph поддерживает пять режимов стриминга. Разберём каждый.

### Режим "values": полное состояние после каждого узла

В режиме `"values"` каждый chunk, приходящий из `app.stream()`, представляет собой полный снимок состояния графа после очередного узла. Можно посмотреть, сколько сообщений накопилось, каков тип последнего сообщения, какие поля обновились. Это удобно для отладки — вы видите всю картину на каждом шаге. Однако для продакшн-UI это избыточно: при большой истории сообщений на каждом шаге передаётся весь массив данных.

```python
for chunk in app.stream(input, stream_mode="values"):
    messages = chunk["messages"]
    print(f"Сообщений: {len(messages)}")
```

> Полный пример: examples_7_3_agent_tools.py, функция example_streaming_values()

### Режим "updates": только изменения (по умолчанию)

Режим `"updates"` передаёт только то, что изменил конкретный узел на данном шаге. Каждый chunk — это словарь вида `{имя_узла: обновление}`. Это компактно и наглядно: вы видите, что узел `agent` вернул `AIMessage` с `tool_calls`, узел `tools` вернул два `ToolMessage` с результатами, а затем `agent` сформировал финальный текстовый ответ. Этот режим используется по умолчанию, если `stream_mode` не указан.

> Полный пример: examples_7_3_agent_tools.py, функция example_streaming_updates()

### Режим "messages": поток токенов

Для пользовательского интерфейса, где нужен посимвольный вывод (как в ChatGPT), существует режим `"messages"`. Он передаёт каждый токен по мере генерации. Каждый элемент потока — это кортеж `(message, metadata)`, где `message` содержит фрагмент текста, а `metadata` указывает, от какого узла пришёл токен. Фильтрация по `metadata["langgraph_node"]` нужна, чтобы показывать только финальный ответ агента, а не промежуточные сообщения от инструментов.

```python
for message, metadata in app.stream(input, stream_mode="messages"):
    if message.content and metadata["langgraph_node"] == "agent":
        print(message.content, end="", flush=True)
```

> Полный пример: examples_7_3_agent_tools.py, функция example_streaming_messages()

### Режим "custom": пользовательские события

Узлы могут отправлять произвольные события в поток с помощью `get_stream_writer()`. Внутри узла вы получаете объект `writer` и вызываете его с любым словарём — статусом обработки, промежуточными результатами, метриками производительности. Эти события не становятся частью состояния графа, но доступны клиенту через `stream_mode="custom"`. Это позволяет передавать прогресс выполнения, что особенно ценно для длительных операций — пользователь видит «Начинаю поиск...», «Найдено 5 результатов, анализирую...», «Формирую ответ» вместо пустого экрана.

> Полный пример: examples_7_3_agent_tools.py, функция example_streaming_custom()

### Множественные режимы одновременно

Можно комбинировать несколько режимов стриминга в одном вызове, передав список в `stream_mode`. Каждый элемент потока в этом случае — кортеж, где первый элемент указывает режим (`"updates"`, `"messages"` и т.д.), а второй — данные. Это полезно, когда UI одновременно показывает и структурированный прогресс (какие узлы отработали), и посимвольный вывод ответа.

> Полный пример: examples_7_3_agent_tools.py, функция example_streaming_multiple_modes()

### Асинхронный streaming

Для продакшн-серверов на FastAPI или Starlette используйте метод `astream` вместо `stream`. Паттерн интеграции — Server-Sent Events (SSE): FastAPI-эндпоинт возвращает `StreamingResponse`, внутри которого асинхронный генератор итерирует по `app.astream()` и отдаёт каждый токен клиенту. Это стандартный способ потокового вывода для веб-приложений, который поддерживается всеми современными браузерами и HTTP-клиентами.

> Полный пример: examples_7_3_agent_tools.py, функция example_streaming_async_fastapi()

### Streaming v2: новый формат ответов (LangGraph 1.1.2+)

Начиная с LangGraph 1.1.2, доступен протокол streaming v2, который включается параметром `version="v2"` в `invoke` и `stream`. Новый протокол решает несколько проблем v1: унифицирует формат ответов, добавляет типизацию и встроенную поддержку прерываний (interrupts).

**invoke с version="v2"** возвращает объект `GraphOutput` вместо обычного словаря. У `GraphOutput` два атрибута: `.value` (собственно состояние графа — то, что раньше возвращалось как dict) и `.interrupts` (кортеж прерываний, если граф был приостановлен). Доступ к данным — через `.value`:

```python
result = app.invoke(input, version="v2")
state = result.value          # dict с состоянием
interrupts = result.interrupts # tuple прерываний (или пустой)
last_msg = result.value["messages"][-1]
```

Обратите внимание: dict-style доступ к `GraphOutput` (например, `result["messages"]`) по-прежнему работает, но выдаёт `DeprecationWarning` и будет удалён в LangGraph 3.0. Используйте `.value`.

**stream с version="v2"** выдаёт типизированные `StreamPart`-словари вместо «сырых» данных. Каждый chunk содержит ключи: `type` (тип события — `"values"`, `"updates"` и т.д.), `ns` (пространство имён, для подграфов), `data` (собственно данные) и `interrupts` (прерывания, если есть):

```python
for chunk in app.stream(input, stream_mode="values", version="v2"):
    print(chunk["type"])       # "values"
    print(chunk["data"])       # dict с состоянием
    print(chunk["interrupts"]) # tuple прерываний
```

По умолчанию используется v1 для обратной совместимости. Рекомендуется переходить на v2 в новых проектах — он станет стандартом в будущих версиях.

> Полный пример: examples_7_3_agent_tools.py, функция example_streaming_v2()

## Часть 4: Продвинутые паттерны с инструментами

### Предобработка перед вызовом инструмента

Иногда нужно модифицировать аргументы перед вызовом инструмента — добавить API-ключ, проверить права доступа, залогировать вызов. В этом случае вместо стандартного `ToolNode` пишется кастомный узел, который вручную итерирует по `tool_calls`, находит соответствующие функции и вызывает их.

Ключевой момент — обработка ошибок. При ошибке инструмента не следует бросать исключение, которое обрушит весь граф. Вместо этого возвращайте `ToolMessage` с описанием ошибки. Модель увидит ошибку и может скорректировать подход — попробовать другие аргументы или выбрать другой инструмент. Это делает агента устойчивым к сбоям внешних сервисов и некорректным входным данным.

> Полный пример: examples_7_3_agent_tools.py, функция example_custom_tool_node()

### Ограничение количества итераций

Без ограничений агент может зациклиться: вызывать инструменты бесконечно, если модель не может решить задачу. LangGraph предоставляет параметр `recursion_limit`, который задаёт максимальное количество шагов (посещений узлов):

```python
result = app.invoke(input, config={"recursion_limit": 15})
```

По умолчанию `recursion_limit=25`. Если лимит превышен, выбрасывается `GraphRecursionError`. Для отладки рекомендуется ставить низкое значение (5-10), чтобы быстро обнаруживать зацикливания.

> Полный пример: examples_7_3_agent_tools.py, функция example_recursion_limit()

### Фильтрация инструментов по контексту

Не всегда нужно давать модели все инструменты. Когда инструментов десятки, модель может ошибаться в выборе — вызывать не тот инструмент или путаться в аргументах. Решение — динамически выбирать набор инструментов в зависимости от контекста запроса. Узел агента анализирует текст последнего сообщения и привязывает к модели только релевантные инструменты через `bind_tools`. Если в вопросе упоминается погода — даём только погодный инструмент, если математика — только калькулятор. Меньше инструментов означает более точный выбор и меньше ошибок.

> Полный пример: examples_7_3_agent_tools.py, функция example_dynamic_tool_filtering()

## Итоги

В этой лекции мы превратили граф в полноценного агента.

**bind_tools + ToolNode** — стандартный способ подключения инструментов. Модель решает, что вызывать, `ToolNode` выполняет. Цикл `agent → tools → agent` — базовый паттерн ReAct.

**create_agent** — фабричная функция для быстрого прототипирования (ранее `create_react_agent`). Импортируется из `langchain`, но под капотом создаёт LangGraph `StateGraph`.

**Command** — механизм маршрутизации из узла. `Command(update=..., goto=...)` обновляет состояние и направляет граф одновременно. Используйте, когда вся задача ноды — решить, куда направить.

**Streaming** — пять режимов: `values` (полное состояние), `updates` (изменения), `messages` (токены), `custom` (пользовательские события), множественные режимы. Для UI — `messages`, для отладки — `updates`. Протокол v2 (`version="v2"`) возвращает типизированные `GraphOutput` и `StreamPart`.

**Продвинутые паттерны** — кастомный узел инструментов с обработкой ошибок, `recursion_limit` для защиты от зацикливания, динамическая фильтрация инструментов по контексту.
