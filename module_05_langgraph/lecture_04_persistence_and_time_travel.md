# Лекция 4: Persistence и Time Travel — сохранение состояния и история выполнения

> **Примечание:** Persistence и Time Travel — термины из официальной документации LangGraph. Подробнее:
>
> - [Persistence](https://docs.langchain.com/oss/python/langgraph/persistence)
> - [How-to: Time Travel](https://docs.langchain.com/oss/python/langgraph/use-time-travel)

## Введение

До сих пор наши графы страдали тяжёлой формой амнезии. Каждый вызов `invoke` начинался с чистого листа. Пользователь задал вопрос, получил ответ — и всё исчезло. Следующий вызов не знает ничего о предыдущем.

Но есть и вторая проблема: отладка. Агент совершил ошибку на третьем шаге из десяти. Чтобы воспроизвести ситуацию, нужно заново выполнить все предыдущие шаги, потратить деньги на API-вызовы — и нет гарантии, что стохастическая модель выдаст тот же ответ.

Persistence — это механизм (сохраняй снимки). Time Travel — это то, что этот механизм делает возможным (перемещайся по снимкам, ветвись).

## Часть 1: Checkpointers — хранители состояния

### Что такое checkpoint

Checkpoint — снимок состояния графа после выполнения узла. Каждый раз, когда узел завершает работу, LangGraph автоматически сериализует и сохраняет полный снимок — при условии, что при компиляции был указан checkpointer.

Что конкретно хранится внутри checkpoint'а? Четыре вещи. Во-первых, **полное состояние** — все поля вашего `TypedDict` со всеми значениями на момент завершения узла. Если у вас `MessagesState`, это будет полный список сообщений. Во-вторых, **метаданные** — какой узел только что выполнился, какие узлы должны выполниться следующими (поле `next`), и временная метка создания. В-третьих, **уникальный `checkpoint_id`** — идентификатор этого конкретного снимка. В-четвёртых, **ссылка на родительский checkpoint** (`parent_id`) — это то, что превращает набор изолированных снимков в связанную цепочку.

Когда именно создаются checkpoint'ы? При каждом переходе графа. Рассмотрим граф с тремя узлами: `research → write → edit`. После `invoke` будет создано четыре checkpoint'а: начальный (сразу после входа, перед первым узлом), после `research`, после `write` и после `edit`. Начальный checkpoint фиксирует входные данные — без него невозможно было бы воспроизвести самый первый шаг. Каждый последующий содержит `parent_id`, указывающий на предыдущий, образуя цепочку:

```text
ck-0 (start) → ck-1 (after research) → ck-2 (after write) → ck-3 (after edit)
```

### MemorySaver — для разработки

Чтобы включить persistence, достаточно одной строки при компиляции графа. Создаётся экземпляр `MemorySaver`, и передаётся в `graph.compile()`:

```python
memory = MemorySaver()
app = graph.compile(checkpointer=memory)
```

Больше ничего менять не нужно — ни узлы, ни рёбра, ни состояние. Граф остаётся тем же, но теперь он умеет запоминать. Каждый вызов `invoke` будет автоматически сохранять checkpoint после каждого узла, а следующий вызов сможет продолжить с того места, где остановился предыдущий.

Под капотом `MemorySaver` — обычный Python-словарь в оперативной памяти процесса. Ключом служит пара `(thread_id, checkpoint_id)`, значением — сериализованное состояние. Это значит: пока процесс жив — всё работает, процесс умер — данные потеряны. Никакого сброса на диск, никакой персистентности между перезапусками. Именно поэтому `MemorySaver` годится только для разработки и тестов — но для этих целей он идеален: нулевая конфигурация, мгновенный старт, не нужен внешний сервис.

> Полный пример: examples_7_4_persistence.py, функция example_memory_saver()

### thread_id — ключ к разговору

Чтобы различать разные разговоры, при вызове указываем `thread_id` в конфигурации:

```python
config = {"configurable": {"thread_id": "user-alice-001"}}
result = app.invoke({"messages": [HumanMessage(content="...")]}, config=config)
```

Все вызовы с одним `thread_id` образуют единый поток: при каждом новом вызове загружается последний checkpoint для этого потока, новое сообщение добавляется через reducer `add_messages`, и модель видит всю предыдущую историю. Разные `thread_id` полностью изолированы — Алиса и Боб ведут параллельные разговоры, не зная друг о друге.

Под капотом механизм прост: при первом вызове с данным `thread_id` checkpoint создаётся с нуля. При втором — загружается последний существующий checkpoint, к нему применяется новый ввод, и граф выполняется дальше. Это и есть память.

Важно понимать, что `thread_id` — произвольная строка. LangGraph не навязывает формат. Но на практике имеет смысл закладывать в неё семантику: `"user-alice-001"`, `"session-2025-03-21-abc"`, `"order-processing-7842"`. Это упростит отладку, когда checkpoint'ов станут тысячи.

> Полный пример: examples_7_4_persistence.py, функция example_thread_id()

### Checkpointers для продакшна

`MemorySaver` хранит всё в оперативной памяти — при перезапуске процесса данные теряются. Для продакшн-приложений LangGraph предлагает два основных варианта.

**PostgresSaver** — основной выбор для продакшна. Данные хранятся на диске, переживают перезапуски, поддерживают транзакции. Рекомендуется асинхронный вариант `AsyncPostgresSaver`, который при первом запуске автоматически создаёт необходимые таблицы через метод `setup()`.

**RedisSaver** — для высоконагруженных систем, где критична минимальная латентность. Redis хранит данные в оперативной памяти с опциональным сбросом на диск, что даёт скорость на порядок выше PostgreSQL ценой меньшей надёжности при отказах.

На практике выбор прост: для разработки и тестов используйте `MemorySaver`, для продакшна — `PostgresSaver` (он покрывает 90% сценариев), для высоконагруженных систем с требованиями к латентности — `RedisSaver`.

## Часть 2: Управление состоянием и recovery

### Просмотр текущего состояния

В любой момент можно запросить текущее состояние потока через `get_state()`. Метод возвращает объект `StateSnapshot`, у которого есть несколько ключевых полей:

- `values` — словарь с текущим состоянием (все поля вашего `TypedDict`)
- `next` — кортеж имён узлов, которые должны выполниться следующими. Пустой кортеж `()` означает, что граф завершён
- `config` — конфигурация этого конкретного checkpoint'а (включая `checkpoint_id`)
- `created_at` — временная метка создания
- `parent_config` — конфигурация родительского checkpoint'а (позволяет «пройти» назад по цепочке)

```python
state = app.get_state(config)
print(state.values)    # Текущее состояние
print(state.next)       # Какие узлы следующие
```

Это незаменимый инструмент как для отладки (понять, в каком состоянии оказался граф), так и для построения интерфейсов — например, показать пользователю прогресс многошагового агента или решить, нужно ли вмешательство человека.

> Полный пример: examples_7_4_persistence.py, функция example_get_state()

### Обновление состояния извне

Метод `update_state()` позволяет изменить состояние потока без выполнения графа. Это полезно, когда нужно внедрить внешнюю информацию — например, результат верификации пользователя или данные из другой системы.

Важный нюанс: `update_state` применяет reducer'ы. Если у поля `messages` reducer `add_messages`, то новое сообщение **добавляется** к истории, а не заменяет её целиком. Это то же поведение, как если бы сообщение вернул узел — просто источником является внешний код, а не граф.

Метод создаёт новый checkpoint с изменениями. Это значит, что обновление не модифицирует существующий снимок — оно порождает новый, сохраняя возможность вернуться к предыдущему состоянию.

> Полный пример: examples_7_4_persistence.py, функция example_update_state()

### Recovery после сбоев

Одно из главных преимуществ persistence — устойчивость к сбоям. Если граф упал на середине выполнения (сеть отвалилась, API вернул ошибку, процесс перезапустился), вся работа до момента сбоя сохранена в checkpoint'ах.

Для продолжения используется вызов `invoke(None, config=config)`. Почему `None`? Потому что мы не передаём новый ввод — мы просим LangGraph загрузить последний checkpoint для этого `thread_id` и продолжить выполнение с того узла, на котором произошёл сбой. Граф смотрит на поле `next` в checkpoint'е, видит, что узел X не был завершён, и выполняет его заново. Узлы, выполненные до сбоя, не повторяются — их результат уже зафиксирован в предыдущих checkpoint'ах.

```python
try:
    result = app.invoke(initial_state, config=config)
except Exception:
    # Последний checkpoint сохранён — продолжаем с него
    result = app.invoke(None, config=config)
```

Этот механизм естественно ложится в паттерн retry с exponential backoff: при каждой попытке проверяем через `get_state()`, есть ли незавершённый поток (поле `next` не пустое), и если да — продолжаем его через `invoke(None, ...)`, а не начинаем заново. Так мы получаем отказоустойчивость практически бесплатно — persistence, который уже настроен для памяти, автоматически даёт нам и устойчивость к сбоям.

> Полный пример: examples_7_4_persistence.py, функции example_state_recovery() и example_retry_with_backoff()

## Часть 3: Store — долгосрочная память между потоками

### Checkpointer vs Store: два вида памяти

Прежде чем разбираться в деталях, важно понять концептуальное различие. Checkpointer и Store решают разные задачи, и путать их — частая ошибка.

**Checkpointer** — это *эпизодическая* память. Он помнит, что произошло в конкретном разговоре: какие сообщения были, какие узлы выполнились, в каком порядке. Каждый поток (`thread_id`) — отдельный эпизод, изолированный от остальных. Алиса может вести 50 разговоров, и в каждом из них checkpointer хранит полную историю. Но разговор номер 51 не знает ничего о предыдущих пятидесяти.

**Store** — это *семантическая* память. Он хранит знания поверх потоков: кто такой пользователь, какие у него предпочтения, какие факты выяснились за все разговоры. Store существует вне конкретного потока — данные, записанные в одном thread'е, доступны в любом другом.

### Кто и как кладёт данные в Store

Важное отличие от checkpointer'а: checkpointer работает автоматически — разработчик подключил его при компиляции, и дальше LangGraph сам сохраняет состояние после каждого узла. Store **не работает автоматически**. Это пустое хранилище, и только код внутри узла решает, что туда записать и когда прочитать.

Store подключается при компиляции графа: `graph.compile(checkpointer=..., store=store)`. После этого узлы получают доступ к нему через keyword-аргумент `store: BaseStore`. Дальше — ответственность разработчика: именно он пишет логику «что запомнить» и «когда вспомнить».

### Пространства имён (namespaces)

Данные в Store организованы через **пространства имён**. Namespace — это кортеж строк, образующий иерархический путь, похожий на директории в файловой системе: `("user_memories", "alice")`, `("preferences", "bob")`, `("knowledge", "python")`.

Namespace'ы **не нужно создавать заранее**. Нет никакого `store.create_namespace(...)`. Пространство появляется автоматически при первом вызове `put()` — как `mkdir -p` в Unix или `defaultdict` в Python. Написали `store.put(("user_memories", "alice"), ...)` — namespace `("user_memories", "alice")` существует. Внутри каждого пространства лежат записи: у каждой уникальный ключ (строка) и значение (словарь).

```python
# Чтение — получаем все записи в пространстве
memories = store.search(("user_memories", user_id))

# Запись — namespace создастся автоматически при первом put
store.put(("user_memories", user_id), key="mem-0", value={"text": "..."})
```

Стратегия организации namespace'ов зависит от задачи. Для пользовательских предпочтений — `("preferences", user_id)`. Для фактов, извлечённых из разговоров — `("facts", user_id)`. Для общих знаний, не привязанных к пользователю — `("knowledge", topic)`. Главное правило: один namespace = один тип информации об одной сущности.

Для разработки используется `InMemoryStore` — аналогично `MemorySaver`, данные живут в оперативной памяти и пропадают при перезапуске. В продакшне Store может опираться на те же бэкенды, что и checkpointer — PostgreSQL, Redis.

> Полный пример: examples_7_4_persistence.py, функция example_store()

## Часть 4: Time Travel — история и воспроизведение

### get_state_history: вся история выполнения

Каждый выполненный узел создаёт checkpoint. Метод `get_state_history()` возвращает всю цепочку checkpoint'ов для данного потока в обратном хронологическом порядке. Для каждого checkpoint'а доступен его `checkpoint_id`, информация о следующих узлах и полное состояние на тот момент.

Представим граф из трёх узлов: research, write, edit. После полного выполнения история будет содержать четыре checkpoint'а — начальный (перед research), после research (перед write), после write (перед edit) и финальный (после edit, следующих узлов нет). Каждый из них — полноценный снимок, к которому можно вернуться.

> Полный пример: examples_7_4_persistence.py, функция example_time_travel()

### Replay: перезапуск с любой точки

Допустим, researcher сделал хорошую работу, а writer — плохую. Вместо того чтобы перезапускать весь граф с нуля (и повторять дорогой вызов research), можно найти checkpoint «после research» и перезапустить оттуда. Для этого итерируемся по истории, находим checkpoint с `next == ("write",)` и вызываем `invoke(None, config=target_config)`. Узел research не повторяется — его результат уже сохранён в checkpoint'е.

```python
for state in app.get_state_history(config):
    if state.next == ("write",):
        result = app.invoke(None, config=state.config)
        break
```

Экономия очевидна: один API-вызов сохранён, контекст точно воспроизведён без стохастических вариаций.

> Полный пример: examples_7_4_persistence.py, функция example_replay()

### Replay с модификацией

Можно не только перезапустить с точки, но и изменить состояние перед перезапуском. Метод `update_state()` принимает параметр `as_node`, который указывает, от имени какого узла применяется обновление.

Зачем нужен `as_node`? LangGraph использует его, чтобы определить, какой узел будет **следующим**. Если мы указываем `as_node="research"`, LangGraph интерпретирует это так: «узел research добавил ещё одно сообщение к своему выводу». Значит, следующим по графу должен выполниться `write` — именно он идёт после `research`. Если бы мы указали `as_node="write"`, следующим был бы `edit`.

Что будет, если указать неправильный `as_node` — например, узел, которого нет в графе? LangGraph выбросит ошибку. А если указать узел, который не является предшественником текущей точки? Обновление применится, но следующий узел определится по рёбрам графа от указанного узла — результат может быть неожиданным. Поэтому `as_node` нужно выбирать осознанно: это не просто метка, а инструкция для маршрутизации.

```python
app.update_state(
    target_config,
    {"messages": [HumanMessage(content="Пиши в научно-популярном стиле")]},
    as_node="research",  # Следующим выполнится write
)
result = app.invoke(None, config=target_config)
```

> Полный пример: examples_7_4_persistence.py, функция example_replay_with_modification()

### Fork: ветвление выполнения

Fork создаёт новую ветку из существующего checkpoint'а. Оригинальная история остаётся нетронутой — это не перемотка, а именно ветвление.

До этого момента история checkpoint'ов была линейной цепочкой: каждый снимок имел ровно одного потомка. Fork превращает эту цепочку в **дерево**. Из одного checkpoint'а теперь могут расходиться несколько веток, каждая со своей последующей историей:

```text
ck-0 (start)
  └── ck-1 (after research)
        └── ck-2 (after write)
              ├── ck-3 (after edit)     ← оригинал
              └── ck-4 (forked)         ← альтернатива
```

Оригинальная и альтернативная ветки существуют параллельно, и к любой из них можно вернуться в любой момент. Это открывает практические возможности: A/B тестирование промптов (одно и то же начало, но разные инструкции на этапе write), исследование альтернативных стратегий (что если дать агенту другие факты?), коррекция ошибок без потери удачного контекста.

> Полный пример: examples_7_4_persistence.py, функция example_fork()

## Часть 5: Debugging — практические техники

### Техника 1: LangSmith

LangSmith — платформа для наблюдения за LLM-приложениями. Достаточно установить две переменные окружения (`LANGSMITH_TRACING` и `LANGSMITH_PROJECT`), и все вызовы записываются автоматически. В интерфейсе LangSmith видно дерево вызовов, полные промпты и ответы, время выполнения, количество токенов и стоимость каждого шага. Для сложных агентов это незаменимый инструмент.

> Полный пример: examples_7_4_persistence.py, функция example_debug_langsmith()

### Техника 2: stream_mode для локальной отладки

Если LangSmith не подключён или нужна быстрая отладка прямо в терминале, LangGraph предлагает несколько режимов стриминга. Они управляются параметром `stream_mode` в методе `stream()`:

- `"values"` — после каждого узла выводит полное состояние. Удобно, чтобы видеть, как состояние меняется шаг за шагом
- `"updates"` — показывает только дельту: что конкретно изменил каждый узел. Это самый полезный режим для отладки — сразу видно, если узел не вернул ожидаемый ключ или вернул пустой результат
- `"debug"` — самый подробный режим: включает метаданные о каждом шаге, информацию о задачах и checkpoint'ах

```python
for event in app.stream(inputs, config=config, stream_mode="updates"):
    print(event)  # Видим, что изменил каждый узел
```

На практике `"updates"` покрывает 90% потребностей в отладке. `"debug"` полезен, когда нужно понять, почему граф пошёл не по тому ребру.

### Техника 3: визуализация графа

LangGraph умеет рендерить структуру графа в нескольких форматах: ASCII-диаграмма прямо в терминале через `get_graph().print_ascii()`, Mermaid-разметка для Jupyter и документации, PNG-изображение для презентаций. Параметр `xray=True` развёртывает подграфы, показывая внутреннюю структуру вложенных графов.

Визуализация особенно полезна перед запуском — убедиться, что рёбра ведут туда, куда ожидается, и нет «островков» (узлов без входящих рёбер).

> Полный пример: examples_7_4_persistence.py, функция example_debug_visualization()

### Типичные ошибки

**Бесконечный цикл** — условие выхода из цикла никогда не выполняется. Лечение: `recursion_limit` как страховка плюс логирование маршрутизатора, чтобы понять, почему агент не выходит из цикла.

**Потерянное состояние** — узел возвращает ключ, которого нет в TypedDict. LangGraph игнорирует такие ключи молча, и данные теряются без ошибок. Лечение: `stream_mode="updates"` для проверки того, что каждый узел вносит ожидаемые изменения.

**Проблемы с сериализацией** — состояние содержит несериализуемые объекты: файловые дескрипторы, сетевые соединения, генераторы. При сохранении в PostgreSQL или Redis это приведёт к ошибке. С `MemorySaver` такой код может работать (он просто хранит ссылки в памяти), но сломается при переходе на продакшн-checkpointer. Правило: хранить в состоянии только примитивы, списки, словари и объекты LangChain (сообщения, документы).

## Часть 6: Управление длиной истории

### Проблема роста контекста

Со временем история сообщений в потоке разрастается. Модель видит всё больший контекст — это дороже, медленнее и в какой-то момент упирается в лимит контекстного окна. Есть два подхода к решению, и они не взаимоисключающие.

### Подход 1: trim_messages — обрезка только для LLM

Функция `trim_messages` из LangChain обрезает историю по количеству токенов перед отправкой модели. Ключевой момент: обрезка происходит **внутри узла**, а не в состоянии. Полная история остаётся в checkpoint'ах нетронутой — модель просто видит окно последних N токенов.

Параметр `strategy="last"` означает: сохранить последние сообщения. `start_on="human"` гарантирует, что обрезанная история начинается с сообщения пользователя, а не с оборванного ответа ассистента. `include_system=True` сохраняет системное сообщение даже при обрезке — иначе модель «забудет» свою роль.

Когда использовать: практически всегда в долгоживущих чат-ботах. Это безопасный подход — данные не теряются, Time Travel работает с полной историей, но модель не захлёбывается от объёма контекста.

### Подход 2: RemoveMessage — обрезка в самом состоянии

Если разговор длится сотни сообщений, полная история раздувает checkpoint'ы. Каждый checkpoint хранит копию всего состояния — при 500 сообщениях это ощутимо, особенно с `PostgresSaver`. `RemoveMessage` решает эту проблему, удаляя старые сообщения из самого состояния.

Механика: узел-очиститель возвращает список объектов `RemoveMessage(id=message.id)`. Reducer `add_messages` обрабатывает их как инструкции на удаление — находит сообщения по `id` и убирает из списка. Удалённые сообщения исчезают из состояния навсегда — в отличие от `trim_messages`, это необратимая операция.

Когда использовать: когда checkpoint'ы разрослись настолько, что влияют на производительность. На практике это тысячи сообщений и выше. Для большинства приложений `trim_messages` достаточно.

Два подхода можно комбинировать: `trim_messages` для ежедневной работы (модель видит окно, состояние полное), а `RemoveMessage` как периодическая «уборка» для долгоживущих потоков.

> Полный пример: examples_7_4_persistence.py, функции example_trim_history() и example_remove_old_messages()

## Итоги

Persistence и Time Travel — два столпа надёжных агентных приложений.

**Checkpointer** автоматически сохраняет состояние после каждого узла. `MemorySaver` для разработки, `PostgresSaver` / `RedisSaver` для продакшна.

**thread_id** изолирует разговоры. Один поток = одна цепочка checkpoint'ов.

**Recovery** — при сбое `invoke(None, config)` продолжает с последнего checkpoint'а.

**Store** — долгосрочная память поверх потоков. Checkpointer = история разговора, Store = профиль пользователя.

**get_state_history** — полная история выполнения для анализа и отладки.

**Replay** — перезапуск с любого checkpoint'а, с возможностью модификации состояния.

**Fork** — ветвление выполнения для A/B тестирования и исследования альтернатив.

**Debugging**: LangSmith для трейсинга, визуализация графа для проверки структуры.
