# Лекция 5: Human-in-the-Loop — когда агенту нужен человек

## Введение: Автономность vs контроль

Идеальный агент делает всё сам. Реальный агент делает глупости. Он может отправить письмо не тому человеку, удалить не тот файл, потратить бюджет на бессмысленные API-вызовы. Чем больше свободы у агента, тем больше ущерб от ошибки.

Human-in-the-Loop (HITL) — это архитектурный паттерн, при котором агент приостанавливается в критических точках и ждёт решения человека. Не каждый шаг требует одобрения — только те, где цена ошибки высока: необратимые действия, внешние коммуникации, финансовые операции.

Представьте систему допусков на заводе. Рядовой рабочий включает станок сам — это рутина, риска нет. Но запуск нового производственного цикла требует подписи мастера. А изменение рецептуры — одобрения технолога. Каждый уровень риска — свой уровень контроля. Агент с HITL работает так же: безопасные действия выполняет автоматически, а перед опасными останавливается и ждёт.

LangGraph делает HITL первоклассной возможностью графа, а не костылём поверх него. Граф может остановиться, сохранить состояние, дождаться ответа человека через минуту или через неделю — и продолжить с того же места. Под капотом работает тот же механизм persistence из лекции 4: checkpoint хранит полное состояние, а `thread_id` связывает все шаги в единую сессию.

## Часть 1: Функция interrupt — как остановить граф

Самый прямой способ приостановить граф — вызвать функцию `interrupt()` внутри узла. Когда выполнение доходит до этого вызова, граф останавливается, состояние сохраняется в checkpoint, а значение, переданное в `interrupt()`, возвращается вызывающему коду — это то, что увидит человек. Граф «засыпает» и ждёт явного продолжения.

Рассмотрим типичный сценарий. Агент составляет черновик письма. Человек должен проверить текст перед отправкой. Граф состоит из трёх узлов: `draft` генерирует текст, `review` показывает его человеку, `send` выполняет отправку. Ключевой момент — в узле `review`:

```python
decision = interrupt({
    "draft": last_msg,
    "question": "Отправить это письмо? (approve / edit / reject)"
})
```

Этот вызов останавливает граф и передаёт человеку черновик вместе с вопросом. Код после `interrupt()` выполнится только тогда, когда человек ответит. Переменная `decision` получит значение, переданное через `Command(resume=...)`.

Важно понимать, что `interrupt()` — это не просто пауза в коде. Это полноценный механизм персистентности. Процесс может быть перезапущен, сервер — перезагружен. Состояние графа живёт не в оперативной памяти, а в checkpoint-хранилище. Когда придёт `Command(resume=...)`, LangGraph загрузит checkpoint и продолжит выполнение узла ровно с того места, где был вызван `interrupt()`. Между паузой и продолжением может пройти секунда, час или неделя — для графа разницы нет.

## Часть 2: Полный цикл — запуск, пауза, продолжение

Полный цикл HITL-взаимодействия состоит из трёх шагов, и каждый стоит разобрать подробно.

На первом шаге вы запускаете граф обычным `app.stream(...)` с входными данными и конфигурацией, содержащей `thread_id`. Граф выполняет узлы последовательно, пока не дойдёт до `interrupt()`. В этот момент в stream приходит специальный chunk с ключом `__interrupt__`, содержащий данные, которые вы передали в `interrupt()`. Именно этот chunk ваш frontend покажет пользователю.

На втором шаге человек изучает полученные данные и принимает решение. Это может происходить в веб-интерфейсе, мессенджере, консоли — в любом канале связи. Между первым и третьим шагом может пройти произвольное время, потому что состояние хранится не в памяти процесса, а в checkpoint-хранилище.

На третьем шаге вы продолжаете граф, передавая решение человека:

```python
for chunk in app.stream(
    Command(resume={"action": "approve"}),
    config=config
):
    print(chunk)
```

Граф загружает сохранённое состояние, подставляет переданное значение в переменную `decision` внутри узла и продолжает выполнение. Обратите внимание: тот же `config` с тем же `thread_id` связывает все три шага в единую сессию. Без `thread_id` LangGraph не знал бы, какой именно приостановленный граф нужно продолжить.

> Полный пример: examples_7_5_hitl.py, функция example_interrupt()

## Часть 3: Декларативные прерывания — interrupt_before и interrupt_after

Помимо вызова `interrupt()` внутри узла, есть более простой, декларативный подход. Точки прерывания можно объявить при компиляции графа:

```python
app = graph.compile(
    checkpointer=MemorySaver(),
    interrupt_before=["send"],
    interrupt_after=["draft"]
)
```

Параметр `interrupt_before=["send"]` означает: граф остановится непосредственно перед выполнением узла `send`. Человек может посмотреть текущее состояние, модифицировать его и дать команду продолжить. Параметр `interrupt_after=["draft"]` означает: граф выполнит узел `draft`, сохранит результат и остановится. Человек увидит, что узел написал, и решит, продолжать ли.

Разница между подходами — в гибкости. Вызов `interrupt()` внутри узла позволяет задать вопрос, передать данные, реализовать сложную логику ветвления прямо в точке прерывания. Декларативные прерывания проще — они не требуют модификации кода узлов, но и не дают интерактивности: граф просто останавливается, и вы видите состояние как есть. Для типового сценария «одобри перед выполнением» декларативный подход часто предпочтительнее — одна строка вместо кода внутри узла.

Классический пример использования `interrupt_before` — одобрение вызова инструмента. Агент с доступом к SQL-базе данных собирается выполнить запрос. Вы хотите, чтобы каждый вызов инструмента проходил через человека. Указываете `interrupt_before=["tools"]` при создании агента. Когда модель решит вызвать инструмент, граф остановится. Через `agent.get_state(config)` вы видите, какой инструмент и с какими параметрами собирается быть вызванным. Если SQL-запрос безопасный — продолжаете граф. Если запрос опасный — модифицируете состояние через `agent.update_state()`, заменяя опасный вызов на безопасную версию, и только тогда продолжаете.

> Полный пример: examples_7_5_hitl.py, функции example_interrupt_before_after() и example_tool_approval()

## Часть 4: Command(resume) — обратная связь от человека

Параметр `resume` объекта `Command` — основной механизм передачи данных от человека обратно в граф. Значение, переданное в `resume`, становится возвращаемым значением `interrupt()` внутри узла. Это может быть строка, словарь, число — любой сериализуемый тип.

```python
answer = interrupt("Вы уверены? (yes/no)")
# answer получит то, что передано в Command(resume=...)
```

Но `Command` умеет больше, чем просто передать данные. При возобновлении можно одновременно перенаправить выполнение в другой узел через параметр `goto`. Представьте: человек решил, что черновик нужно переделать. Вместо того чтобы возвращать «redo» и писать в узле обработку этого случая, вы отправляете граф напрямую обратно в узел генерации: `Command(resume={"action": "redo"}, goto="draft")`. Граф пропускает оставшуюся часть текущего узла и переходит в `draft`. А параметр `update` позволяет одновременно обновить состояние — добавить комментарий, пометку об одобрении, любые данные, которые нужны следующим узлам.

> Полный пример: examples_7_5_hitl.py, функция example_command_resume()

## Часть 5: Четыре паттерна HITL

На практике HITL-взаимодействие принимает одну из нескольких устоявшихся форм. Разберём четыре основных паттерна — от самого простого к самому контролирующему.

Первый и самый распространённый — **Approval Gate**, одобрение перед действием. Агент подготавливает необратимое действие — отправку email, выполнение SQL-запроса на удаление, перевод денег — и вызывает `interrupt()`, передавая человеку всю информацию: что именно будет сделано, с какими параметрами, какие последствия. Человек отвечает «одобрить» или «отклонить». Решение бинарное — действие либо выполняется целиком, либо не выполняется вовсе. Этот паттерн ставят перед любым действием, которое нельзя откатить.

Второй паттерн — **Review and Edit** — даёт человеку не просто право вето, а возможность отредактировать результат работы агента. Агент генерирует контент — отчёт, письмо, документ — а человек через `interrupt()` получает черновик и может либо одобрить его как есть, либо отправить свою отредактированную версию. Это особенно полезно для задач генерации текста: модель может ошибиться в деталях, но общая структура верна. Вместо отклонения и полной перегенерации человек вносит точечные правки — быстрее и надёжнее.

Третий — **Tiered Approval**, уровни одобрения. Не все действия одинаково опасны. Паттерн разделяет инструменты на категории по уровню риска. Безопасные — поиск, калькулятор, запрос погоды — выполняются автоматически. Опасные — SQL-запросы, отправка email, удаление файлов — требуют одобрения через `interrupt()`. Реализация сводится к проверке имени инструмента: входит в список опасных — пауза, не входит — выполняем. Агент работает быстро в безопасных сценариях, но останавливается, когда на кону что-то серьёзное. Именно этот паттерн чаще всего используется в продакшне, потому что он балансирует скорость и безопасность.

Четвёртый — **Multi-step Confirmation**, подтверждение каждого шага. Для сложных многошаговых задач, где каждый шаг влияет на следующий, агент перед выполнением каждого действия вызывает `interrupt()`, показывая номер текущего шага, описание действия и уже выполненные шаги. Человек может одобрить шаг, пропустить его или полностью прервать план. Максимальный контроль — но и максимальная нагрузка на человека. Этот паттерн стоит применять только для критичных процессов, где ошибка на любом этапе может быть по-настоящему дорогостоящей.

> Полные примеры: examples_7_5_hitl.py, функции example_approval_gate(), example_review_edit(), example_tiered_approval(), example_multistep_confirmation()

## Часть 6: HITL в реальных приложениях

В реальном приложении человек взаимодействует не через Python-консоль, а через веб-интерфейс. Типичная архитектура выглядит так. Backend на FastAPI запускает граф, и при достижении `interrupt()` возвращает клиенту статус `waiting_for_approval` вместе с данными для принятия решения. Frontend — будь то React, Streamlit или что-то ещё — отображает эти данные пользователю. Когда пользователь нажимает «Одобрить» или «Отклонить», frontend отправляет решение на backend, который продолжает граф через `Command(resume=...)`.

Важная деталь: граф не висит в памяти между паузой и продолжением. Состояние хранится в checkpoint-хранилище, а `thread_id` связывает запросы в единую сессию. Это значит, что backend может быть перезапущен, масштабирован горизонтально, и даже обработать продолжение графа на другом инстансе сервера. На стороне backend достаточно двух эндпоинтов: один для запуска агента, другой для продолжения с решением человека.

Остаётся вопрос: что если человек не ответил? Граф может ждать вечно — checkpoint не имеет срока действия. В реальных системах бесконечное ожидание недопустимо. Решение — обёртка с таймаутом на стороне приложения. Если ответ не получен за заданный интервал, система автоматически формирует `Command(resume=...)` с отклонением по таймауту и уведомляет менеджера или записывает инцидент в лог.

> Полные примеры: examples_7_5_hitl.py, функции example_fastapi_hitl() и example_timeout_escalation()

## Часть 7: Когда НЕ нужен HITL

HITL — мощный инструмент, но его злоупотребление превращает агента в бесполезную программу, которая на каждый чих спрашивает «вы уверены?». Если агент не может сделать ни одного шага без одобрения, зачем вообще нужен агент?

Есть простая эвристика: Risk x Impact. Если произведение вероятности ошибки на масштаб последствий превышает допустимый порог — ставьте HITL. Для всего остального — доверяйте агенту и контролируйте постфактум через логи и мониторинг.

На практике HITL не нужен для операций чтения — поиск в интернете, запрос к базе без модификации данных, получение погоды. Не нужен для промежуточных рассуждений агента — внутренние шаги цепочки мыслей не создают внешних последствий. Не нужен для обратимых действий с низкой ценой ошибки — если действие легко откатить и последствия незначительны, контроль постфактум эффективнее.

HITL необходим для необратимых действий — удаление, отправка, публикация, перевод денег. Необходим для внешних коммуникаций — email, мессенджеры, любое общение от имени пользователя. И необходим в ситуациях неопределённости — когда модель «не уверена», запрос неоднозначен, или последствия ошибки непропорционально велики по сравнению с вероятностью.

Хорошо спроектированная система логирования и алертов часто даёт больше уверенности, чем ручное одобрение каждого действия. Лучший HITL — тот, который срабатывает редко, но в нужных местах.

## Итоги

Human-in-the-Loop в LangGraph — это не костыль, а первоклассная возможность, построенная на persistence из лекции 4. Функция `interrupt()` останавливает граф, сохраняет состояние в checkpoint и передаёт данные человеку. `Command(resume=...)` возвращает решение обратно, причём между паузой и продолжением может пройти произвольное время. Декларативные `interrupt_before` и `interrupt_after` при компиляции графа дают тот же эффект без модификации кода узлов — проще, но менее гибко.

Четыре паттерна покрывают основные сценарии: Approval Gate для бинарных решений перед необратимыми действиями, Review and Edit для редактирования контента, Tiered Approval для разграничения инструментов по уровню риска, Multi-step Confirmation для поэтапного контроля критичных процессов.

Главное правило: HITL нужен там, где Risk x Impact превышает порог. Для всего остального — доверяйте агенту и контролируйте через логи.
