# Лекция 2: State Management — состояние как фундамент графа

## Введение: Почему состояние — это всё

В прошлой лекции мы узнали, что StateGraph — это граф, по которому перемещаются данные. Но мы лишь поверхностно затронули главный вопрос: что именно перемещается? Как устроена структура данных, которую узлы читают и обновляют? Как два узла, работающих с одним и тем же полем, не затирают результаты друг друга?

Представьте состояние как общую доску в офисе. Каждый сотрудник (узел) подходит к доске, читает то, что написали другие, дописывает свою часть и уходит. Критически важно, чтобы правила «дописывания» были чёткими. Если один сотрудник написал список задач, а второй стёр его и написал свой — хаос. Если оба дописывают в общий список — порядок. Эти правила в LangGraph называются **reducer'ами**.

В этой лекции мы разберём всё: от простого TypedDict до вложенных состояний, подграфов и приватных данных узлов.

---

## Часть 1: Определение состояния — TypedDict и Annotation

### Простейший способ: TypedDict

Самый базовый способ описать состояние — использовать `TypedDict` из модуля `typing`. Вы определяете класс, перечисляя поля и их типы:

```python
class ResearchState(TypedDict):
    question: str
    sources: list[str]
    answer: str
    confidence: float
```

Это работает, но с ограничением: при обновлении каждое поле **полностью заменяется** новым значением. Если узел A записал в `sources` список `["wiki"]`, а потом узел B записал `["arxiv"]` — первый список пропал. Для полей вроде `answer` или `confidence` замена — это ожидаемое поведение. Но для `sources` мы скорее всего хотим **дополнение**: каждый узел добавляет свои источники к общему списку.

> Полный пример: examples_7_2_state.py, функция `example_typed_dict_state()`

### Annotation: состояние с reducer'ами

Для управления логикой обновления LangGraph использует стандартный Python-механизм `Annotated`. Синтаксис выглядит так: `Annotated[тип, reducer]`, где reducer — это функция двух аргументов (старое значение и новое), возвращающая результат объединения.

```python
sources: Annotated[list[str], operator.add]
messages: Annotated[list[BaseMessage], add_messages]
attempt_count: Annotated[int, lambda old, new: old + new]
```

Разберём, что даёт каждый из этих reducer'ов. Поле без аннотации (например, `question: str`) ведёт себя по умолчанию — новое значение заменяет старое. Для `sources` используется `operator.add`, который для списков означает конкатенацию: если текущее значение `["wiki"]` и узел вернул `["arxiv"]`, результат будет `["wiki", "arxiv"]`. Для `messages` применяется специальный reducer `add_messages` из LangGraph, который умнее простого дополнения — он обрабатывает дубликаты по `id` сообщения и поддерживает удаление. Наконец, для счётчика `attempt_count` используется лямбда, суммирующая старое и новое значения: если текущее значение `2` и узел вернул `1`, результат будет `3`.

На практике чаще всего встречаются следующие сценарии: полная замена (без аннотации), дополнение списка (`operator.add`), работа с сообщениями (`add_messages`), счётчики (лямбда-суммирование), множества без дубликатов (лямбда с `set`), а также паттерн «последнее непустое значение» (лямбда, возвращающая новое значение, если оно не пустое, или старое в противном случае).

> Полный пример: examples_7_2_state.py, функция `example_reducers()`

### Overwrite — обход reducer'а

Иногда reducer мешает. Представьте: у вас поле `findings: Annotated[list[str], operator.add]`, куда каждый узел дополняет результаты. Но в определённый момент вам нужно **сбросить** этот список и начать заново — например, при переходе к новой фазе исследования. Обычное обновление `{"findings": ["новый_старт"]}` не поможет: reducer дополнит список вместо замены.

Для этого в LangGraph 1.1.2 появился тип `Overwrite` из модуля `langgraph.types`. Он позволяет обойти reducer и записать значение напрямую:

```python
from langgraph.types import Overwrite

def reset_node(state) -> dict:
    # Полностью заменяет findings, игнорируя reducer operator.add
    return {"findings": Overwrite(["только_это"])}
```

`Overwrite` полезен, когда нужно сбросить накопленное состояние — например, очистить историю промежуточных результатов, перезапустить счётчик или заменить список целиком. При этом определение поля с reducer'ом остаётся неизменным — `Overwrite` действует точечно, только в конкретном обновлении.

> Полный пример: examples_7_2_state.py, функция `example_overwrite()`

## Часть 2: add_messages — главный reducer в агентных приложениях

### Почему для сообщений нужен особый reducer

Список сообщений — центральная структура данных в любом агентном приложении. Каждый вызов LLM добавляет сообщения (запрос и ответ). Каждый вызов инструмента добавляет `ToolMessage`. История растёт.

Простой `operator.add` для сообщений работает, но грубо. Что если модель «перегенерировала» ответ — вы хотите заменить последнее сообщение, а не добавить дубликат? Что если нужно удалить старые сообщения, чтобы не раздувать контекст?

`add_messages` решает эти задачи, работая в трёх режимах.

### Как работает add_messages

Первый режим — **добавление**. Если новое сообщение имеет `id`, которого нет в текущем списке, оно просто дописывается в конец. Это стандартное поведение при обычном общении.

Второй режим — **обновление по id**. Если сообщение с таким `id` уже существует, оно заменяется новым. Это полезно, когда модель перегенерирует ответ и вы хотите обновить его вместо добавления дубликата.

Третий режим — **удаление**. Для этого используется специальный тип `RemoveMessage`: вы передаёте `RemoveMessage(id="1")`, и сообщение с этим `id` удаляется из списка. Это мощный механизм управления контекстным окном — можно «сжимать» историю, удаляя старые сообщения.

```python
# Удаление сообщения по id
new = [RemoveMessage(id="1")]
result = add_messages(old_messages, new)
```

> Полный пример: examples_7_2_state.py, функция `example_add_messages()`

### MessagesState — готовый шаблон

LangGraph предоставляет готовый класс `MessagesState`, который по сути является `TypedDict` с единственным полем `messages`, уже аннотированным reducer'ом `add_messages`. Если вашему графу нужны только сообщения — используйте его напрямую. Если нужны дополнительные поля — наследуйтесь от него и добавляйте свои:

```python
class AgentState(MessagesState):
    current_plan: str
    tools_used: Annotated[list[str], operator.add]
    iteration: int
```

Так вы получаете `messages` с правильным reducer'ом и добавляете свои поля без необходимости повторять определение.

> Полный пример: examples_7_2_state.py, функция `example_messages_state()`

## Часть 3: Как узлы обновляют состояние

### Правило: узел возвращает только изменения

Узел графа — функция, которая принимает полное состояние и возвращает словарь с обновлениями. Не нужно возвращать всё состояние — только те поля, которые изменились. Например, узел, создающий план, возвращает только `plan` и `current_step`, не трогая `goal` и `result`. А узел, выполняющий шаг, обновляет `current_step` и `result`, не затрагивая `plan`.

Это удобно: каждый узел «знает» только о тех полях, которые ему нужны. Это снижает связность между узлами и упрощает тестирование — можно протестировать узел изолированно, передав ему минимальное состояние.

> Полный пример: examples_7_2_state.py, функция `example_partial_updates()`

### Что происходит при обновлении

Когда узел возвращает словарь, LangGraph обрабатывает каждое поле по определённому алгоритму. Если у поля есть reducer, вызывается `reducer(old_value, new_value)`. Если reducer'а нет, старое значение просто заменяется новым. Поля, не упомянутые в возвращённом словаре, остаются без изменений.

Это важно понимать при отладке. Если вы видите, что поле «сбрасывается» — проверьте, не возвращает ли его какой-то узел со значением по умолчанию (пустая строка, пустой список, ноль). Такие ошибки бывает сложно найти, потому что сброс происходит неявно.

## Часть 4: Валидация состояния через Pydantic

### Зачем Pydantic, если есть TypedDict

`TypedDict` — это аннотация типов, но она не выполняет проверки в runtime. Если узел вернёт строку вместо числа для поля `confidence: float`, вы не получите ошибку — просто сломанное состояние, которое проявит себя позже в непредсказуемом месте.

Pydantic решает эту проблему. Вместо `TypedDict` вы наследуете состояние от `BaseModel` и описываете поля через `Field`. Каждое поле получает тип, значение по умолчанию, описание и, при необходимости, ограничения (например, `ge=0.0, le=10.0` для числового диапазона). Можно также определить кастомные валидаторы через декоратор `@field_validator`, которые будут проверять бизнес-правила при каждом обновлении состояния.

Использование Pydantic даёт несколько преимуществ. Валидация типов ловит ошибки сразу при обновлении, а не через три узла. Значения по умолчанию избавляют от необходимости инициализировать каждое поле при первом вызове. Описания в `Field(description=...)` служат документацией. Кастомные валидаторы позволяют проверять бизнес-правила — например, что оценка находится в допустимом диапазоне.

> Полный пример: examples_7_2_state.py, функция `example_pydantic_validation()`

### Важные ограничения Pydantic-состояния

У Pydantic-состояния в LangGraph есть несколько неочевидных особенностей, о которых важно знать.

Во-первых, **выход графа не является экземпляром Pydantic-модели** — это обычный словарь (`dict`). Даже если состояние определено как `BaseModel`, результат `app.invoke(...)` вернёт `dict`, а не объект вашей модели. Это значит, что вы не сможете вызывать методы модели на результате или полагаться на Pydantic-валидацию при чтении выходных данных.

Во-вторых, **runtime-валидация происходит только на входе первого узла**. Между узлами промежуточные обновления не проходят через валидаторы Pydantic. Если узел вернёт некорректное значение, ошибка может проявиться не сразу, а позже — при чтении этого поля другим узлом.

В-третьих, Pydantic-модели хуже работают с **частичными обновлениями**. TypedDict позволяет узлу вернуть словарь только с изменёнными полями, и это работает естественно. С Pydantic частичные обновления тоже поддерживаются, но механика менее прозрачна.

По этим причинам **TypedDict обычно рекомендуется** как основной способ определения состояния, особенно если вам нужны частичные обновления и прозрачное поведение. Pydantic стоит использовать целенаправленно — когда валидация входных данных критически важна.

## Часть 5: Вложенные состояния и подграфы

### Зачем нужны подграфы

Когда граф разрастается, хочется разбить его на независимые модули. Подграф (subgraph) — это граф внутри графа. Он имеет собственное состояние и собственную логику, но может быть вставлен как узел в родительский граф.

### Пример: агент с подграфом для исследования

Допустим, у нас есть агент-аналитик, который отвечает на вопросы. Один из его шагов — исследование: поиск по нескольким источникам, объединение результатов. Это сложная логика, которую удобно выделить в подграф.

Подграф определяет собственное состояние `ResearchState` с полями `query` и `findings` (с reducer'ом `operator.add` для накопления). Внутри подграфа несколько узлов — например, поиск в интернете и поиск в документации — могут запускаться **параллельно**. Для этого достаточно провести из `START` два ребра к этим узлам. LangGraph автоматически выполнит их одновременно и дождётся обоих результатов перед переходом к следующему узлу (суммаризации). Такой паттерн называется **fan-out / fan-in**: fan-out — один поток разветвляется на несколько параллельных, fan-in — несколько параллельных потоков сходятся обратно в один. Здесь это встроенный механизм LangGraph.

> Полный пример: examples_7_2_state.py, функция `example_subgraph()`

### Встраивание подграфа в родительский граф

Скомпилированный подграф можно использовать как узел в основном графе, передав его в `add_node`. Здесь есть тонкость: состояние подграфа и состояние родительского графа — разные типы. LangGraph автоматически проксирует поля с одинаковыми именами. Если имена не совпадают, нужно явно описать маппинг.

### Альтернатива: подграф через вызов внутри узла

Для более тонкого контроля можно вызвать подграф как обычную функцию внутри узла — через `compiled_research.invoke(...)`. Этот подход позволяет вручную маппить поля между состояниями родительского графа и подграфа. Например, взять `question` из родительского состояния, передать его как `query` подграфу, а из результата подграфа извлечь последний элемент `findings` и записать его в `research_results` родительского состояния.

Однако у этого подхода есть обратная сторона: подграф не будет виден в визуализации и трейсинге как отдельная структура. Выбирайте между автоматическим маппингом (удобство и прозрачность) и ручным вызовом (полный контроль) в зависимости от ваших потребностей.

> Полный пример: examples_7_2_state.py, функция `example_subgraph_in_parent()`

## Часть 6: Приватное состояние узла

### Проблема: данные, нужные одному узлу

Иногда узлу нужны промежуточные данные, которые не должны быть видны остальным узлам и не должны попадать в финальный результат. Например, узел поиска хочет хранить «сырые» результаты API до обработки. Или узел планирования хочет хранить внутренний scratchpad.

Засорять общее состояние такими полями — плохая практика. Это раздувает объём данных, путает других разработчиков и усложняет сериализацию.

### Решение: input/output schema

LangGraph позволяет определить три схемы для графа: полное внутреннее состояние (`OverallState`), входную схему (`InputState`) и выходную схему (`OutputState`). При создании графа они передаются так:

```python
graph = StateGraph(OverallState, input_schema=InputState, output_schema=OutputState)
```

Теперь при вызове `app.invoke({"question": "..."})` граф принимает только поля из `InputState` и возвращает только поля из `OutputState`. Внутренние поля (такие как `internal_notes` и `search_cache`) живут внутри графа, но не видны снаружи. Это чистый API: вызывающий код не знает и не заботится о внутренней кухне.

> Полный пример: examples_7_2_state.py, функция `example_private_state()`

## Часть 7: Практический пример — агент-исследователь

Давайте построим агента, который получает исследовательский вопрос, планирует подход, ищет информацию, оценивает качество и итерирует при необходимости.

Состояние агента `ResearcherState` содержит шесть полей: вопрос, план, накопленные находки (с reducer'ом `operator.add`), оценку качества, финальный ответ и счётчик итераций. Граф состоит из четырёх узлов и одного маршрутизатора.

Узел **planner** получает вопрос от пользователя и через LLM составляет краткий план исследования из двух-трёх шагов.

Узел **researcher** выполняет реальный поиск через DuckDuckGo. Сначала LLM формулирует поисковый запрос на основе плана, а затем `DuckDuckGoSearchRun` выполняет поиск. При повторных вызовах (в цикле) учитываются уже найденное и обратная связь от evaluator'а. Результаты дополняются благодаря reducer'у `operator.add` — каждая итерация добавляет новые данные, а не затирает предыдущие. Здесь же инкрементируется счётчик итераций.

Узел **evaluator** оценивает, достаточно ли собранных данных для полного ответа. Он отвечает SUFFICIENT или INSUFFICIENT и объясняет, чего не хватает.

Маршрутизатор **should_continue** принимает решение на основе оценки: если данных достаточно или исчерпан лимит в три итерации — направляет поток к узлу-писателю. Иначе — ещё один раунд исследования.

Узел **writer** синтезирует финальный ответ из всех собранных данных.

Цикл `research -> evaluate -> research` может выполниться от одного до трёх раз. Каждая итерация обогащает `findings` новыми данными. Evaluator направляет исследование, указывая на пробелы, а researcher при следующем заходе учитывает эту обратную связь.

> Полный пример: examples_7_2_state.py, функция `example_research_agent()`

## Итоги

Состояние — это фундамент графа в LangGraph.

**TypedDict** — минимальный способ описать структуру состояния. Подходит для прототипов.

**Annotated + reducer** — механизм управления тем, как обновляются поля. `operator.add` для списков, `add_messages` для сообщений, лямбды для кастомной логики. **Overwrite** из `langgraph.types` позволяет точечно обойти reducer и записать значение напрямую.

**add_messages** — специализированный reducer для работы с историей сообщений. Поддерживает добавление, обновление по `id` и удаление.

**Pydantic** — для продакшн-состояний с валидацией типов, значениями по умолчанию и бизнес-правилами. Но помните: выход графа — обычный `dict`, а валидация работает только на входе первого узла. TypedDict обычно предпочтительнее.

**Подграфы** — позволяют разбить сложный граф на независимые модули с собственным состоянием.

**Input/Output schema** — разделяет внешний API графа и его внутреннее состояние.
