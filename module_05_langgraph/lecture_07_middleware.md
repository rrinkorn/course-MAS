# Лекция 7: Middleware — декларативная устойчивость агентов

## Введение

В лекции 6 мы разбирали ручные стратегии обработки ошибок: try/except в узлах, retry с backoff, fallback-рёбра, глобальные лимиты. Каждая из этих стратегий даёт тонкий контроль — вы решаете, что именно происходит при ошибке в каждом конкретном узле. Но у этого подхода есть цена: повторяющийся код. Retry с backoff для вызова модели — это пять-шесть строк try/except с циклом. И этот же код нужно написать в каждом узле, где есть вызов модели. Умножьте на десять узлов — и получите полсотни строк одинаковой обработки ошибок.

Middleware — принципиально другой подход. Вместо того чтобы писать обработку ошибок в каждом узле, вы объявляете поведение один раз при создании агента. Retry модели, fallback на запасную модель, лимит вызовов, сжатие длинной истории, даже HITL-одобрение инструментов — всё подключается через параметр `middleware=[]` в `create_agent`. Код узлов остаётся чистым — они занимаются бизнес-логикой, а middleware берут на себя инфраструктуру.

На сегодняшний день в `langchain.agents.middleware` доступно шестнадцать встроенных middleware. Мы разберём самые важные, научимся их комбинировать и напишем свои.

## Часть 1: Как middleware устроены

Представьте middleware как матрёшку. Снаружи — первый middleware, внутри него — второй, ещё глубже — третий, а в самом центре — реальный вызов модели или инструмента. Когда агент хочет вызвать модель, запрос проходит через все слои middleware снаружи внутрь. Каждый слой может модифицировать запрос, перехватить ошибку, повторить вызов или прервать выполнение. Ответ модели проходит обратно — изнутри наружу — и каждый слой может его модифицировать.

Middleware передаются списком в `create_agent`. Порядок имеет значение, причём по-разному для разных типов хуков. «Before»-хуки выполняются в прямом порядке: первый middleware в списке срабатывает первым. «After»-хуки — в обратном: первый middleware срабатывает последним. «Wrap»-хуки вкладываются друг в друга как матрёшка: первый в списке — внешний слой.

На практике порядок редко критичен, но если вы комбинируете retry и fallback, логично поставить retry снаружи: сначала попробовать основную модель несколько раз, и только если все попытки провалились — переключиться на запасную.

> Полный пример: examples_07_middleware.py, функция `example_basic_middleware()`

## Часть 2: Устойчивость — retry и fallback

Три middleware образуют первую линию обороны: повторные попытки при транзиентных сбоях и автоматическое переключение на запасные модели.

`ModelRetryMiddleware` — retry при сбоях модели. Самый частый сбой — модель недоступна. Таймаут, rate limit, серверная ошибка. Middleware перехватывает ошибку, ждёт заданное время и повторяет вызов. Задержка растёт экспоненциально с jitter'ом, чтобы десять агентов, одновременно получивших rate limit, не ломились к API одновременно. При исчерпании попыток параметр `on_failure` определяет, что делать: вернуть AIMessage с описанием ошибки (`"continue"`) или пробросить исключение наверх (`"error"`). Параметр `retry_on` позволяет фильтровать ошибки — повторять только при 429 и 503, но не при 400.

`ToolRetryMiddleware` — аналог для инструментов. Внешний API вернул таймаут, база данных кратковременно недоступна — middleware повторяет вызов. Ключевое отличие: при исчерпании попыток ошибка по умолчанию возвращается модели как `ToolMessage`, и та может скорректировать подход — попробовать другие аргументы, выбрать другой инструмент или ответить без инструмента. Это тот же паттерн, что мы видели с `handle_tool_errors` на `ToolNode` в лекции 6, но реализованный через middleware. Параметр `tools` позволяет применить retry только к конкретным инструментам.

`ModelFallbackMiddleware` — каскадный fallback. Если основная модель недоступна не временно, а надолго, retry не поможет. Middleware автоматически переключается на запасную модель. Модели пробуются последовательно: основная → первый fallback → второй fallback. Типичная конфигурация: `gpt-5.4` → `gpt-5.4-mini` → `claude-haiku-4-5`. Пользователь получает ответ от наилучшей доступной модели.

> Полный пример: examples_07_middleware.py, функция `example_reliability_middleware()`

## Часть 3: Лимиты и управление контекстом

Вторая группа middleware защищает от расходов и переполнения контекста.

`ModelCallLimitMiddleware` ограничивает количество вызовов модели. Можно задать лимит на один запуск (`run_limit`) или на весь thread (`thread_limit`, требует checkpointer). Если агент исчерпал лимит, middleware может либо завершить агента досрочно (`exit_behavior="end"`), либо выбросить ошибку. Этот middleware — страховка от зацикливания: если модель раз за разом генерирует tool_call'ы, но не может завершить задачу, ModelCallLimitMiddleware остановит её после N итераций.

`ToolCallLimitMiddleware` — аналогичная страховка, но для инструментов. Можно задать глобальный лимит на все инструменты или лимит на конкретный инструмент по имени. Особенно полезно для дорогих инструментов: web_search — не больше десяти вызовов, database_query — не больше пяти.

`SummarizationMiddleware` решает проблему длинных разговоров. История сообщений растёт неограниченно — в какой-то момент она превысит контекстное окно модели. Middleware автоматически вызывает отдельную (обычно более дешёвую) модель, чтобы суммаризировать старые сообщения. Последние N сообщений сохраняются для контекста текущего разговора, более ранние заменяются кратким резюме. Порог срабатывания задаётся по токенам, по количеству сообщений или через `"auto"` — middleware сам адаптируется к размеру контекстного окна модели.

`ContextEditingMiddleware` — более хирургический подход к управлению контекстом. Вместо суммаризации он вырезает конкретные части истории — например, старые результаты вызовов инструментов, которые занимают много места, но больше не актуальны. Настраивается через правила: при превышении порога в N токенов — очистить все результаты инструментов кроме последних трёх. Это повторяет подход Anthropic с `clear_tool_uses` — удаление неактуальных tool use блоков из контекста.

> Полный пример: examples_07_middleware.py, функция `example_limits_and_context()`

## Часть 4: Безопасность и контроль

Третья группа middleware решает задачи, которые ручными стратегиями из лекции 6 покрыть трудно.

`HumanInTheLoopMiddleware` — HITL через middleware. В лекции 5 мы подробно разбирали `interrupt()` внутри узлов. Этот middleware делает то же самое, но декларативно: вы указываете, какие инструменты требуют одобрения, и middleware автоматически останавливает агента перед их вызовом. Человек может одобрить вызов, отредактировать аргументы или отклонить. Не нужно модифицировать код узлов — одна строка при создании агента, и инструмент `send_email` всегда требует одобрения, а `web_search` выполняется автоматически. Это по сути паттерн Tiered Approval из лекции 5, реализованный без единой строки кода в узлах.

`PIIMiddleware` — обнаружение и редактирование персональных данных. Middleware может проверять входящие сообщения, ответы модели и результаты инструментов на наличие email-адресов, номеров кредитных карт, IP-адресов и других чувствительных данных. При обнаружении — редактирование, маскирование, хэширование или полная блокировка. Можно передать собственный детектор — функцию или регулярное выражение.

`LLMToolSelectorMiddleware` — оптимизация для агентов с большим набором инструментов. Если у агента двадцать инструментов, их описания занимают значительную часть контекстного окна. Middleware использует быструю модель, чтобы перед каждым вызовом основной модели отобрать только релевантные инструменты — например, три из двадцати. Основная модель получает компактный набор и тратит меньше токенов на промпт.

> Полный пример: examples_07_middleware.py, функция `example_safety_middleware()`

## Часть 5: Комбинирование и полная экосистема

Помимо разобранных десяти middleware, в `langchain.agents.middleware` есть ещё шесть, закрывающих специализированные задачи: `TodoListMiddleware` добавляет агенту инструмент планирования задач, `LLMToolEmulator` подменяет реальные инструменты LLM-ответами для тестирования, `ShellToolMiddleware` даёт агенту доступ к shell-сессии с политиками безопасности (включая Docker-изоляцию), `FilesystemFileSearchMiddleware` добавляет инструменты glob и grep для работы с файловой системой. Экосистема растёт — в будущих версиях middleware будут появляться новые.

Главная сила middleware — в композиции. Каждый решает одну задачу. Вместе они обеспечивают комплексную устойчивость. Разберём типичный продакшн-стек. Агент вызывает модель — запрос проходит через `ModelRetryMiddleware`, который при сбое повторяет вызов до трёх раз. Если все попытки исчерпаны — `ModelFallbackMiddleware` переключает на запасную модель. `ModelCallLimitMiddleware` следит, чтобы модель не вызывалась больше пятнадцати раз за запуск. Агент вызывает инструмент search — `ToolRetryMiddleware` повторяет при ConnectionError. `ToolCallLimitMiddleware` следит за лимитом вызовов. `HumanInTheLoopMiddleware` останавливает агента перед send_email. `SummarizationMiddleware` сжимает историю, если она разрослась. Семь слоёв защиты, семь строк при создании агента.

> Полный пример: examples_07_middleware.py, функция `example_combined_middleware()`

## Часть 6: Кастомные middleware

Встроенные middleware покрывают типовые сценарии. Но иногда нужна своя логика: ограничить число сообщений в разговоре, логировать каждый вызов модели, добавить динамический контекст в промпт, проверить ответ модели на запрещённый контент. Для этого есть два подхода: декораторный и class-based.

Шесть декораторов определяют, в какой момент жизненного цикла агента выполняется ваш код. `@before_model` — перед каждым вызовом модели. `@after_model` — после каждого ответа. `@wrap_model_call` — оборачивает вызов целиком, давая доступ и к запросу, и к ответу. Аналогично `@wrap_tool_call` — для инструментов. `@before_agent` и `@after_agent` — один раз в начале и в конце работы агента.

Функция-хук принимает текущее состояние и объект runtime, а возвращает либо `None` (ничего не менять), либо словарь с обновлениями состояния. Специальный ключ `"jump_to": "end"` в возвращаемом словаре досрочно завершает агента — полезно для лимитов: если сообщений больше пятидесяти, если модель вернула запрещённый контент, если бюджет исчерпан.

Wrap-декораторы — самые мощные. Функция получает объект `ModelRequest` и handler — функцию реального вызова. Вы можете модифицировать запрос до вызова через `request.override(model=..., tools=...)` — например, динамически выбрать модель или отфильтровать инструменты. Затем выполнить вызов через handler. Затем проинспектировать или модифицировать ответ. Если хук должен обновить состояние графа, он возвращает `ExtendedModelResponse` — обёртку, которая содержит и ответ модели, и `Command` с обновлениями состояния.

Для middleware с внутренним состоянием — счётчиками, кэшами, конфигурацией — удобнее наследоваться от класса `AgentMiddleware`. Вы определяете методы `before_model`, `after_model`, `wrap_model_call` и другие, а внутреннее состояние хранится в атрибутах экземпляра. Декоратор `@hook_config(can_jump_to=["end"])` на методе класса объявляет, что этот хук может перенаправлять в `end`.

На практике декораторный подход используется чаще — он проще и покрывает большинство случаев. Class-based нужен, когда middleware должен хранить состояние между вызовами или когда логика достаточно сложна для выделения в отдельный класс с параметрами.

> Полные примеры: examples_07_middleware.py, функции `example_custom_decorator()` и `example_custom_class()`

## Часть 7: Middleware vs RetryPolicy

В LangGraph есть два механизма, которые решают похожую задачу — retry при ошибках. Важно понимать разницу.

`RetryPolicy` из `langgraph.types` — механизм на уровне графа. Он применяется к любому узлу через параметр `retry_policy` в `add_node` или к любой задаче через декоратор `@task(retry_policy=...)`. Если узел выбросил исключение — LangGraph повторяет его целиком. Работает с любым графом, не только с агентами. Мы разбирали этот механизм в лекции 6.

Middleware — механизм на уровне агента. Работает только с агентами, созданными через `create_agent`. Но зато умеет гораздо больше: не только retry, но и fallback на другие модели, лимиты вызовов, суммаризацию истории, HITL, PII-фильтрацию, кастомные хуки на любом этапе жизненного цикла. И retry работает точнее — не на уровне узла целиком, а на уровне отдельных вызовов модели или инструментов внутри узла.

Когда что выбирать? Если вы строите граф вручную через StateGraph — `RetryPolicy`, это единственный вариант. Если используете `create_agent` — middleware удобнее и мощнее. Оба механизма можно комбинировать: `RetryPolicy` на узлах внешнего графа, middleware внутри агентов, встроенных как узлы. Два уровня защиты.

## Итоги

В `langchain.agents.middleware` — шестнадцать готовых middleware. Три для устойчивости: ModelRetryMiddleware, ToolRetryMiddleware, ModelFallbackMiddleware. Четыре для лимитов и контекста: ModelCallLimitMiddleware, ToolCallLimitMiddleware, SummarizationMiddleware, ContextEditingMiddleware. Три для безопасности и контроля: HumanInTheLoopMiddleware, PIIMiddleware, LLMToolSelectorMiddleware. И шесть специализированных — от эмуляции инструментов до shell-доступа. Все подключаются через `create_agent(middleware=[...])`.

Кастомные middleware создаются через шесть декораторов (`@before_model`, `@after_model`, `@wrap_model_call`, `@wrap_tool_call`, `@before_agent`, `@after_agent`) или через класс `AgentMiddleware`. Wrap-хуки — самые мощные: доступ к запросу и ответу, модификация через `ModelRequest.override()`, обновление состояния через `ExtendedModelResponse`.

Middleware — для агентов через `create_agent`. RetryPolicy — для произвольных графов и задач. Комбинируются для многоуровневой защиты.
